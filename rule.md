# KUWorkspace Rule Reference

## 문서 목적

- 이 문서는 `PLAN.md`가 아니라 현재 코드베이스 구현을 기준으로 정리한 규칙 문서다.
- 구현상 이미 고정된 내용을 `확정 정책`으로 정리한다.
- 코드만으로는 최종 정책으로 보기 어려운 부분은 `미정 정책`으로 분리한다.

## 기준 소스

- 진입/세션 흐름: `main.py`
- 메뉴/UI 흐름: `src/cli/guest_menu.py`, `src/cli/user_menu.py`, `src/cli/admin_menu.py`, `src/cli/clock_menu.py`, `src/cli/menu.py`, `src/cli/validators.py`
- 인증/예약/정책/패널티: `src/domain/auth_rules.py`, `src/domain/auth_service.py`, `src/domain/room_service.py`, `src/domain/equipment_service.py`, `src/domain/policy_service.py`, `src/domain/penalty_service.py`, `src/domain/restriction_rules.py`
- 상수/운영 시계/저장 규칙: `src/config.py`, `src/runtime_clock.py`, `src/storage/repositories.py`

## 확정 정책

## 1. 시스템 공통 규칙

### 1-1. 시스템 형태

- 로컬 CLI 프로그램이다.
- 데이터는 `data/*.txt` JSON Lines 파일에 저장된다.
- 쓰기 작업은 전역 락과 `UnitOfWork`를 전제로 한다.

### 1-2. 운영 시계

- 프로그램 시작 시 운영 시작 시점을 직접 입력한다.
- 운영 시점은 `09:00`, `18:00`만 허용된다.
- 시점 이동 순서는 `09:00 -> 18:00 -> 다음날 09:00`만 가능하다.
- 게스트는 운영 시계를 조회만 할 수 있다.
- 로그인 사용자는 운영 시계 메뉴에 접근할 수 있다.

### 1-3. 상위 사용자 흐름

`프로그램 시작 -> 데이터 파일 준비 -> 운영 시작 시점 입력 -> 게스트 메뉴 -> 로그인 -> 역할별 메뉴 -> 로그아웃 -> 게스트 메뉴 복귀 -> 종료`

### 1-4. 역할 구분

- 게스트:
  - 로그인
  - 회원가입
  - 운영 시계 조회
  - 종료
- 일반 사용자:
  - 회의실/장비 조회
  - 회의실/장비 예약 생성, 변경, 취소
  - 회의실 체크인 요청
  - 회의실 퇴실 신청
  - 장비 픽업 요청
  - 장비 반납 신청
  - 내 상태 조회
  - 운영 시계
  - 로그아웃
- 관리자:
  - 회의실/장비 목록 및 상태 변경
  - 전체 예약 조회
  - 시작 승인
  - 종료 승인
  - 예약 변경/취소
  - 노쇼 처리
  - 지연 종료 처리
  - 사용자 조회
  - 수동 패널티 부여
  - 운영 시계
  - 로그아웃

## 2. 입력 규칙

### 2-1. 시작 시점 입력

- 날짜 형식은 `YYYY-MM-DD`여야 한다.
- 시작 슬롯은 `09:00` 또는 `18:00`만 허용된다.
- 입력한 시작 시점은 기존 데이터의 최신 시각보다 빠를 수 없다.

### 2-2. 인증 입력

- 사용자명:
  - 앞뒤 공백 제거 후 사용한다.
  - 필수 입력이다.
  - 3자 이상 20자 이하다.
  - 영문, 숫자, `_`만 허용된다.
- 비밀번호:
  - 앞뒤 공백 제거 후 사용한다.
  - 필수 입력이다.
  - 4자 이상 50자 이하다.
- 회원가입 시 비밀번호 확인이 일치해야 한다.
- 사용자명은 중복될 수 없다.
- 비밀번호는 평문 저장이다.

### 2-3. 메뉴/선택 입력

- 메뉴 선택은 숫자 입력 기반이다.
- 목록 선택은 번호 기반이며 기본적으로 `0`은 취소다.
- 확인 입력은 `y/yes/예/ㅇ`, `n/no/아니오/ㄴ`를 지원한다.
- 잘못된 입력에는 오류 메시지를 출력한다.

### 2-4. 예약 날짜 입력

- 예약 UI는 날짜 범위 입력형이다.
- 시간은 직접 입력하지 않는다.
- 취소 입력으로 `q`, `quit`, `취소`를 허용한다.
- 시작일은 내일부터 가능하다.
- 시작일은 현재 시점 기준 최대 6개월 이내여야 한다.
- 종료일은 시작일보다 빠를 수 없다.
- 예약 기간은 최대 14일이다.
- 실제 예약 시간은 자동으로 `시작일 09:00 ~ 종료일 18:00`으로 고정된다.

### 2-5. 숫자 입력

- 회의실 예약 인원은 1~100 범위를 사용한다.
- 관리자 수동 패널티 점수는 1~5 범위만 허용한다.

## 3. 정책 점검 규칙

### 3-1. 정책 점검 실행 시점

- 게스트 메뉴 진입 시
- 로그인 직후
- 사용자 메뉴 루프 진입 시
- 사용자 예약 생성/변경/취소 직전
- 관리자 메뉴 루프 진입 시
- 시점 이동 성공 직후

### 3-2. 정책 점검이 수행하는 일

- 90일 경과 패널티 초기화
- 제한 기간 만료 반영
- 6점 이상 사용자 미래 예약 자동 취소

### 3-3. 정책 점검이 수행하지 않는 일

- 노쇼 자동 처리
- 지연 종료 자동 처리

## 4. 예약 공통 규칙

### 4-1. 자원 상태

- 회의실/장비 상태는 `available`, `maintenance`, `disabled`다.
- `available`이 아니면 신규 예약이 불가능하다.

### 4-2. 활성 예약 정의

- 회의실 활성 예약:
  - `reserved`
  - `checkin_requested`
  - `checked_in`
  - `checkout_requested`
- 장비 활성 예약:
  - `reserved`
  - `pickup_requested`
  - `checked_out`
  - `return_requested`
- 충돌 판정은 활성 예약만 대상으로 한다.

### 4-3. 예약 가능 수

- 정상 사용자는 회의실 활성 예약 1건, 장비 활성 예약 1건까지 가능하다.
- 제한 사용자는 회의실과 장비를 합쳐 활성 예약 총 1건만 가능하다.
- 금지 사용자는 신규 예약이 불가능하다.

## 5. 사용자 시나리오 규칙

### 5-1. 로그인/회원가입

#### 회원가입

- 게스트 메뉴에서 수행한다.
- 사용자명/비밀번호 검증을 통과해야 한다.
- 기본 역할은 일반 사용자다.

#### 로그인

- 사용자명과 비밀번호가 모두 필요하다.
- 성공 후 정책 점검을 한 번 더 수행한다.
- 제한/금지 상태 사용자도 로그인 자체는 가능하다.

### 5-2. 회의실 예약

#### 생성

- 정책 점검 후 진행한다.
- 금지 사용자는 예약할 수 없다.
- 제한 사용자는 전체 활성 예약이 이미 1건이면 예약할 수 없다.
- 이용 인원은 1명 이상이어야 한다.
- 회의실 수용 인원이 이용 인원보다 작으면 예약할 수 없다.
- 회의실 상태가 `available`이어야 한다.
- 동일 회의실에 활성 예약 충돌이 없어야 한다.
- 생성 후 상태는 `reserved`다.

#### 변경

- 정책 점검 후 진행한다.
- 본인 예약만 변경할 수 있다.
- `reserved` 상태만 변경할 수 있다.
- 새 기간도 예약 날짜 규칙과 충돌 규칙을 다시 통과해야 한다.

#### 취소

- 정책 점검 후 진행한다.
- 본인 예약만 취소할 수 있다.
- `reserved` 상태만 취소할 수 있다.
- 예약 시작까지 남은 시간이 `0~60분`이면 직전 취소 패널티 `+2점`이 부여된다.
- 취소 후 상태는 `cancelled`다.

#### 체크인 요청

- 본인 예약만 요청할 수 있다.
- `reserved` 상태만 가능하다.
- 현재 운영 시점이 예약 시작 시각과 정확히 같아야 한다.
- 요청 후 상태는 `checkin_requested`다.
- 관리자 승인 전까지 실제 이용 시작은 아니다.

#### 퇴실 신청

- 본인 예약만 신청할 수 있다.
- `checked_in` 상태만 가능하다.
- 현재 운영 시점이 예약 종료 시각과 정확히 같아야 한다.
- 신청 후 상태는 `checkout_requested`다.
- 관리자 승인 전까지 종료 완료가 아니다.

### 5-3. 장비 예약

#### 생성

- 정책 점검 후 진행한다.
- 금지/제한 규칙은 회의실 예약과 동일하다.
- 먼저 장비 종류를 선택하고 그 다음 개별 자산을 선택한다.
- 장비는 개별 자산 단위로 충돌을 판정한다.
- 장비 상태가 `available`이어야 한다.
- 생성 후 상태는 `reserved`다.

#### 변경

- 정책 점검 후 진행한다.
- 본인 예약만 변경할 수 있다.
- `reserved` 상태만 변경할 수 있다.
- 같은 장비 자산 기준으로 활성 예약 충돌이 없어야 한다.

#### 취소

- 정책 점검 후 진행한다.
- 본인 예약만 취소할 수 있다.
- `reserved` 상태만 취소할 수 있다.
- 예약 시작까지 남은 시간이 `0~60분`이면 직전 취소 패널티 `+2점`이 부여된다.
- 취소 후 상태는 `cancelled`다.

#### 픽업 요청

- 본인 예약만 요청할 수 있다.
- `reserved` 상태만 가능하다.
- 현재 운영 시점이 예약 시작 시각과 정확히 같아야 한다.
- 요청 후 상태는 `pickup_requested`다.
- 관리자 승인 전까지 실제 대여 시작은 아니다.

#### 반납 신청

- 본인 예약만 신청할 수 있다.
- `checked_out` 상태만 가능하다.
- 현재 운영 시점이 예약 종료 시각과 정확히 같아야 한다.
- 신청 후 상태는 `return_requested`다.
- 관리자 승인 전까지 실제 반납 완료는 아니다.

### 5-4. 내 상태 조회

- 현재 패널티 점수
- 제한 상태
- 제한 해제일
- 정상 이용 연속 횟수
- 회의실/장비 활성 예약
- 전체 예약 이력 요약
- 최근 이력 일부
- 패널티 이력

## 6. 관리자 시나리오 규칙

### 6-1. 회의실/장비 상태 변경

- 관리자만 수행 가능하다.
- `maintenance` 또는 `disabled`로 변경하면 미래 `reserved` 예약만 자동 취소된다.
- 자동 취소된 예약 상태는 `admin_cancelled`다.
- 자동 취소에는 사용자 패널티가 부여되지 않는다.

### 6-2. 관리자 예약 변경

- 관리자만 수행 가능하다.
- 현재 메뉴 기준으로 `reserved` 상태 예약만 변경 가능하다.
- 새 기간도 예약 날짜 규칙과 충돌 규칙을 다시 통과해야 한다.

### 6-3. 관리자 예약 취소

- 관리자만 수행 가능하다.
- `reserved` 상태 예약만 취소 가능하다.
- 취소 후 상태는 `admin_cancelled`다.
- 사용자 패널티는 부여되지 않는다.

### 6-4. 시작 승인

#### 회의실 체크인 승인

- `checkin_requested` 상태만 가능하다.
- 현재 운영 시점이 예약 시작 시각과 정확히 같아야 한다.
- 승인 후 상태는 `checked_in`이다.

#### 장비 대여 시작 승인

- `pickup_requested` 상태만 가능하다.
- 현재 운영 시점이 예약 시작 시각과 정확히 같아야 한다.
- 승인 후 상태는 `checked_out`이다.

### 6-5. 종료 승인

#### 회의실 퇴실 승인

- `checkout_requested` 상태만 가능하다.
- 현재 운영 시점이 예약 종료 시각과 정확히 같아야 한다.
- 승인 후 상태는 `completed`다.
- 정상 이용으로 기록된다.

#### 장비 반납 승인

- `return_requested` 상태만 가능하다.
- 현재 운영 시점이 예약 종료 시각과 정확히 같아야 한다.
- 승인 후 상태는 `returned`다.
- 정상 이용으로 기록된다.

### 6-6. 노쇼 처리

#### 회의실

- `reserved` 상태만 처리 가능하다.
- 현재 운영 시점이 예약 시작 시각과 정확히 같아야 한다.
- 처리 후 상태는 `no_show`다.
- 노쇼 패널티 `+3점`이 부여된다.

#### 장비

- `reserved` 상태만 처리 가능하다.
- 현재 운영 시점이 예약 시작 시각과 정확히 같아야 한다.
- 처리 후 상태는 `no_show`다.
- 노쇼 패널티 `+3점`이 부여된다.

### 6-7. 지연 종료 처리

#### 회의실

- `checked_in` 상태만 처리 가능하다.
- 현재 운영 시점이 예약 종료 시각과 정확히 같아야 한다.
- 처리 후 상태는 `completed`다.
- 지연 패널티 `+2점`이 부여된다.

#### 장비

- `checked_out` 상태만 처리 가능하다.
- 현재 운영 시점이 예약 종료 시각과 정확히 같아야 한다.
- 처리 후 상태는 `returned`다.
- 지연 패널티 `+2점`이 부여된다.

### 6-8. 수동 패널티 부여

- 관리자만 수행 가능하다.
- 일반 사용자에게만 부여하는 흐름으로 UI가 구성되어 있다.
- 관련 예약 1건을 선택해야 한다.
- 점수는 1~5점이다.
- 사유를 비워두면 기본값 `"파손/오염"`을 사용한다.

## 7. 상태 전이 규칙

### 7-1. 회의실 예약 상태

- 생성: `reserved`
- 사용자 체크인 요청: `reserved -> checkin_requested`
- 관리자 체크인 승인: `checkin_requested -> checked_in`
- 사용자 퇴실 신청: `checked_in -> checkout_requested`
- 관리자 퇴실 승인: `checkout_requested -> completed`
- 관리자 지연 종료 처리: `checked_in -> completed`
- 사용자 취소: `reserved -> cancelled`
- 관리자 취소/자동 취소: `reserved -> admin_cancelled`
- 관리자 노쇼 처리: `reserved -> no_show`

### 7-2. 장비 예약 상태

- 생성: `reserved`
- 사용자 픽업 요청: `reserved -> pickup_requested`
- 관리자 대여 시작 승인: `pickup_requested -> checked_out`
- 사용자 반납 신청: `checked_out -> return_requested`
- 관리자 반납 승인: `return_requested -> returned`
- 관리자 지연 반납 처리: `checked_out -> returned`
- 사용자 취소: `reserved -> cancelled`
- 관리자 취소/자동 취소: `reserved -> admin_cancelled`
- 관리자 노쇼 처리: `reserved -> no_show`

## 8. 패널티 및 제한 규칙

### 8-1. 패널티 점수

- 노쇼: `+3`
- 직전 취소: `+2`
- 지연 퇴실/반납: `+2`
- 파손/오염: `+1~5`

### 8-2. 패널티 발생 시 효과

- 정상 이용 연속 카운트는 `0`으로 리셋된다.
- 누적 점수는 사용자 레코드에 반영된다.

### 8-3. 제한 상태

- `0~2점`: 정상
- `3~5점`이며 제한 기간 유효:
  - 제한 상태
  - 회의실과 장비를 합쳐 활성 예약 총 1건만 허용
- `6점 이상`이며 제한 기간 유효:
  - 이용 금지 상태
  - 신규 예약 불가
  - 미래 `reserved` 예약 자동 취소

### 8-4. 제한 기간

- 3점 이상이 되면 현재 시각 기준 7일 제한이 설정된다.
- 6점 이상이 되면 현재 시각 기준 30일 이용 금지가 설정된다.

### 8-5. 정상 이용 보너스

- 정상 종료 승인 시 정상 이용 1회가 기록된다.
- 정상 이용이 10회 연속이면 패널티 1점이 차감된다.
- 차감 후 연속 카운트는 0으로 초기화된다.

### 8-6. 90일 초기화

- 마지막 패널티 후 90일 이상 지나면 점수를 0으로 초기화한다.
- 이때 제한 상태도 함께 해제된다.

## 9. 운영 시계 차단 규칙

### 9-1. `09:00 -> 18:00` 이동 전 차단

- 시작 시각이 현재 슬롯인 회의실 예약이 `reserved`면 차단된다.
- 시작 시각이 현재 슬롯인 회의실 예약이 `checkin_requested`면 차단된다.
- 시작 시각이 현재 슬롯인 장비 예약이 `reserved`면 차단된다.
- 시작 시각이 현재 슬롯인 장비 예약이 `pickup_requested`면 차단된다.

즉, 시작 슬롯 예약은 다음 중 하나로 정리돼야 한다.

- 사용자 요청 후 관리자 승인 완료
- 관리자 노쇼 처리 완료

### 9-2. `18:00 -> 다음날 09:00` 이동 전 차단

- 종료 시각이 현재 슬롯인 회의실 예약이 `checked_in`이면 차단된다.
- 종료 시각이 현재 슬롯인 회의실 예약이 `checkout_requested`면 차단된다.
- 종료 시각이 현재 슬롯인 장비 예약이 `checked_out`이면 차단된다.
- 종료 시각이 현재 슬롯인 장비 예약이 `return_requested`면 차단된다.

즉, 종료 슬롯 예약은 다음 중 하나로 정리돼야 한다.

- 사용자 신청 후 관리자 승인 완료
- 관리자 지연 종료 처리 완료

## 미정 정책

## 1. 컷오프 시각 정책

- 코드에는 시작 컷오프 `10`, 종료 컷오프 `19` 상수가 존재한다.
- 하지만 운영 시계는 `09:00`, `18:00`만 실제로 사용한다.
- 따라서 현재 구현만 보면 요청/처리 가능 시점은 시작/종료 슬롯 그 자체다.
- 아래 중 무엇이 진짜 정책인지 확정이 필요하다.
  - `10:00`, `19:00`까지의 실제 유예 시간 개념을 도입할지
  - 지금처럼 슬롯 시점에서만 처리하는 구조를 유지할지

## 2. 운영 시계 이동 권한

- 현재는 일반 사용자도 운영 시계를 이동할 수 있다.
- 최종 정책이 아래 중 무엇인지 확정이 필요하다.
  - 관리자만 시점 이동 가능
  - 로그인 사용자 모두 시점 이동 가능
  - 게스트 포함 전원 조회만 가능

## 3. 제한/금지 사용자의 허용 범위

- 현재 제한/금지 사용자는 로그인 자체는 가능하다.
- 신규 예약은 제한되지만, 다른 기능의 허용 범위는 별도 정책으로 고정돼 있지 않다.
- 아래 범위를 어디까지 허용할지 정할 필요가 있다.
  - 체크인/픽업 요청
  - 퇴실/반납 신청
  - 예약 조회
  - 운영 시계 접근

## 4. 자원 비활성화 시 진행 중 예약 처리

- 현재는 미래 `reserved` 예약만 자동 취소된다.
- 이미 `checkin_requested`, `checked_in`, `pickup_requested`, `checked_out` 등 진행 중 상태는 그대로 남는다.
- 아래 중 어떤 정책인지 확정이 필요하다.
  - 진행 중 예약은 유지
  - 진행 중 예약도 강제 종료
  - 상태에 따라 다르게 처리

## 5. 사용자 요청 철회 규칙

- 현재는 체크인 요청, 픽업 요청, 퇴실 신청, 반납 신청 철회 기능이 없다.
- 요청 실수나 관리자 승인 대기 중 취소를 허용할지 확정이 필요하다.

## 6. 관리자 예약 변경 허용 범위

- 현재 관리자 메뉴는 `reserved` 예약 기준으로 변경 흐름을 제공한다.
- 별도 서비스에는 “이미 시작된 예약은 변경 불가”라는 시간형 규칙도 존재한다.
- 최종 정책으로 아래 중 무엇을 채택할지 정리가 필요하다.
  - 관리자도 미래 예약만 변경 가능
  - 관리자에게 더 넓은 예외 권한 부여

## 7. 제한 상태 문구 정리

- 현재 동작은 제한 상태에서 “회의실과 장비를 합쳐 총 1건”이다.
- 하지만 일부 화면 문구는 사용자에게 혼동을 줄 수 있다.
- 최종 문구를 아래처럼 명확히 통일할 필요가 있다.
  - “회의실/장비 합산 활성 예약 1건만 허용”

## 8. 노쇼/지연 종료 자동화 여부

- 현재는 운영자가 직접 노쇼/지연 종료를 처리해야 한다.
- 자동 처리 없이 시점 이동만 차단하는 방식이 최종 정책인지 정할 필요가 있다.

## 9. 관리자 취소 사유 필수 여부

- 현재 관리자 취소는 사유 입력란이 있지만 빈 값도 허용한다.
- 감사 로그 품질을 위해 사유를 필수로 둘지 결정이 필요하다.

## 10. 파손/오염 사유 구분

- enum에는 `damage`, `contamination`이 따로 있다.
- 하지만 현재 관리자 UI는 “파손/오염” 단일 흐름이다.
- 아래 중 무엇을 최종 정책으로 할지 정할 필요가 있다.
  - 파손과 오염을 별도 유형으로 분리
  - 현재처럼 통합 관리
