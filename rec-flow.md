# 추천 사용자 흐름 및 규칙

## 문서 목적

이 문서는 현재 구현을 그대로 설명하는 문서가 아니라,
기획서와 실제 코드를 비교했을 때 더 일관되고 설명 가능한 사용자 흐름과 규칙을 제안하는 문서다.

핵심 목표:

- 사용자 입장에서 배우기 쉬운 흐름
- 입력 문법의 전역 일관성
- 상태 전이의 해석 가능성
- 정책 설명과 실제 동작의 불일치 제거

---

## 1. 추천 방향 요약

현재 시스템은 이미 다음 장점이 있다.

- 게스트/사용자/관리자 역할 분리가 명확하다
- 예약, 패널티, 자동 정책 점검이 분리되어 있다
- 파일 기반 동시성 제어가 들어가 있다

하지만 다음은 정리할 필요가 있다.

- 메인 메뉴 숫자 입력과 목록 선택 입력의 문법 차이
- `시간 예약 시스템`처럼 보이지만 실제 CLI는 `일 단위 예약`이라는 점
- 퇴실/반납이 즉시 완료가 아니라 `신청 -> 승인` 흐름이라는 점
- 취소 문법과 자유 텍스트 규칙의 비일관성

따라서 추천안은 아래 두 가지 중 하나를 명확히 택하는 방향이어야 한다.

1. `일 단위 예약 시스템`으로 명시적으로 고정한다.
2. 아니면 CLI를 수정해서 `시간 단위 예약 시스템`으로 실제 노출 동작을 맞춘다.

현재 코드와 가장 적게 충돌하는 추천은 1번이다.

## 2. 추천 사용자 흐름

## 2.1 제품 설명을 먼저 고정

사용자에게 시스템을 다음처럼 설명하는 것을 권장한다.

> 이 시스템은 회의실과 장비를 `일 단위`로 예약하는 CLI 프로그램이며,
> 실제 사용 시간은 매일 `09:00 ~ 18:00`으로 고정된다.

이 한 문장만 명확해져도 현재 구현과 사용자 기대치의 충돌이 크게 줄어든다.

## 2.2 추천 게스트 흐름

1. 시작 화면
   - 로그인
   - 회원가입
   - 종료
2. 회원가입
   - 사용자명
   - 비밀번호
   - 비밀번호 확인
3. 로그인
   - 사용자명
   - 비밀번호
4. 로그인 직후
   - 현재 상태 요약 1회 출력
   - 역할별 메뉴 진입

추천 이유:

- 로그인 직후 경고 메시지만 따로 뜨는 방식보다
- `현재 패널티 점수 / 제한 여부 / 활성 예약 수`를 짧게 보여주는 것이 더 이해하기 쉽다

## 2.3 추천 일반 사용자 흐름

메뉴 구조를 다음처럼 재구성하는 것을 권장한다.

### 사용자 메인 메뉴

1. 회의실 예약
2. 장비 예약
3. 내 예약 관리
4. 내 상태 조회
0. 로그아웃

### 회의실 예약 서브플로우

1. 이용 인원 입력
2. 시작일, 종료일 입력
3. 조건에 맞는 회의실 목록 확인
4. 회의실 선택
5. 예약 확인
6. 예약 완료

### 장비 예약 서브플로우

1. 장비 종류 선택
2. 시작일, 종료일 입력
3. 가능한 개별 장비 목록 확인
4. 장비 선택
5. 예약 확인
6. 예약 완료

### 내 예약 관리 서브플로우

1. 회의실 예약 보기
2. 장비 예약 보기
3. 예약 변경
4. 예약 취소
5. 퇴실/반납 신청

추천 이유:

- 현재 메뉴는 기능 수가 많아 처음 보는 사용자가 진입점을 판단하기 어렵다
- 예약 생성과 예약 관리가 분리되면 흐름 학습 비용이 줄어든다

## 2.4 추천 관리자 흐름

관리자 메뉴도 작업 성격 기준으로 묶는 것을 권장한다.

### 관리자 메인 메뉴

1. 운영 자원 관리
2. 예약 운영 처리
3. 사용자 관리
0. 로그아웃

### 운영 자원 관리

- 회의실 목록
- 회의실 상태 변경
- 장비 목록
- 장비 상태 변경

### 예약 운영 처리

- 전체 회의실 예약 조회
- 전체 장비 예약 조회
- 체크인/대여 시작 처리
- 퇴실/반납 승인 처리
- 관리자 예약 변경
- 관리자 예약 취소

### 사용자 관리

- 사용자 목록
- 사용자 상세
- 파손/오염 패널티 부여

추천 이유:

- 현재 관리자 메뉴는 17개 단일 목록이라 탐색성이 떨어진다
- 운영 대상과 조작 목적을 분리하면 업무 흐름이 명확해진다

## 3. 추천 문법 규칙

이 절은 전역 공통 문법으로 통일하는 것을 권장하는 규칙이다.

## 3.1 공통 정규화 규칙

모든 입력에 대해 아래 순서를 공통 적용하는 것을 권장한다.

1. 원문 입력 수신
2. 앞뒤 공백 제거
3. 빈 입력 여부 판단
4. 취소 토큰 여부 판단
5. 문법 검사
6. 의미 검사

이 순서를 전역 표준으로 문서화해야 한다.

## 3.2 메뉴 숫자 입력 규칙

추천 규칙:

- 모든 메뉴 입력은 정수 파싱 기반으로 통일
- `"01"`은 `1`로 허용
- 허용 범위 밖 숫자는 에러
- 빈 입력은 에러가 아니라 재입력

권장 메시지:

- 형식 오류: `숫자를 입력해주세요.`
- 범위 오류: `0~N 사이의 번호를 입력해주세요.`

## 3.3 취소 입력 규칙

추천 규칙:

- 모든 입력 화면에서 취소 토큰 공통 허용

권장 토큰:

- `q`
- `quit`
- `취소`

예외:

- 메인 메뉴에서는 `0`을 종료/상위 복귀로 유지해도 된다

## 3.4 확인 입력 규칙

현재 토큰은 충분히 실용적이다.
다만 안내 문구를 항상 동일하게 쓰는 것을 권장한다.

권장 규칙:

- 허용: `y`, `yes`, `예`, `ㅇ`, `n`, `no`, `아니오`, `ㄴ`
- 프롬프트 고정: `(y/n 또는 예/아니오)`

## 3.5 사용자명 규칙

현재 규칙을 유지해도 무방하다.

권장 규칙:

- 길이: `3~20`
- 허용 문자: 영문 대소문자, 숫자, `_`
- 대소문자 구분: 유지
- 앞뒤 공백 제거 후 저장

문서화 예시:

```regex
^[A-Za-z0-9_]{3,20}$
```

추가 권고:

- 예약어 금지 목록을 둘지 결정
  - 예: `admin`, `system`, `root`

## 3.6 비밀번호 규칙

현재 구현은 최소 길이만 보는 매우 약한 규칙이다.

추천 규칙:

- 길이: `8~50`
- 앞뒤 공백 제거
- 내부 공백 허용 여부를 명시
- 최소 조합 규칙은 과제 범위에 따라 선택

가장 무난한 추천:

- 영문 또는 숫자를 하나 이상 포함
- 너무 단순한 비밀번호 차단은 선택 사항

## 3.7 날짜 규칙

현재 예약 흐름을 유지한다면 날짜 규칙을 제품 문법으로 승격시키는 것이 좋다.

권장 규칙:

- 입력 형식: `YYYY-MM-DD`
- 시작일은 내일부터
- 시작일은 오늘 기준 6개월 이내
- 종료일은 시작일 이상
- 기간은 최대 14일

문서 표기:

```text
예약기간 := 시작일 ".." 종료일
시작일 := YYYY-MM-DD
종료일 := YYYY-MM-DD
```

## 3.8 자유 텍스트 규칙

현재 가장 느슨한 부분이라 명시가 필요하다.

권장 규칙:

- 앞뒤 공백 제거
- 공백만 있는 입력은 빈값으로 간주
- 길이 제한 부여
  - 취소 사유: 최대 100자
  - 패널티 사유: 최대 200자
- 줄바꿈 금지

추천 이유:

- 감사 로그 가독성 유지
- JSONL 저장 안정성 향상
- 관리자 입력 품질 개선

## 4. 추천 의미 규칙

## 4.1 제품 정체성 규칙

가장 먼저 아래 둘 중 하나를 명세에서 확정해야 한다.

### 추천안 A: 일 단위 예약으로 확정

- 사용자 입력은 날짜만 받는다
- 매일 `09:00 ~ 18:00` 고정 사용
- 현재 CLI와 가장 잘 맞는다

### 추천안 B: 시간 단위 예약으로 전환

- 사용자 입력은 `시작일시`, `종료일시`
- 30분 단위 예약 허용
- 현재 도메인 API와는 맞지만 CLI를 수정해야 한다

현재 저장소 기준 추천은 `추천안 A`다.

## 4.2 상태 전이 규칙 명문화

현재 코드가 이미 사용 중인 요청 상태를 공식 상태 모델에 포함하는 것을 권장한다.

회의실:

- `reserved`
- `checked_in`
- `checkout_requested`
- `completed`
- `cancelled`
- `no_show`
- `admin_cancelled`

장비:

- `reserved`
- `checked_out`
- `return_requested`
- `returned`
- `cancelled`
- `no_show`
- `admin_cancelled`

추천 이유:

- 사용자 신청 후 관리자 승인이라는 실제 운영 프로세스를 정확히 반영한다
- 지연 계산 기준 시점을 설명하기 쉬워진다

## 4.3 지연 계산 규칙 명시

현재 구현은 타당하다.
이 규칙을 공식 명세로 채택하는 것을 권장한다.

권장 규칙:

- 지연 패널티는 `사용자 신청 시각` 기준으로 계산
- 관리자 승인 지연은 사용자 책임으로 보지 않음

이 규칙을 문서에 반드시 적어야 한다.

## 4.4 제한 상태 규칙 단순화

현재 규칙도 충분히 명확하지만 문장으로 더 단순화하는 것을 권장한다.

권장 문구:

- `0~2점`: 정상 이용
- `3~5점`: 7일간 전체 활성 예약 1건 제한
- `6점 이상`: 30일간 예약 금지, 미래 예약 자동 취소

여기서 `전체 활성 예약`의 정의도 같이 적는다.

활성 예약:

- 회의실: `reserved`, `checked_in`, `checkout_requested`
- 장비: `reserved`, `checked_out`, `return_requested`

## 4.5 관리자 상태 변경 규칙 보강

현재는 미래의 `reserved`만 자동 취소한다.
이 규칙을 유지해도 되지만 설명을 명확히 해야 한다.

권장 명세:

- 자원을 `maintenance` 또는 `disabled`로 바꾸면 미래의 미시작 예약만 자동 취소
- 이미 진행 중인 예약은 별도 운영 판단 대상

추가 권고:

- 관리자가 상태 변경 시 영향을 받는 예약 수를 사전 미리보기로 보여주면 좋다

## 4.6 관리자 취소 규칙 보강

현재는 취소 사유가 선택 사항이다.
실무 흐름을 생각하면 관리자 취소 사유는 필수로 두는 편이 좋다.

추천 규칙:

- 관리자 취소 시 사유 필수
- 길이 `1~100`
- 감사 로그와 사용자 고지 메시지에 동일 사용

## 4.7 문법 오류와 의미 오류 분리

이 규칙은 꼭 채택하는 것이 좋다.

권장 원칙:

- 문법 오류:
  - 형식이 틀림
  - 숫자가 아님
  - 날짜 포맷이 아님
- 의미 오류:
  - 과거 날짜
  - 예약 기간 초과
  - 충돌 발생
  - 권한 없음
  - 제한 상태

권장 효과:

- 사용자 입장에서 왜 실패했는지 더 빨리 이해한다
- 테스트 작성도 쉬워진다

## 5. 추천 명세 초안

현재 코드와 가장 잘 맞는 명세를 한 번에 적으면 아래와 같다.

## 5.1 추천 사용자 흐름 명세

1. 사용자는 로그인 후 `회의실 예약`, `장비 예약`, `내 예약 관리`, `내 상태 조회` 중 하나를 선택한다.
2. 회의실과 장비 예약은 모두 `일 단위 예약`이며, 실제 사용 시간은 매일 `09:00~18:00`으로 고정한다.
3. 사용자는 예약 생성 후 `reserved` 상태를 가진다.
4. 회의실은 관리자 체크인 후 `checked_in`, 사용자 퇴실 신청 후 `checkout_requested`, 관리자 승인 후 `completed`가 된다.
5. 장비는 관리자 대여 시작 후 `checked_out`, 사용자 반납 신청 후 `return_requested`, 관리자 승인 후 `returned`가 된다.
6. 패널티가 `3~5점`이면 전체 활성 예약 1건 제한, `6점 이상`이면 30일간 이용 금지다.

## 5.2 추천 문법 명세

```text
메뉴선택 := 정수
취소토큰 := "q" | "quit" | "취소"
확인입력 := yes-token | no-token
yes-token := "y" | "yes" | "예" | "ㅇ"
no-token := "n" | "no" | "아니오" | "ㄴ"
사용자명 := [A-Za-z0-9_]{3,20}
비밀번호 := 길이 8~50의 문자열
날짜 := YYYY-MM-DD
예약기간 := 시작일, 종료일
양의정수 := min 이상 max 이하의 정수
```

## 5.3 추천 의미 명세

```text
시작일 >= 내일
시작일 <= 오늘 + 6개월
종료일 >= 시작일
예약기간 <= 14일
관리자 취소 사유는 필수
지연 패널티는 사용자 신청 시각 기준
문법 검증 후 의미 검증 수행
```

## 6. 우선순위 추천

가장 먼저 정리할 것을 우선순위로 적으면 다음과 같다.

1. 시스템을 `일 단위 예약`으로 공식 확정할지 결정
2. 요청 상태(`checkout_requested`, `return_requested`)를 공식 상태 모델에 반영
3. 메뉴 숫자 입력 규칙을 전역 통일
4. 취소 토큰을 전역 통일
5. 자유 텍스트 길이와 필수 여부를 명시
6. 문법 오류와 의미 오류를 분리한 메시지 체계로 정리

---

## 7. 결론

현재 코드와 가장 잘 맞는 추천안은 다음이다.

- 제품을 `일 단위 예약 CLI`로 정의한다
- 퇴실/반납은 `사용자 신청 -> 관리자 승인` 2단계 흐름으로 공식화한다
- 입력 문법은 `숫자 입력`, `취소 토큰`, `확인 입력`, `자유 텍스트 길이`를 전역 공통 규칙으로 통일한다
- 상태 전이와 패널티 규칙을 문서에 명시해 사용자 기대와 실제 동작의 차이를 줄인다

## 8. 회의에서 결정해야 할 안건

아래는 구현 정리만으로 끝나지 않고, 팀이 명시적으로 결정해야 하는 안건 목록이다.

| 안건 | 현재 구현 | 추천 방향 |
| --- | --- | --- |
| 예약 단위 | CLI는 일 단위, 도메인엔 시간 단위 API도 존재 | 제품 설명을 일 단위로 고정하거나 CLI를 시간 단위로 개편 |
| 활성 예약 한도 설명 | 정상 시 회의실 3 + 장비 3, 제한 시 전체 1건 | 사용자 안내 문구를 한 문장으로 통일 |
| 동일 사용자 동시 중복 예약 | 허용됨 | 시간대 중복 금지 여부를 결정 |
| 체크인/대여 시작 가능 시점 | 시작 전에도 가능 | 허용 시작 시각 규칙 명시 |
| 조기 퇴실/반납 | 허용됨 | 조기 종료를 공식 기능으로 둘지 결정 |
| 경계값 판정 | 노쇼는 15분 초과, 직전 취소는 60분 미만 | `<`와 `<=` 중 하나로 문구와 코드 통일 |
| 자원 상태 변경 영향 범위 | 미래 `reserved`만 자동 취소 | 진행 중 예약 강제 종료 정책 필요 여부 결정 |
| 관리자 취소 사유 | 선택 입력 | 필수 입력으로 격상 권장 |
| 관리자 계정 운영 | 시드 계정 중심 | 초기 비밀번호 변경, 관리자 추가 절차 결정 |
| 인증 보안 수준 | 평문 저장, 비밀번호 변경 없음 | 과제 범위 내 최소 보안선 결정 |
| 사용자명 정책 | 대소문자 구분, 예약어 금지 없음 | 예약어와 표시명 분리 여부 결정 |
| 자동 정책 점검 시점 | 로그인/메뉴 진입에도 상태 변경 | 조회 시 부수효과 허용 여부 결정 |
| 자유 텍스트 제약 | 길이 제한 없음 | 필드별 최대 길이와 공백 처리 정책 명시 |
| 패널티 사유 체계 | enum은 세분화, UI는 단순화 | 운영/통계 기준 사유 체계 정리 |

## 8.1 바로 회의에 올리기 좋은 질문

1. 이 시스템을 공식적으로 `일 단위 예약`으로 볼 것인가, `시간 단위 예약`으로 확장할 것인가?
2. 한 사용자가 같은 시간대에 회의실 여러 개나 장비 여러 개를 예약하는 것을 허용할 것인가?
3. 체크인/대여 시작은 예약 시작 몇 분 전부터 허용할 것인가?
4. 조기 퇴실/반납은 허용할 것인가, 허용한다면 다음 예약에 영향을 줄 것인가?
5. 정확히 60분 전 취소와 정확히 15분 후 노쇼를 어떻게 판정할 것인가?
6. 자원을 `maintenance`로 바꿀 때 진행 중 이용도 중단시킬 것인가?
7. 관리자 취소 사유와 패널티 사유를 필수 입력으로 만들 것인가?
8. 관리자 계정은 시드 스크립트 외에 어떻게 운영할 것인가?
9. 과제 범위 내에서 비밀번호 저장과 변경 정책을 어디까지 요구할 것인가?
10. 자동 정책 점검이 로그인이나 조회 시점에도 데이터를 바꾸는 현재 구조를 유지할 것인가?
