# KUWorkspace OpenCode 정책 및 동작 명세서

이 문서는 현재 저장소의 **실제 코드베이스**를 기준으로 KUWorkspace CLI의 정책, 입력 문법, 의미 규칙, 상태 전이, 사용자/관리자 흐름, 저장 규칙을 다시 정리한 운영 명세서다. `docs/policy.md`, `docs/rule.md`, `docs/things.md`, `docs/PLAN.md`, `README.md`는 참고 자료일 수 있으나, **현재 동작에 대한 사실 판단은 코드가 우선**한다.

---

## 1. 문서 기준과 해석 규칙

### 1.1 기준 원칙

- 이 문서의 기준은 현재 코드다.
- 문서와 코드가 충돌하면 코드가 현재 동작을 대표한다.
- 테스트와 샘플 데이터는 보조 근거이며, 과거 정책 흔적이 남아 있을 수 있다.
- 이 문서는 다음 세 층위를 구분한다.

| 구분 | 의미 |
| --- | --- |
| 구현 규칙 | 현재 코드가 실제로 강제하는 동작 |
| 내부 동작 | 서비스/저장 계층에는 있으나 사용자 CLI 계약과는 구분해서 봐야 하는 동작 |
| 정책 공백/드리프트 | 코드, 샘플 데이터, 과거 문서 사이에 남아 있는 충돌 또는 미확정 항목 |

### 1.2 주 기준 코드

- 진입/시계: `main.py`, `src/clock_bootstrap.py`, `src/runtime_clock.py`
- 메뉴: `src/cli/guest_menu.py`, `src/cli/user_menu.py`, `src/cli/admin_menu.py`, `src/cli/clock_menu.py`, `src/cli/menu.py`, `src/cli/validators.py`
- 도메인: `src/domain/auth_rules.py`, `src/domain/auth_service.py`, `src/domain/daily_booking_rules.py`, `src/domain/room_service.py`, `src/domain/equipment_service.py`, `src/domain/policy_service.py`, `src/domain/penalty_service.py`, `src/domain/restriction_rules.py`, `src/domain/message_service.py`, `src/domain/models.py`
- 저장/동시성: `src/storage/repositories.py`, `src/storage/jsonl_handler.py`, `src/storage/atomic_writer.py`, `src/storage/file_lock.py`, `src/config.py`

## 2. 용어

- **운영 시계**: 실제 벽시계가 아니라 세션 메모리 안에서 유지되는 가상 시계.
- **운영 슬롯**: 운영 시계가 허용하는 시각 단위. 현재 구현은 `09:00`, `18:00` 두 값만 허용한다.
- **시작 경계**: 예약 `start_time`과 현재 운영 시점이 정확히 같은 시점.
- **종료 경계**: 예약 `end_time`과 현재 운영 시점이 정확히 같은 시점.
- **활성 예약**: 충돌 판정과 예약 한도 계산에 포함되는 예약.
- **제한 사용자**: 패널티 점수와 `restriction_until` 조합상 총 활성 예약 1건만 허용되는 사용자.
- **금지 사용자**: 패널티 점수와 `restriction_until` 조합상 신규 예약이 금지되는 사용자.
- **논리적 append-only**: 도메인 의미상 기존 레코드를 수정/삭제하지 않는다는 뜻. 실제 파일 쓰기는 전체 재작성일 수 있다.
- **전역 락**: `data/.lock` 파일을 이용해 쓰기 경로를 직렬화하는 잠금.
- **UnitOfWork**: 여러 파일 변경을 하나의 원자적 커밋으로 묶는 스테이징 트랜잭션.

## 3. 시스템 개요와 상위 흐름

### 3.1 시스템 성격

- Python 기반 CLI 프로그램이다.
- 데이터베이스를 사용하지 않는다.
- `data/` 아래 JSON Lines 파일을 영속 저장소로 사용한다.
- 다중 인스턴스 실행을 고려해 쓰기 경로는 전역 락과 원자적 파일 교체를 사용한다.

### 3.2 상위 레이어

| 레이어 | 역할 |
| --- | --- |
| CLI 레이어 | 메뉴 출력, 입력 수집, 확인/취소 흐름 |
| 도메인 서비스 레이어 | 인증, 예약, 정책 점검, 패널티, 메시지 처리 |
| 저장소 레이어 | JSONL 읽기/쓰기, 충돌 조회, UnitOfWork 스테이징 |
| 시계 레이어 | 운영 시점 유지, 다음 슬롯 계산 |

### 3.3 프로그램 시작 순서

1. `ensure_data_dir()`가 `data/`와 기본 파일을 준비한다.
2. `prompt_initial_clock()`가 시작 날짜와 시작 슬롯을 입력받는다.
3. 시작 시점은 `get_latest_data_timestamp()`가 반환한 최신 데이터 시각보다 빠를 수 없다.
4. `SystemClock`를 활성화한다.
5. 게스트 메뉴로 진입한다.
6. 로그인 성공 시 `User.role`에 따라 사용자 메뉴 또는 관리자 메뉴로 라우팅한다.
7. 로그아웃하면 다시 게스트 메뉴로 복귀한다.

### 3.4 역할별 상위 분기

| 상태 | 가능 흐름 |
| --- | --- |
| 게스트 | 로그인, 회원가입, 운영 시계 조회, 종료 |
| 일반 사용자 | 회의실/장비 예약 및 요청, 내 상태 조회, 문의/신고, 운영 시계, 로그아웃 |
| 관리자 | 자원 상태 변경, 전체 예약 조회, 승인/노쇼/지연 처리, 사용자 조회, 패널티 부여, 메시지 조회, 운영 시계, 로그아웃 |

| 항목 | 규칙 |

| 오류 처리 | 잘못된 날짜/슬롯/시각이면 재입력 |

## 5. 데이터 모델과 저장 규칙

### 5.1 핵심 열거형 값

| 구분 | 값 |
| --- | --- |
| `UserRole` | `user`, `admin` |
| `ResourceStatus` | `available`, `maintenance`, `disabled` |
| `RoomBookingStatus` | `reserved`, `checkin_requested`, `checked_in`, `checkout_requested`, `completed`, `cancelled`, `no_show`, `admin_cancelled` |
| `EquipmentBookingStatus` | `reserved`, `pickup_requested`, `checked_out`, `return_requested`, `returned`, `cancelled`, `no_show`, `admin_cancelled` |
| `PenaltyReason` | `no_show`, `late_cancel`, `late_return`, `damage`, `contamination`, `other` |
| `MessageType` | `inquiry`, `report` |

### 5.2 데이터 파일 매핑

| 파일 | 저장 객체 | 의미 |
| --- | --- | --- |
| `users.txt` | `User` | 계정, 역할, 누적 패널티, 제한 종료 시각 |
| `rooms.txt` | `Room` | 회의실 메타데이터 |
| `equipment_assets.txt` | `EquipmentAsset` | 장비 자산 메타데이터 |
| `room_bookings.txt` | `RoomBooking` | 회의실 예약 |
| `equipment_bookings.txt` | `EquipmentBooking` | 장비 예약 |
| `penalties.txt` | `Penalty` | 패널티 이력 |
| `audit_log.txt` | `AuditLog` | 감사 로그 |
| `message.txt` | `Message` | 문의/신고 이력 |

### 5.3 저장 형식 규칙

- 모든 파일은 JSON Lines 형식이다.
- 각 줄은 하나의 JSON 객체다.
- 빈 줄은 무시된다.
- 시간 필드는 ISO datetime 문자열이다.

### 5.4 저장 의미와 구현의 차이

- `Penalty`, `AuditLog`, `Message`는 **논리적으로 append-only**다.
- 하지만 Repository는 현재 전체 레코드 목록을 다시 직렬화해 원자적으로 저장할 수 있다.
- 즉, 도메인 의미는 append-only지만 물리적 파일 쓰기 방식은 “파일 끝에 append”로 고정되지 않는다.

### 5.5 락과 트랜잭션

- 모든 쓰기 작업은 전역 락이 잡힌 상태여야 한다.
- 전역 락은 `portalocker` 기반이며 기본 타임아웃은 30초다.
- 락은 같은 프로세스 안에서 재진입 가능하다.
- `UnitOfWork`는 중첩 시 부모 스테이징 버퍼에 합류하고, **최외곽 `UnitOfWork`만 커밋**한다.

### 5.6 원자적 쓰기

다중 파일 커밋은 다음 순서를 따른다.

1. 각 파일 내용을 `*.tmp`에 먼저 기록한다.
2. 기존 파일은 필요 시 `*.bak`으로 백업한다.
3. 모든 `tmp` 파일을 원본으로 교체한다.
4. 실패 시 이미 교체된 파일은 백업에서 복구하거나 새 파일을 삭제한다.
5. 성공 시 백업 파일을 정리한다.

### 5.7 부트스트랩 시각 검사 범위

- 초기 시계 입력 검증에 쓰이는 `get_latest_data_timestamp()`는 `users`, `rooms`, `equipment_assets`, `room_bookings`, `equipment_bookings`, `penalties`, `audit_log`를 읽는다.
- **`message.txt`는 최신 시각 스캔 대상에 포함되지 않는다.**

## 6. 입력 문법 규칙

### 6.1 공통 메뉴/목록 입력

- 메뉴 선택은 숫자 문자열이다.
- 목록 선택은 번호 기반이며 기본적으로 `0`은 취소다.
- 잘못된 번호 입력 시 오류 메시지를 출력한다.

### 6.2 공통 확인 입력

`confirm()`은 다음을 허용한다.

- 참: `y`, `yes`, `예`, `ㅇ`
- 거짓: `n`, `no`, `아니오`, `ㄴ`

단, **문의/신고 제출 확인은 예외**다.

- `UserMenu._submit_message()`는 공통 `confirm()`을 쓰지 않고 문자 그대로 `y` 또는 `n`만 허용한다.

### 6.3 인증 입력

| 항목 | 규칙 |
| --- | --- |
| 사용자명 | trim 후 검사, 3~20자, 영문/숫자/`_`만 허용 |
| 비밀번호 | trim 후 검사, 4~50자 |
| 회원가입 | 비밀번호 확인 일치 필요 |
| 로그인 | 사용자명/비밀번호 둘 다 필수 |

### 6.4 사용자용 일일 예약 입력

사용자 CLI의 기본 예약 흐름은 시간 입력형이 아니라 날짜 범위 입력형이다.

| 항목 | 규칙 |
| --- | --- |
| 시작/종료 형식 | `YYYY-MM-DD` |
| 취소 입력 | `q`, `quit`, `취소` |
| 시작 가능일 | 내일부터 |
| 예약 가능 창 | 시작일 기준 최대 6개월 |
| 최대 기간 | 14일 |
| 실제 저장 시간 | `시작일 09:00` ~ `종료일 18:00` |

### 6.5 숫자 입력

| 항목 | 규칙 |
| --- | --- |
| 회의실 이용 인원 | 1~100 |
| 관리자 수동 패널티 점수 | 1~5 |

### 6.6 내부/보조 시간 입력 규칙

- `src/cli/validators.py`에는 `HH:MM`, 30분 단위, 14일 이내를 검증하는 일반 시간 입력 규칙이 있다.
- 그러나 현재 기본 사용자 예약 UI는 이 경로를 직접 쓰지 않고, 일일 예약 전용 날짜 범위 입력을 사용한다.

## 7. 핵심 의미 규칙

### 7.1 자원 상태 규칙

- 회의실/장비 상태는 `available`, `maintenance`, `disabled`다.
- 신규 예약은 `available` 상태에서만 가능하다.

### 7.2 활성 예약과 충돌 규칙

| 자원 | 활성 예약 상태 |
| --- | --- |
| 회의실 | `reserved`, `checkin_requested`, `checked_in`, `checkout_requested` |
| 장비 | `reserved`, `pickup_requested`, `checked_out`, `return_requested` |

- 충돌은 같은 회의실 ID 또는 같은 장비 ID 기준으로 계산한다.
- 판정식은 `not (end <= existing.start or start >= existing.end)`다.

### 7.3 이용 제한 규칙

| 누적 점수 | 상태 | 효과 |
| --- | --- | --- |
| `0~2` | 정상 | 회의실 1건 + 장비 1건 |
| `3~5` + 제한기간 유효 | 제한 | 전체 활성 예약 합산 1건만 허용 |
| `6+` + 제한기간 유효 | 금지 | 신규 예약 불가, 미래 `reserved` 예약 자동 취소 |

### 7.4 패널티 규칙

| 사유 | 점수 | 구현 함수 |
| --- | --- | --- |
| 노쇼 | `+3` | `PenaltyService.apply_no_show()` |
| 직전 취소 | `+2` | `PenaltyService.apply_late_cancel()` |
| 지연 퇴실/반납 | `+2` | `PenaltyService.apply_late_return()` |
| 파손/오염 | `+1~5` | `PenaltyService.apply_damage()` |

추가 규칙:

- 패널티가 발생하면 `normal_use_streak`는 0으로 리셋된다.
- 마지막 패널티 후 90일이 지나면 누적 점수와 제한 상태를 초기화한다.
- 정상 이용 10회를 연속 달성하면 패널티 1점을 차감하고 연속 카운트를 0으로 리셋한다.

### 7.5 메시지 규칙

| 항목 | 규칙 |
| --- | --- |
| 허용 유형 | `inquiry`, `report` |
| 길이 | 1~100자 |
| 금지 | 빈 문자열, 공백만 입력, 줄바꿈 포함 |

### 8.3 내부 서비스 헬퍼

- `RoomService.check_out()`와 `EquipmentService.return_equipment()`은 직접 종료 헬퍼다.
- 현재 관리자 메뉴의 정상 종료 흐름은 `approve_checkout_request()` / `approve_return_request()`를 사용한다.
- 따라서 **사용자에게 보이는 정상 운영 계약은 요청 후 승인 모델**로 보는 것이 맞다.

## 9. 구현된 CLI 흐름

### 9.1 게스트, 회원가입, 로그인

- 게스트 메뉴 진입 시 `PolicyService.run_all_checks()`를 먼저 수행한다.
- 게스트 메뉴 항목은 로그인, 회원가입, 운영 시계 조회, 종료다.
- 로그인 성공 후 다시 정책 점검을 수행한 뒤 최신 사용자 정보를 읽고, 필요하면 패널티 경고를 출력한다.

### 9.2 사용자 메뉴 공통 규칙

- 사용자 메뉴 루프 시작마다 정책 점검을 수행한다.
- 사용자 메뉴 루프 시작마다 최신 사용자 정보를 다시 읽는다.
- 금지/제한 사용자는 메뉴 진입은 가능하지만 경고를 본다.

### 9.3 회의실 예약 생성/변경/취소

- 생성 전 정책 점검과 최신 사용자 갱신을 수행한다.
- `PolicyService.check_user_can_book()`로 신규 예약 가능 여부를 먼저 판정한다.
- 생성은 참석 인원, 날짜 범위, 회의실 상태, 수용 인원, 활성 예약 한도, 충돌 여부를 모두 통과해야 한다.
- 변경/취소는 UI에서 `reserved` 상태만 노출되고, 서비스가 본인 소유와 `reserved` 상태를 다시 검증한다.
- 직전 취소 판정은 `0 <= 시작까지 남은 분 <= 60`이다. 정확히 60분 전도 패널티 대상이다.

### 9.4 회의실 체크인 요청/퇴실 신청

체크인 요청:

- 본인 예약만 가능
- `reserved` 상태만 가능
- 현재 운영 시점이 예약 시작 시각과 정확히 같아야 함
- 추가로 `현재 시(hour) < 10`

퇴실 신청:

- 본인 예약만 가능
- `checked_in` 상태만 가능
- 현재 운영 시점이 예약 종료 시각과 정확히 같아야 함
- 추가로 `현재 시(hour) < 19`

구현 주의:

- UI는 상태값만 기준으로 선택 목록을 만든다.
- 실제 허용 여부는 서비스의 `_require_start_request_window()` / `_require_end_request_window()`가 최종 판정한다.

### 9.5 장비 예약 생성/변경/취소/요청

- 회의실과 동일한 구조를 따른다.
- 장비는 먼저 `asset_type`을 고른 뒤 개별 자산을 선택한다.
- 충돌은 개별 장비 ID 기준으로 계산한다.
- 픽업 요청과 반납 신청도 각각 시작/종료 경계의 정확한 일치와 `10시/19시` 컷오프 검사를 사용한다.

### 9.6 내 상태 조회

- 누적 패널티 점수, 제한/금지 상태, 제한 해제일, 정상 이용 연속 횟수, 활성 예약, 예약 이력 요약, 최근 패널티 이력을 보여 준다.

### 9.7 문의/신고 제출

- 정책 점검 후 진행한다.
- 메시지 유형은 `inquiry`, `report`만 허용한다.
- 내용은 1~100자, 공백 전용 금지, 줄바꿈 금지다.

### 9.8 관리자 메뉴 공통 규칙

- 관리자 메뉴 루프 시작마다 정책 점검을 수행한다.
- 루프마다 최신 관리자 정보를 읽고 실제 관리자 권한을 다시 검증한다.

### 9.9 관리자 처리 규칙

| 흐름 | 구현 규칙 |
| --- | --- |
| 자원 상태 변경 | `maintenance`/`disabled` 전환 시 미래 `reserved` 예약만 자동 취소 |
| 정상 시작 승인 | 회의실 `checkin_requested -> checked_in`, 장비 `pickup_requested -> checked_out` |
| 정상 종료 승인 | 회의실 `checkout_requested -> completed`, 장비 `return_requested -> returned` |
| 노쇼 처리 | `reserved` + 시작 경계 일치 |
| 지연 종료 처리 | 회의실 `checked_in`, 장비 `checked_out` + 종료 경계 일치 |
| 관리자 시간 변경 | 현재 CLI 기준 `reserved` 예약만 가능 |
| 관리자 취소 | 현재 CLI 기준 `reserved` 예약만 가능 |
| 진행 중 자원 교체 | 회의실은 `checked_in`, 장비는 `checked_out`만 가능 |
| 파손/오염 패널티 | 일반 사용자만 대상으로 UI 제공, 점수 1~5, 빈 사유는 기본값 `파손/오염` |
| 메시지 조회 | 최신순 최대 30건, 상세 조회 가능 |

### 9.10 내부/서비스 전용 동작

- `RoomService.create_booking()` / `EquipmentService.create_booking()`는 임의 datetime 기반 예약을 지원한다.
- 이 경로는 시작/종료가 30분 단위인지, 과거 시간이 아닌지, 시작이 14일 이내인지 등을 직접 검증한다.
- 그러나 현재 기본 사용자 CLI는 이 메서드들을 직접 노출하지 않고, **일일 예약(date-range + 고정 09:00~18:00)** 흐름만 제공한다.

## 10. 자동 정책 점검과 시점 이동 규칙

### 10.1 `run_all_checks()`가 수행하는 일

- 90일 경과 패널티 초기화
- 제한 기간 만료 반영
- 6점 이상 사용자 미래 `reserved` 예약 자동 취소

### 10.2 수행하지 않는 일

- 노쇼 자동 처리
- 지연 종료 자동 처리
- 백그라운드 스케줄러 실행

### 10.3 호출 시점

현재 코드에서 정책 점검은 다음 지점들에서 호출된다.

- 게스트 메뉴 진입 시
- 로그인 성공 직후
- 사용자/관리자 메뉴 루프 시작 시
- 생성/변경/취소/상태 변경 같은 일부 쓰기 서비스 직전

구현 주의:

- 요청/승인/노쇼/지연 종료 메서드는 **서비스 전반에 균일하게** `run_all_checks()`를 호출하지 않는다.
- 현재 설계는 메뉴 루프 진입 시점의 정책 점검에 많이 의존한다.

### 10.4 시점 이동 차단 규칙

| 이동 구간 | 차단 조건 | 해소 방법 |
| --- | --- | --- |
| `09:00 -> 18:00` | 회의실 `reserved`/`checkin_requested`, 장비 `reserved`/`pickup_requested`가 시작 경계에 남아 있음 | 사용자 요청 + 관리자 승인 또는 관리자 노쇼 처리 |
| `18:00 -> 다음날 09:00` | 회의실 `checked_in`/`checkout_requested`, 장비 `checked_out`/`return_requested`가 종료 경계에 남아 있음 | 사용자 신청 + 관리자 승인 또는 관리자 지연 종료 처리 |

### 10.5 시점 이동 결과

- 차단 시 `clock_advance_blocked` 감사 로그를 남긴다.
- 성공 시 운영 시계를 다음 슬롯으로 이동하고 자동 정책 점검을 수행한다.
- 성공 시 `clock_advance` 감사 로그를 남긴다.
- 게스트는 차단 목록만 볼 수 있고 실제 이동은 못 한다.

### 10.6 감사 로그 규칙

- 사용자 예약 생성/변경/취소/요청은 감사 로그를 남긴다.
- 관리자 승인, 노쇼 처리, 지연 종료 처리, 자원 상태 변경, 재배정/교체, 수동 패널티 부여도 감사 로그를 남긴다.
- 정책 자동 처리(제한 만료, 90일 초기화, 금지 사용자 미래 예약 취소)도 시스템 actor로 감사 로그를 남긴다.
- 감사 로그는 논리적으로 append-only이며, 물리적 저장은 원자적 전체 재작성일 수 있다.

## 11. 구현 드리프트와 문서 충돌

### 11.1 컷오프 시각과 슬롯 시계의 충돌

- 코드에는 `START_REQUEST_CUTOFF_HOUR = 10`, `END_REQUEST_CUTOFF_HOUR = 19`가 있다.
- 그러나 운영 시계는 `09:00`, `18:00`만 허용한다.
- 요청 메서드는 경계 시각의 **정확한 일치**를 먼저 요구하므로, 현재 CLI 시계 모델에서는 `10:00`/`19:00` 유예 시간이 실질적으로 실행되지 않는다.

### 11.2 지연 패널티 계산 드리프트

- 현재 구현의 지연 패널티는 고정 `+2`다.
- 과거 자료나 샘플 데이터의 `ceil(delay/10)` 또는 가변 점수 흔적은 현재 코드와 맞지 않는다.

### 11.3 일반 사용자의 시계 이동 권한

- 현재 CLI는 게스트만 read-only이고, 로그인한 일반 사용자와 관리자 모두 시계를 이동할 수 있다.
- “관리자만 시점 이동”은 현재 구현 규칙이 아니다.

### 11.4 자원 재배정 노출 범위

- 현재 관리자 메뉴는 **회의실 재배정과 장비 교체 둘 다 노출**한다.
- “회의실 재배정이 메뉴에 연결되지 않았다”는 설명은 현재 코드 기준으로는 stale하다.

### 11.5 사용자 선택 목록과 실제 허용 여부의 차이

- 사용자 메뉴의 체크인/픽업/퇴실/반납 목록은 주로 상태값으로 구성된다.
- 따라서 목록에 보여도 실제 서비스 호출 시 경계 시각/컷오프 검증으로 실패할 수 있다.

### 11.6 샘플 데이터와 과거 자동 노쇼 흔적

- 저장소 산출물에는 `auto_no_show_*` 같은 과거 흔적이 남아 있을 수 있다.
- 현재 코드의 정책 서비스는 노쇼를 자동 처리하지 않는다.

## 12. 추가로 정책을 명문화해야 하는 항목

현재 코드만으로는 최종 정책이라고 단정하기 어려운 항목들이다.

1. **제한/금지 사용자의 허용 범위**
   - 현재는 신규 예약만 직접 제한된다.
   - 로그인, 조회, 체크인/픽업 요청, 퇴실/반납 신청, 운영 시계 접근은 별도 금지 규칙이 없다.

2. **요청 철회 규칙**
   - 체크인 요청, 픽업 요청, 퇴실 신청, 반납 신청 철회 기능이 없다.

3. **시간형 서비스 메서드의 공개 계약 여부**
   - 서비스 계층에는 30분 단위 임의 시각 예약 메서드가 남아 있지만, 기본 CLI는 일일 예약만 노출한다.

4. **관리자 취소 사유의 필수 여부**
   - 현재는 빈 문자열도 허용한다.

5. **`damage`와 `contamination`의 구분 정책**
   - enum은 분리돼 있지만 현재 관리자 UI는 사실상 통합 흐름이다.

6. **timezone 정책**
   - 현재 구현은 naive datetime만 사용한다.

7. **패널티 파일과 `User.penalty_points` 불일치 시 복구 규칙**
   - 현재 코드는 두 값을 함께 갱신하려고 하지만, 불일치 복구 정책은 문서화돼 있지 않다.

8. **부트스트랩 최신 시각 검사 범위**
   - 현재는 메시지 파일이 제외되는데, 이것이 의도인지 확정이 필요하다.

9. **메시지 작성 시 사용자 존재 검증 범위**
   - 현재 CLI는 로그인 사용자의 ID만 전달하므로 정상 흐름에서는 문제없다.
   - 하지만 `MessageService.create_message()` 자체는 `user_id`의 실존 여부를 별도로 검증하지 않는다.

## 13. 문서 작성 시 해석 우선순위

이 문서를 유지보수할 때는 다음 우선순위를 따른다.

1. 현재 코드
2. 코드가 실제 호출하는 현재 CLI 흐름
3. 현재 테스트가 검증하는 동작
4. 과거 문서, README, 샘플 데이터

이 우선순위를 지키지 않으면, 실행 가능한 현재 동작과 문서가 다시 어긋나게 된다.
