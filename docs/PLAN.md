## 1. 제품 개요

### Summary

- 구현 형태는 `Python 기반 CLI 메뉴형 프로그램`으로 확정한다.
- 저장소는 `DBMS 없이 여러 개의 .txt 파일(JSON Lines 형식)`을 사용한다.
- 회의실 예약과 장비 예약은 메뉴와 데이터 파일을 분리해 독립 운영하되, 인증, 패널티, 이용 제한, 운영 시계 정책은 공통 로직으로 묶는다.
- 시스템 시간은 실제 벽시계가 아니라 `세션 단위 가상 시계`를 기준으로 동작한다.
- 운영 시점은 `09:00`, `18:00` 두 슬롯만 허용하며, 시점 이동은 메뉴에서 명시적으로 수행한다.
- 동시 실행 인스턴스를 허용하므로 모든 쓰기 작업은 `전역 잠금 파일 + 원자적 파일 교체` 방식으로 처리한다.

### 기술 및 구조

- 언어: `Python`
- UI: `CLI 메뉴형`
- 저장 방식: `JSON Lines .txt`
- 쓰기 안전성: 전역 잠금 + `*.tmp` 저장 후 원자적 교체
- 레이어 구조
  - `CLI 입출력`
  - `도메인 서비스(auth / reservation / policy / penalty)`
  - `저장소(JSONL 파일 처리)`

### 주요 파일 인터페이스

- `users.txt`: 사용자 계정, 역할, 패널티 점수, 정상 이용 연속 횟수, 제한 종료 시각
- `rooms.txt`: 회의실 기본 정보와 운영 상태
- `equipment_assets.txt`: 개별 장비 자산 정보와 운영 상태
- `room_bookings.txt`: 회의실 예약 레코드
- `equipment_bookings.txt`: 장비 예약 레코드
- `penalties.txt`: 패널티 이력
- `audit_log.txt`: 관리자 조작, 시점 이동, 자동 정책 처리 로그

## 2. 사용자 흐름

### 전체 진입 흐름

```text
프로그램 시작
-> 데이터 파일 준비
-> 운영 시작 시점 입력
-> 게스트 메뉴 진입
-> 로그인 성공
-> 역할별 메뉴 분기
-> 로그아웃
-> 게스트 메뉴 복귀
-> 종료
```

### 운영 시작 시점

- 프로그램 시작 시 `YYYY-MM-DD`와 시작 슬롯(`09:00` 또는 `18:00`)을 입력받는다.
- 새 시작 시점은 기존 `data/`에 저장된 최신 시각보다 빠를 수 없다.
- 운영 시점은 세션 메모리 전용이며 프로그램 재시작 후 유지되지 않는다.

### 게스트 흐름

- 가능 기능
  - 로그인
  - 회원가입
  - 운영 시계
  - 종료
- 게스트 상태에서는 운영 시계 메뉴에 접근할 수 있지만 조회만 가능하다.

### 일반 사용자 흐름

- 가능 기능
  - 회의실 목록 조회
  - 회의실 예약 / 변경 / 취소 / 내 예약 조회
  - 회의실 퇴실 신청
  - 장비 목록 조회
  - 장비 예약 / 변경 / 취소 / 내 예약 조회
  - 장비 반납 신청
  - 내 상태 조회
  - 운영 시계
  - 로그아웃
- 사용자 메뉴 진입 전과 주요 작업 직전에는 정책 점검 루틴을 실행한다.
- 사용자는 운영 시계 메뉴에 접근하고 실제 시점 이동을 수행할 수 있다.

### 관리자 흐름

- 가능 기능
  - 유저 목록 및 상태 조회
  - 회의실 / 장비 목록 조회와 운영 상태 변경
  - 전체 예약 조회 및 미래 예약 변경 / 취소
  - 회의실 체크인 승인
  - 장비 대여 시작 승인
  - 회의실 퇴실 승인
  - 장비 반납 승인
  - 회의실 / 장비 노쇼 수동 처리
  - 회의실 / 장비 지연 종료 처리
  - 파손 / 오염 패널티 부여
  - 운영 시계
  - 로그아웃

### 운영 시계 흐름

- 시점 이동 순서는 `09:00 -> 18:00 -> 다음날 09:00`만 허용한다.
- 여러 슬롯을 한 번에 건너뛰는 이동은 허용하지 않는다.
- 정책 판정용 cutoff 시각은 별도로 둔다.
  - 시작 cutoff: `10:00`
  - 종료 cutoff: `19:00`
- 시작 경계(`09:00`)에서는 사용자가 `체크인 요청 / 픽업 요청`을 남기고, 관리자가 이를 승인해 실제 이용 상태로 전환한다.
- `10:00`까지 시작 요청이 없는 예약은 사용자가 더 이상 시작 요청할 수 없고, 관리자가 `노쇼 처리`해야 다음 시점 이동 블로킹을 해제할 수 있다.
- 종료 경계(`18:00`)에서는 사용자가 `퇴실 / 반납 요청`을 남기고, 관리자가 이를 승인해 종료한다.
- `19:00`까지 종료 요청이 없는 예약은 사용자가 더 이상 종료 요청할 수 없고, 관리자가 `지연 종료 처리`와 패널티 부여를 수행해야 다음 시점 이동 블로킹을 해제할 수 있다.
- 관리자 미승인 시작 요청(`checkin_requested`, `pickup_requested`) 또는 종료 요청(`checkout_requested`, `return_requested`)이 남아 있으면 다음 시점으로 이동할 수 없다.
- 시점 이동 성공 / 차단 이벤트는 모두 화면과 감사 로그에 남긴다.
- 사용자 또는 관리자가 시점 이동을 수행하면 감사 로그의 행위자는 해당 사용자 ID로 기록한다.

## 3. 정책 명세

### 예약 규칙

- 예약 UI는 시간 입력형이 아니라 날짜 범위 기반이다.
- 실제 이용 구간은 회의실과 장비 모두 `시작일 09:00 ~ 종료일 18:00`으로 해석한다.
- 예약은 조건 충족 시 즉시 확정한다. 승인 대기 상태는 두지 않는다.
- 예약 시작일은 `내일부터` 선택할 수 있다.
- 예약 가능 기간은 시작일 기준 `최대 6개월 이내`다.
- 1회 예약 기간은 `최대 14일`이다.
- 정상 사용자 예약 한도는 `회의실 1건`, `장비 1건`이다.
- 장비는 `개별 자산 단위`로 관리한다. 같은 종류라도 각 장비는 고유 ID를 가진다.

### 자원 운영 상태

- 회의실 / 장비 운영 상태는 `available`, `maintenance`, `disabled`로 둔다.
- `maintenance` 또는 `disabled` 상태인 자원은 신규 예약이 불가하다.
- 자원 상태가 `maintenance` 또는 `disabled`로 변경되면 관련 미래 예약은 `admin_cancelled`로 전환한다.
- 시스템 사유 취소에는 사용자 패널티를 부여하지 않는다.
- 실제 점유 여부는 자원 자체 상태로 저장하지 않고 예약 데이터와 현재 시각으로 계산한다.

### 예약 상태 모델

- 회의실 예약 상태
  - `reserved`
  - `checkin_requested`
  - `checked_in`
  - `checkout_requested`
  - `completed`
  - `cancelled`
  - `no_show`
  - `admin_cancelled`
- 장비 예약 상태
  - `reserved`
  - `pickup_requested`
  - `checked_out`
  - `return_requested`
  - `returned`
  - `cancelled`
  - `no_show`
  - `admin_cancelled`

### 상태 전이

- 회의실
  - 생성: `reserved`
  - 사용자 체크인 요청: `reserved -> checkin_requested`
  - 관리자 체크인 승인: `checkin_requested -> checked_in`
  - 사용자 퇴실 신청: `checked_in -> checkout_requested`
  - 관리자 퇴실 승인: `checkout_requested -> completed`
  - 관리자 지연 종료 처리: `checked_in -> completed`
  - 사용자 취소: `reserved -> cancelled`
  - 관리자 취소: `reserved -> admin_cancelled`
  - 관리자 수동 노쇼 처리: `reserved -> no_show`
- 장비
  - 생성: `reserved`
  - 사용자 픽업 요청: `reserved -> pickup_requested`
  - 관리자 대여 시작 승인: `pickup_requested -> checked_out`
  - 사용자 반납 신청: `checked_out -> return_requested`
  - 관리자 반납 승인: `return_requested -> returned`
  - 관리자 지연 종료 처리: `checked_out -> returned`
  - 사용자 취소: `reserved -> cancelled`
  - 관리자 취소: `reserved -> admin_cancelled`
  - 관리자 수동 노쇼 처리: `reserved -> no_show`

### 변경 / 종료 권한

- 사용자
  - `reserved` 상태 예약만 변경 / 취소 가능
  - 회의실은 `reserved` 상태에서 시작 cutoff 전까지 체크인 요청 가능
  - 장비는 `reserved` 상태에서 시작 cutoff 전까지 픽업 요청 가능
  - 회의실은 `checked_in` 상태에서만 퇴실 신청 가능
  - 장비는 `checked_out` 상태에서만 반납 신청 가능
  - 종료 cutoff 이후에는 더 이상 퇴실 / 반납 요청 가능하지 않다
- 관리자
  - 미래 시점의 `reserved` 예약만 시간 변경 가능
  - 체크인 승인 / 대여 시작 승인 / 퇴실 승인 / 반납 승인 / 노쇼 판정 가능
  - 종료 요청이 없는 진행 중 예약에 대해 지연 종료 처리 가능
- 정상 시작은 `사용자 요청 + 관리자 승인`으로만 허용한다.
- 정상 종료는 `사용자 신청 + 관리자 승인`으로만 허용한다.
- 단, 종료 cutoff 이후 요청이 없는 진행 중 예약은 관리자가 지연 종료 처리로 마감할 수 있다.

### 패널티 정책

- 노쇼
  - 자동 판정하지 않는다.
  - 예약 시작 후 `1시간`이 지나 시작 cutoff(`10:00`)까지 체크인 요청 / 픽업 요청이 없으면 노쇼 처리 대상이 된다.
  - cutoff 이후에는 사용자가 더 이상 시작 요청할 수 없다.
  - 관리자가 수동으로 노쇼 처리하면 `+3점`을 부여한다.
- 직전 취소
  - 예약 시작 1시간 이내에 사용자가 취소하면 `+2점`을 부여한다.
  - 관리자 취소와 시스템 취소에는 패널티를 부여하지 않는다.
- 지연 퇴실 / 반납
  - 예약 종료 후 `1시간`이 지나 종료 cutoff(`19:00`)까지 퇴실 / 반납 요청이 없으면 지연 종료 처리 대상이 된다.
  - cutoff 이후에는 사용자가 더 이상 퇴실 / 반납 요청할 수 없다.
  - 관리자가 지연 종료 처리하면 고정 `+2점`을 부여한다.
  - 지연 패널티 기준은 관리자 승인 시각이 아니라 사용자 요청 유무와 cutoff 경과 여부다.
- 파손 / 오염
  - 관리자 수동 입력만 허용
  - `1~5점`

### 이용 제한 정책

- 누적 점수 `< 3점`: 정상 이용
- 누적 점수 `3점 이상 6점 미만`: 향후 7일간 `전체 활성 예약 1건만 허용`
- 누적 점수 `6점 이상`: 향후 30일간 이용 불가, 미래 예약 자동 취소
- 마지막 패널티 발생일로부터 90일이 지나면 누적 점수 초기화
- 정상 이용 10회를 연속 달성하면 패널티 점수 `1점 차감`하고 연속 카운트는 `0`으로 리셋

### 자동 처리 정책

- 백그라운드 서버는 두지 않는다.
- 정책 점검 루틴은 다음 시점에 실행한다.
  - 앱 시작 시
  - 로그인 직후
  - 예약 생성 / 변경 / 취소 직전
  - 관리자 예약 관리 메뉴 진입 시
- 점검 루틴은 다음 작업을 수행한다.
  - `90일 경과 패널티 초기화`
  - `제한 기간 만료 반영`
  - `6점 이상 사용자 미래 예약 자동 취소`
- 노쇼와 지연 종료는 자동 확정하지 않는다.

## 4. 동시성 / 저장 설계

### 잠금 전략

- `data/.lock` 같은 전역 잠금 파일 1개를 둔다.
- 다음 작업은 모두 전역 잠금 아래에서 수행한다.
  - 예약 생성
  - 예약 변경
  - 예약 취소
  - 체크인 / 대여 시작
  - 퇴실 승인 / 반납 승인
  - 퇴실 신청 / 반납 신청
  - 패널티 부여
  - 자원 상태 변경
  - 자동 정책 점검
  - 운영 시점 이동
- 단순 조회는 락 없이 읽되, 쓰기 직전에는 최신 파일을 다시 읽어 재검증한다.

### 쓰기 절차

1. 전역 락 획득
2. 관련 txt 파일 전체 로드
3. 메모리에서 정책 검증 및 수정
4. 각 파일을 `*.tmp`에 저장
5. 검증 완료 후 원본 파일로 원자적 교체
6. 감사 로그까지 저장한 뒤 락 해제

### 충돌 처리

- 같은 자원에 대해 동시 예약이 들어오면 먼저 락을 잡은 요청만 성공한다.
- 뒤늦게 들어온 요청은 최신 상태 재검증에서 충돌을 감지하고 즉시 실패 처리한다.
- 사용자에게는 최신 상태 재조회가 필요하다는 메시지를 보여준다.

### 상태 계산 원칙

- `user_status`, `in_use`, `checked_out` 같은 중복 상태는 별도 저장하지 않는다.
- 이용 제한 여부는 `penalty_points`, `restriction_until`, 현재 시각으로 계산한다.
- 회의실 점유 여부는 `checked_in`, `checkout_requested` 상태와 현재 시각으로 계산한다.
- 장비 점유 여부는 `checked_out`, `return_requested` 상태와 현재 시각으로 계산한다.

### 감사 로그

- 상태 없음, `append-only`
- 다음 이벤트를 모두 남긴다.
  - 관리자 조작
  - 사용자 예약 상태 변경
  - 운영 시점 이동
  - 시점 이동 차단
  - 자동 정책 처리

## 5. 테스트 기준

### 흐름 검증

- 회원가입 후 로그인, 역할별 메뉴 분기, 로그아웃 복귀
- 운영 시작 시점 입력 검증
  - 날짜 형식 오류 차단
  - `09:00` / `18:00` 외 슬롯 차단
  - 기존 저장 데이터 최신 시각보다 이른 시작 시점 차단
- 게스트는 운영 시계 조회만 가능하고, 일반 사용자 / 관리자는 실제 이동 가능 여부 확인
- 시점 이동 순서가 `09:00 -> 18:00 -> 다음날 09:00`만 허용되는지 확인

### 예약 및 상태 전이 검증

- 회의실 예약 충돌 검사
- 장비 예약 충돌 검사
- 정상 사용자 예약 한도 `회의실 1건 + 장비 1건` 적용 확인
- `reserved` 상태에서만 사용자 변경 / 취소가 가능한지 확인
- 회의실 `reserved -> checkin_requested -> checked_in -> checkout_requested -> completed` 흐름 확인
- 장비 `reserved -> pickup_requested -> checked_out -> return_requested -> returned` 흐름 확인
- 정상 시작은 사용자 요청 후 관리자 승인으로만 가능한지 확인
- 정상 종료는 사용자 신청 후 관리자 승인으로만 완료되는지 확인
- 시작 / 종료 cutoff 이후에는 사용자가 더 이상 요청할 수 없는지 확인
- 관리자 미승인 시작 요청 / 종료 요청이 남아 있으면 시점 이동이 차단되는지 확인

### 패널티 및 제한 검증

- 시작 cutoff 경과 후 요청 없는 예약이 노쇼 처리 대상이 되는지 확인
- 관리자 수동 노쇼 처리 시 `+3점` 반영 확인
- 직전 취소 시 `+2점` 반영 확인
- 종료 cutoff 경과 후 요청 없는 진행 중 예약에 대해 관리자 지연 종료 처리와 `+2점` 반영 확인
- 파손 / 오염 수동 패널티 반영 확인
- 패널티 `3~5점` 구간에서 전체 활성 예약 1건 제한 확인
- 패널티 `6점 이상` 구간에서 30일 이용 제한 및 미래 예약 자동 취소 확인
- 마지막 패널티 후 90일 경과 시 점수 초기화 확인
- 정상 이용 10회 연속 시 1점 차감 확인

### 동시성 및 저장 검증

- 두 인스턴스가 동시에 같은 회의실 또는 같은 장비를 예약할 때 한쪽만 성공하는지 확인
- 예약 생성 중 프로그램이 중단되어도 원본 txt가 손상되지 않는지 확인
- 자원 상태 변경 시 미래 예약 자동 취소와 무패널티 처리 확인
- 모든 쓰기 작업이 감사 로그와 함께 원자적으로 반영되는지 확인

## Assumptions

- 과제 범위에서는 비밀번호를 평문 저장한다.
- 현재 범위에서는 `deleted`, `archived`, `overdue` 같은 추가 상태를 두지 않는다.
- `flow.md`는 과거 구현 기록용으로만 참고하고 정책/우선순위는 `PLAN.md`가 가진다.
