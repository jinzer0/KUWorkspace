# 공유 오피스 예약 및 장비 대여 관리 CLI 프로그램 기획안

## Summary
- 구현 형태는 `Python 기반 CLI 메뉴형 프로그램`으로 확정한다. 웹/GUI/TUI는 사용하지 않고, 한 번 실행하면 로그인 후 역할별 메뉴를 따라가는 구조로 설계한다.
- 저장소는 `DBMS 없이 여러 개의 .txt 파일(JSON Lines 형식)`을 사용한다. 회의실 예약과 장비 예약은 파일과 메뉴를 분리해 독립 운영하되, 인증, 패널티, 이용 제한 정책은 공통 로직으로 묶는다.
- 시스템 시간은 실제 벽시계가 아니라 `세션 단위 가상 시계`를 기준으로 동작한다. 운영 시점은 `09:00`, `18:00` 두 슬롯만 허용되며, 공용 시계 메뉴에서만 다음 시점으로 이동할 수 있다.
- 동시 실행 인스턴스를 허용하므로, 모든 쓰기 작업은 `전역 잠금 파일 + 원자적 파일 교체` 방식으로 처리한다. 외부 라이브러리는 `portalocker` 1개만 사용하고, 나머지는 표준 라이브러리로 구성한다.

## Key Changes
- 기술 스택
  - 언어: `Python`
  - UI: `CLI 메뉴형`
  - 라이브러리: 표준 라이브러리 중심, `portalocker`만 추가 사용
  - 출력 방식: 단순 번호 메뉴 + 고정폭 텍스트 표, 별도 UI 컴포넌트 라이브러리 미사용
- 프로그램 구조
  - `main.py`에서 시작해 운영 시작 시점을 먼저 입력받는다.
  - 이후 `비회원 메뉴(회원가입/로그인/운영 시계/종료)`로 진입한다.
  - 로그인 후 `사용자 메뉴`와 `관리자 메뉴`를 분기한다.
  - 내부 레이어는 `CLI 입출력`, `도메인 서비스(auth/reservation/policy/penalty)`, `저장소(JSONL 파일 처리)` 3층으로 나눈다.
- 사용자 기능
  - 회원가입, 로그인
  - 회의실 목록 조회, 회의실 예약 신청/변경/확인/취소
  - 장비 목록 조회, 장비 예약 신청/변경/확인/취소
  - 체크인 이후 회의실 퇴실 신청, 장비 반납 신청
  - 내 상태 조회: 현재 패널티 점수, 제한 여부, 활성 예약, 예약 이력 요약
  - 운영 시계 메뉴에서 현재 시점, 다음 시점, 미해결 사건 조회
- 관리자 기능
  - 유저 목록 및 상태 조회
  - 회의실 / 장비 목록 조회와 운영 상태 변경
  - 전체 예약 조회 및 변경/취소
  - 회의실 체크인, 퇴실 승인, 장비 대여 시작, 반납 승인
  - 회의실 / 장비 노쇼 수동 처리
  - 파손/오염 패널티 확정 처리
  - 운영 시계 메뉴에서 시점 이동과 차단 사유 확인
- 예약 정책
  - 예약 UI는 시간 입력형이 아니라 날짜 범위 기반이다.
  - 실제 이용 구간은 회의실/장비 모두 `시작일 09:00 ~ 종료일 18:00`으로 해석한다.
  - 예약은 조건 충족 시 즉시 확정한다. 승인 대기 상태는 두지 않는다.
  - 예약 시작일은 `내일부터` 선택할 수 있다.
  - 예약 가능 기간은 시작일 기준 `최대 6개월 이내`다.
  - 1회 예약 기간은 `최대 14일`이다.
  - 정상 사용자 예약 한도는 현재 사용자 흐름 기준으로 `회의실 1건`, `장비 1건`으로 본다.
  - 장비는 `개별 자산 단위`로 관리한다. 같은 종류라도 각 장비는 고유 ID를 가진다.
  - 회의실/장비 운영 상태는 `available`, `maintenance`, `disabled`로 두고, 실제 점유 여부는 예약 데이터로 계산한다.
- 시간 및 시점 관리 규칙
  - 프로그램 시작 시 운영 시작 시점을 `YYYY-MM-DD`와 슬롯(`09:00` 또는 `18:00`)으로 입력받는다.
  - 운영 시점은 세션 메모리 전용이며 프로그램 재시작 후 유지되지 않는다.
  - 단, 새 시작 시점은 기존 `data/`에 저장된 최신 시각보다 빠를 수 없다.
  - 시점 이동은 `09:00 -> 18:00 -> 다음날 09:00` 순서만 허용한다.
  - 여러 슬롯을 한 번에 건너뛰는 이동은 허용하지 않는다.
  - 시작 경계(`09:00`)에서는 해당 시점에 시작하는 예약을 관리자가 `체크인/대여 시작` 또는 `노쇼 처리`로 명시적으로 해결해야 한다.
  - 종료 경계(`18:00`)에서는 해당 시점에 끝나는 예약에 대해 사용자의 `퇴실/반납 신청`과 관리자의 `승인`이 모두 끝나야 다음 시점으로 이동할 수 있다.
  - 시점 이동이 차단되면 시스템은 미해결 예약 목록을 화면에 보여주고 감사 로그에도 남긴다.
  - 시점 이동이 성공하면 이벤트 요약을 화면에 보여주고 감사 로그에도 남긴다.
- 패널티 및 이용 제한
  - 노쇼: 관리자 수동 노쇼 처리 시 `+3점`
  - 파손/오염: 자동 즉시 제한은 없애고, 관리자가 검토 후 `1~5점` 수동 부여
  - 직전 취소 패널티는 현재 운영 정책에서 사용하지 않는다.
  - 분 단위 지연 퇴실 / 반납 벌점은 현재 운영 정책에서 사용하지 않는다.
  - 누적 점수 `< 3점`: 사용 가능, 경고 메시지 표시 가능
  - 누적 점수 `3점 이상 6점 미만`: 향후 7일간 `전체 시스템 기준 활성 예약 1건만 허용`
  - 누적 점수 `6점 이상`: 향후 30일간 이용 불가, 미래 예약 자동 취소
  - 마지막 패널티 발생일로부터 90일이 지나면 누적 점수 초기화
  - 정상 이용 10회를 연속 달성하면 패널티 점수 `1점 차감`하고 연속 카운트는 `0`으로 리셋
- 자동 처리 방식
  - 백그라운드 서버는 두지 않는다.
  - 정책 점검 루틴은 앱 시작 시, 로그인 직후, 예약 생성/변경/취소 직전, 관리자 예약 관리 메뉴 진입 시 실행한다.
  - 이 루틴은 `90일 경과 패널티 초기화`, `제한 기간 만료 반영`, `6점 이상 사용자 미래 예약 자동 취소`를 수행한다.
  - 노쇼는 자동 판정하지 않고, 운영 시계 이동 시 블로커로 노출한 뒤 관리자가 수동 처리한다.
- 파일 인터페이스와 데이터 타입
  - `users.txt`: 사용자 계정, 역할, 패널티 점수, 연속 정상 이용 횟수, 제한 종료일
  - `rooms.txt`: 회의실 기본 정보와 운영 상태
  - `equipment_assets.txt`: 개별 장비 자산 정보와 운영 상태
  - `room_bookings.txt`: 회의실 예약 레코드
  - `equipment_bookings.txt`: 장비 예약 레코드
  - `penalties.txt`: 패널티 이력 레코드
  - `audit_log.txt`: 관리자 조작, 시점 이동, 자동 정책 처리 로그
  - 모든 파일은 JSON Lines `.txt` 형식으로 저장하고, 각 레코드는 고유 ID와 생성/수정 시각을 가진다.
- 주요 상태값
  - 회의실 예약: `reserved`, `checked_in`, `checkout_requested`, `completed`, `cancelled`, `no_show`, `admin_cancelled`
  - 장비 예약: `reserved`, `checked_out`, `return_requested`, `returned`, `cancelled`, `no_show`, `admin_cancelled`
  - 회의실 / 장비 운영 상태: `available`, `maintenance`, `disabled`
  - 사용자 역할: `user`, `admin`

## Brainstorming Notes
- 현재 구현은 가상 시계, 경계 시점 강제, 수동 노쇼, 시점 이동 블로킹을 기준으로 안정화되어 있다.
- 정상 사용자 기본 예약 한도도 코드와 문서 모두 `회의실 1건 + 장비 1건`으로 일치시켰다.
- 제한 사용자(`3점 이상 6점 미만`)는 자원 종류와 무관하게 전체 활성 예약 1건만 유지할 수 있다.
- 이후 문서/흐름 점검에서는 이 `1+1` 한도를 전제로 예시와 테스트 설명을 유지하면 된다.

## Test Plan
- 회원가입 후 로그인, 역할별 메뉴 분기, 잘못된 로그인 차단
- 운영 시작 시점 입력 검증
  - 날짜 형식 오류 차단
  - `09:00` / `18:00` 외 슬롯 차단
  - 기존 저장 데이터 최신 시각보다 이른 시작 시점 차단
- 시점 이동 규칙 검증
  - `09:00 -> 18:00`, `18:00 -> 다음날 09:00` 순서만 허용
  - 다중 점프 불가
  - 시작 경계 미처리 예약이 있으면 이동 차단
  - 종료 경계 미완료 신청/승인이 있으면 이동 차단
- 회의실 예약 충돌 검사: 같은 기간 중복 예약 차단
- 장비 예약 충돌 검사: 동일 장비 중복 예약 차단
- 정상 사용자 예약 한도 적용 확인: 현재 사용자 흐름 기준 회의실 1건, 장비 1건
- 패널티 3점 도달 시 7일간 활성 예약 1건 제한 확인
- 패널티 6점 도달 시 30일 이용 제한 및 미래 예약 자동 취소 확인
- 관리자 수동 노쇼 처리 시 `+3점` 반영 확인
- 정상 이용 10회 연속 시 1점 차감 검증
- 90일 경과 시 점수 초기화 검증
- 관리자 상태 변경 후 `maintenance`/`disabled` 자원은 신규 예약 불가인지 검증
- 시점 이동 성공/차단 이벤트가 화면과 감사 로그에 모두 남는지 검증
- 두 인스턴스가 동시에 예약 생성 시 파일 잠금으로 데이터 손상 없이 한쪽만 성공하는지 검증

## Assumptions
- 과제 성격상 비밀번호는 사용자가 선택한 대로 `평문 저장`으로 간다. 보안 고도화는 이번 범위에서 제외한다.
- 관리자만 체크인, 대여 시작, 퇴실 승인, 반납 승인, 노쇼 판정, 파손/오염 판정을 입력한다.
- 사용자 퇴실/반납은 종료 경계 시점에 신청하고, 관리자가 같은 시점에 승인하는 흐름을 기본으로 한다.
- 시스템은 자동 노쇼 판정 대신 `시점 이동 차단 + 관리자 수동 노쇼 처리`를 채택한다.
- `3점 이상 6점 미만` 제재 구간의 “최대 예약 가능 횟수 1회 제한”은 회의실/장비를 합친 전체 활성 예약 기준으로 해석한다.
- 문서는 현재 코드와 사용자 흐름을 기준으로 작성하며, 정책 변경이 생기면 문서와 테스트를 함께 갱신한다.
