## 1. 현재 시스템의 사용자 흐름

## 1.1 전체 진입 흐름

```text
프로그램 시작
-> 데이터 파일 준비
-> 서비스 객체 생성
-> 게스트 메뉴 진입
-> 로그인 성공
-> 역할 분기
   -> 일반 사용자 메뉴
   -> 관리자 메뉴
-> 로그아웃
-> 게스트 메뉴 복귀
-> 종료 선택 시 프로그램 종료
```

상세 단계:

1. 프로그램 시작 시 `ensure_data_dir()`로 데이터 파일이 준비된다.
2. 서비스 객체를 생성한다.
   - 인증
   - 회의실 예약
   - 장비 예약
   - 패널티
   - 정책 자동 처리
3. 게스트 메뉴로 진입한다.
4. 게스트 메뉴에서 로그인 성공 시 역할에 따라 분기한다.
   - `user`면 사용자 메뉴
   - `admin`이면 관리자 메뉴
5. 각 역할 메뉴에서 로그아웃하면 다시 게스트 메뉴로 돌아간다.
6. 게스트 메뉴에서 종료를 선택하면 프로그램이 끝난다.

## 1.2 게스트 흐름

### 게스트 메인 플로우

```text
게스트 메뉴
-> 1. 로그인
-> 2. 회원가입
-> 0. 종료
```

게스트는 세 가지 행동만 가능하다.

- 로그인
- 회원가입
- 종료

### 게스트 공통 진입 플로우

1. 게스트 메뉴 진입 직전에 정책 점검 루틴이 한 번 실행된다.
2. 정책 점검에 실패하면 오류를 보여주고 프로그램 흐름이 종료될 수 있다.
3. 정책 점검을 통과하면 게스트 메뉴를 출력한다.

### 로그인 서브플로우

```text
게스트 메뉴
-> 로그인 선택
-> 사용자명 입력
-> 비밀번호 입력
-> 인증
-> 로그인 직후 정책 점검
-> 최신 사용자 정보 재조회
-> 패널티 경고 출력 가능
-> 역할별 메뉴 이동
```

상세 단계:

1. `사용자명` 입력
2. `비밀번호` 입력
3. 인증 서비스로 로그인 시도
4. 로그인 성공 직후 정책 점검 루틴 재실행
5. 최신 사용자 정보 재조회
6. 경고 메시지가 있으면 출력
7. 역할별 메뉴로 이동

### 회원가입 서브플로우

```text
게스트 메뉴
-> 회원가입 선택
-> 사용자명 입력 및 검증
-> 비밀번호 입력 및 검증
-> 비밀번호 확인
-> 계정 생성
-> 성공 메시지
-> 게스트 메뉴 복귀
```

상세 단계:

1. 사용자명 입력
2. 사용자명 문법 검증
3. 비밀번호 입력
4. 비밀번호 문법 검증
5. 비밀번호 확인 입력
6. 두 비밀번호 일치 확인
7. 계정 생성
8. 성공 메시지 출력 후 게스트 메뉴로 복귀

### 종료 서브플로우

```text
게스트 메뉴
-> 종료 선택
-> 종료 확인
-> 프로그램 종료 또는 게스트 메뉴 유지
```

상세 단계:

1. 종료 선택
2. `confirm()`으로 종료 여부 재확인
3. 확인 시 프로그램 종료
4. 취소 시 게스트 메뉴 유지

## 1.3 일반 사용자 흐름

### 사용자 메인 플로우

```text
사용자 메뉴 진입
-> 정책 점검
-> 최신 사용자 정보 재조회
-> 현재 패널티 상태 표시
-> 사용자 메뉴 출력
-> 기능 선택
-> 작업 수행
-> 사용자 메뉴로 복귀
-> 로그아웃 선택 시 게스트 메뉴 복귀
```

### 사용자 메뉴 구조

```text
[회의실]
1. 회의실 목록 조회
2. 회의실 예약하기
3. 내 회의실 예약 조회
4. 회의실 예약 변경
5. 회의실 예약 취소
6. 회의실 퇴실 신청

[장비]
7. 장비 목록 조회
8. 장비 예약하기
9. 내 장비 예약 조회
10. 장비 예약 변경
11. 장비 예약 취소
12. 장비 반납 신청

[내 정보]
13. 내 상태 조회

0. 로그아웃
```

사용자 메뉴는 회의실, 장비, 내 정보의 세 묶음으로 나뉜다.

### 사용자 공통 진입 서브플로우

1. 사용자 메뉴 루프 시작
2. 정책 점검 실행
3. 최신 사용자 정보 재조회
4. 패널티 상태 조회
5. 금지 또는 제한 상태 메시지 출력 가능
6. 메뉴 표시
7. 기능 수행 후 다시 메뉴 루프로 복귀

```text
사용자 메뉴
-> 회의실 목록 조회

-> 이름/수용인원/위치/상태 표 출력

```

- 회의실 목록 조회
- 회의실 예약하기
- 내 회의실 예약 조회
- 회의실 예약 변경
- 회의실 예약 취소
- 회의실 퇴실 신청

#### 회의실 예약 생성 서브플로우

```text
사용자 메뉴
-> 회의실 예약하기
-> 정책 점검
-> 사용자 상태 재조회
-> 예약 가능 여부 확인
-> 이용 인원 입력
-> 시작일/종료일 입력
-> 예약 가능 회의실 목록 조회
-> 회의실 선택
-> 예약 생성
-> 결과 출력
-> 사용자 메뉴 복귀
```

1. 정책 점검 실행
2. 사용자 최신 상태 재조회
3. 예약 가능 여부 확인
   - 금지 상태면 차단
   - 제한 상태면 전체 활성 예약 수 확인
4. 일 단위 예약 안내 문구 출력
5. 이용 인원 입력
6. 시작일, 종료일 입력
7. 시스템이 매일 `09:00 ~ 18:00` 고정 시간으로 기간을 계산
8. 인원 수용 가능 + 해당 기간 충돌 없는 회의실 목록 조회
9. 회의실 선택
10. 예약 생성

```text
사용자 메뉴
```

```text
사용자 메뉴

-> 예약 변경
```

1. 정책 점검 실행
2. 사용자 최신 상태 재조회
3. 본인의 `reserved` 상태 예약만 목록에 노출
4. 예약 선택
5. 새 시작일, 종료일 입력
6. 같은 회의실에서 충돌 검사 후 변경

#### 회의실 예약 취소 서브플로우

```text
사용자 메뉴
-> 회의실 예약 취소
-> 정책 점검
-> 사용자 상태 재조회
-> 취소 가능한 reserved 예약 목록 조회

-> 직전 취소 패널티 여부 출력
-> 사용자 메뉴 복귀
```

1. 정책 점검 실행
2. 사용자 최신 상태 재조회
3. 본인의 `reserved` 상태 예약만 목록에 노출
4. 예약 선택
5. 취소 확인
6. 취소 처리
7. 시작 1시간 이내였다면 직전 취소 패널티 부여

#### 회의실 퇴실 신청 서브플로우

```text
사용자 메뉴
-> 회의실 퇴실 신청
-> 사용자 상태 재조회
-> checked_in 예약 목록 조회
-> 예약 선택
-> 퇴실 신청
-> checkout_requested 전이
-> 사용자 메뉴 복귀
```

1. 사용자는 `checked_in` 상태 예약에 대해 퇴실 신청만 할 수 있다.
2. 퇴실 신청 시 상태가 `checkout_requested`로 바뀐다.
3. 관리자가 승인하면 `completed`가 된다.
4. 패널티 계산은 관리자 승인 시각이 아니라 사용자의 퇴실 신청 시각을 기준으로 계산된다.

## 2. 현재 문법 규칙

이 절은 입력 문자열 자체의 형식 규칙을 다룬다.
즉, 형식이 맞는지 여부만 본다.

## 2.1 공통 입력 정규화

- 대부분의 입력은 `input(...).strip()` 이후 검사한다.
- 앞뒤 공백은 제거된다.
- 빈 문자열 처리 방식은 입력 종류마다 다르다.

## 2.2 최상위 메뉴 입력

게스트, 사용자, 관리자 메인 메뉴는 공통 validator를 쓰지 않고 문자열 비교로 분기한다.

허용 형식:

- 게스트 메뉴: `"1"`, `"2"`, `"0"`
- 사용자 메뉴: `"1"`부터 `"13"`, `"0"`
- 관리자 메뉴: `"1"`부터 `"17"`, `"0"`

특징:

- `"01"`은 `"1"`과 다르므로 잘못된 선택으로 처리된다.
- 빈 입력도 잘못된 선택으로 처리된다.
- 숫자가 아닌 입력도 잘못된 선택으로 처리된다.
- 이 세 메뉴는 `int` 변환이 아니라 문자열 비교를 사용한다.

## 2.3 목록 선택 입력 `select_from_list()`

형식:

- 숫자 입력만 허용
- `1`부터 `N`까지가 실제 항목
- `allow_cancel=True`일 때 `0`은 취소

특징:

- 내부적으로 `int(choice)`를 사용한다.
- `"01"`은 `1`로 해석되어 허용된다.
- 빈 입력은 재입력 루프로 돌아간다.
- 숫자가 아니면 `"숫자를 입력해주세요."`

## 2.4 확인 입력 `confirm()`

허용 토큰:

- Yes 계열: `y`, `yes`, `예`, `ㅇ`
- No 계열: `n`, `no`, `아니오`, `ㄴ`

처리 방식:

- 소문자화 후 비교한다.
- 그 외 값은 재입력

## 2.5 취소 키워드

일부 validator 입력에서는 다음 문자열이 취소로 해석된다.

- `q`
- `quit`
- `취소`

적용 위치:

- 날짜 입력
- 시간 입력
- 날짜+시간 입력
- 일 단위 시작일/종료일 입력
- 양의 정수 입력

주의:

- 최상위 메뉴, 로그인, 회원가입, 일반 문자열 입력에는 이 취소 문법이 공통 적용되지 않는다.

## 2.6 사용자명 문법

정규화:

- 앞뒤 공백 제거

형식 규칙:

- 길이: `3~20`
- 허용 문자: 영문 대소문자, 숫자, `_`

정규표현식:

```regex
^[a-zA-Z0-9_]+$
```

## 2.7 비밀번호 문법

정규화:

- 앞뒤 공백 제거

형식 규칙:

- 길이: `4~50`
- 문자 종류 제한 없음

주의:

- 내부 공백 금지 규칙은 없다.
- 앞뒤 공백은 제거되어 저장된다.

## 2.8 날짜 문법

사용자 CLI에서 실제 예약 입력은 시간 없는 날짜 범위만 받는다.

형식:

```text
YYYY-MM-DD
```

정규표현식:

```regex
^\d{4}-\d{2}-\d{2}$
```

추가 특징:

- 시작일과 종료일 두 값을 한 세트로 입력한다.
- 실제 존재하는 날짜여야 한다.

## 2.9 시간 문법

현재 사용자 흐름에서는 직접 노출되지 않지만 validator와 도메인 서비스에 존재한다.

형식:

```text
HH:MM
```

정규표현식:

```regex
^\d{2}:\d{2}$
```

추가 규칙:

- 30분 단위여야 한다.

## 2.10 정수 입력 문법

`validate_positive_int()` 기준:

- 숫자 문자열만 허용
- 부호(`+`, `-`) 불허
- 소수점 불허

예시:

- 허용: `1`, `5`, `100`
- 비허용: `0`(범위에 따라), `-1`, `+1`, `1.5`, `abc`

사용 위치:

- 회의실 이용 인원
- 관리자 파손/오염 패널티 점수

## 2.11 관리자 자유 텍스트 입력

자유 텍스트는 다음처럼 사용된다.

- 관리자 예약 취소 사유
- 파손/오염 패널티 사유

형식 규칙:

- `strip()`만 적용
- 길이 제한 없음
- 허용 문자 제한 없음
- 빈 문자열도 가능

## 3. 현재 의미 규칙

이 절은 입력 형식이 맞다고 가정한 뒤 도메인 차원에서 적용되는 규칙이다.

## 3.1 정책 점검 실행 시점

정책 점검 루틴은 현재 다음 시점에 실행된다.

- 게스트 메뉴 진입 전
- 로그인 직후
- 사용자 메뉴 루프 진입 시 매 반복
- 관리자 메뉴 루프 진입 시 매 반복
- 예약 생성/변경/취소 관련 도메인 메서드 내부

자동 처리 내용:

- 회의실 노쇼 판정
- 장비 노쇼 판정
- 90일 경과 패널티 초기화
- 제한 기간 만료 처리
- 6점 이상 사용자의 미래 예약 자동 취소

## 3.2 사용자 및 권한 규칙

- 로그인은 존재하는 사용자만 가능
- 비밀번호는 평문 비교
- 관리자 전용 기능은 현재 저장소의 최신 사용자 역할로 다시 확인한다
- 삭제되었거나 권한이 바뀐 사용자는 메뉴에서 바로 차단될 수 있다

## 3.3 회의실 예약 의미 규칙

현재 CLI 기준 회의실 예약은 시간 슬롯 예약이 아니라 일 단위 예약이다.

핵심 규칙:

- 시작일은 내일부터 가능
- 시작일은 현재 시점 기준 최대 6개월 이내
- 종료일은 시작일보다 빠를 수 없음
- 예약 기간은 최대 14일
- 실제 예약 시간은 자동으로
  - 시작일 `09:00`
  - 종료일 `18:00`
- 회의실 상태가 `available`이어야 함
- 이용 인원은 1명 이상
- 회의실 수용 인원이 이용 인원 이상이어야 함
- 같은 회의실의 활성 예약과 기간 충돌이 없어야 함
- 기본 회의실 활성 예약 한도는 사용자당 3건

회의실 활성 예약으로 간주되는 상태:

- `reserved`
- `checked_in`
- `checkout_requested`

회의실 사용자 변경/취소 가능 상태:

- `reserved`만 가능

회의실 관리자 시간 변경 가능 상태:

- `reserved`만 가능
- 일반 시간 변경 API는 이미 시작된 예약을 막지만
- 현재 CLI가 사용하는 일 단위 관리자 변경 API는 시작 여부를 별도로 막지 않는다
- 다만 일 단위 예약 규칙상 새 시작일은 항상 내일부터라서 실사용 상 대부분 미래 예약으로만 흘러간다

회의실 상태 전이:

| 행동 | 전이 |
| --- | --- |
| 예약 생성 | `reserved` |
| 관리자 체크인 | `reserved -> checked_in` |
| 사용자 퇴실 신청 | `checked_in -> checkout_requested` |
| 관리자 퇴실 승인 | `checkout_requested -> completed` |
| 사용자 취소 | `reserved -> cancelled` |
| 관리자 취소 | `reserved -> admin_cancelled` |
| 자동 노쇼 | `reserved -> no_show` |

## 3.4 장비 예약 의미 규칙

장비 예약도 현재 CLI 기준 일 단위 예약이다.

핵심 규칙:

- 시작일은 내일부터 가능
- 시작일은 현재 시점 기준 최대 6개월 이내
- 종료일은 시작일보다 빠를 수 없음
- 예약 기간은 최대 14일
- 실제 예약 시간은 자동으로
  - 시작일 `09:00`
  - 종료일 `18:00`
- 장비 상태가 `available`이어야 함
- 장비 종류를 먼저 선택한 뒤 개별 자산을 선택
- 같은 자산의 활성 예약과 기간 충돌이 없어야 함
- 기본 장비 활성 예약 한도는 사용자당 3건

장비 활성 예약으로 간주되는 상태:

- `reserved`
- `checked_out`
- `return_requested`

장비 사용자 변경/취소 가능 상태:

- `reserved`만 가능

장비 상태 전이:

| 행동 | 전이 |
| --- | --- |
| 예약 생성 | `reserved` |
| 관리자 대여 시작 | `reserved -> checked_out` |
| 사용자 반납 신청 | `checked_out -> return_requested` |
| 관리자 반납 승인 | `return_requested -> returned` |
| 사용자 취소 | `reserved -> cancelled` |
| 관리자 취소 | `reserved -> admin_cancelled` |
| 자동 노쇼 | `reserved -> no_show` |

## 3.5 패널티 및 제한 규칙

패널티 규칙:

- 노쇼: `+3점`
- 직전 취소: `+2점`
- 지연 퇴실/반납: `ceil(지연 분 / 10)`점
- 파손/오염: 관리자 수동 `1~5점`

제한 규칙:

- `0~2점`: 정상
- `3~5점`: 제한 상태
  - 제한 시작 시점부터 7일간
  - 회의실/장비를 합친 전체 활성 예약 1건만 허용
- `6점 이상`: 금지 상태
  - 30일간 예약 불가
  - 미래의 `reserved` 예약 자동 취소

추가 규칙:

- 패널티 발생 시 `normal_use_streak`는 0으로 리셋
- 정상 이용 10회 연속이면 1점 차감
- 이때 점수가 이미 0이면 감점 없이 streak만 0으로 리셋
- 마지막 패널티 발생일로부터 90일이 지나면 누적 점수 초기화
- 제한 해제 시 `restriction_until`은 자동으로 제거될 수 있다

## 3.6 노쇼 규칙

- 회의실:
  - 예약 시작 후 15분이 지나도 `checked_in`이 아니면 `no_show`
- 장비:
  - 예약 시작 후 15분이 지나도 `checked_out`이 아니면 `no_show`

노쇼는 자동 정책 점검 시 처리된다.

## 3.7 직전 취소 규칙

- 사용자가 `reserved` 예약을 취소할 때만 적용
- 예약 시작까지 남은 시간이 60분 미만이면 직전 취소
- 관리자 취소와 시스템 취소에는 적용되지 않음

## 3.8 지연 퇴실/반납 규칙

현재 구현의 중요한 특징:

- 관리자 승인 시각이 아니라 사용자의 신청 시각을 기준으로 지연 여부를 계산한다.
- 즉, 사용자가 제때 신청했고 관리자가 늦게 승인해도 사용자 패널티는 늘지 않는다.

계산식:

```text
delay_minutes = max(0, request_time - scheduled_end_time)
penalty_points = ceil(delay_minutes / 10)
```

## 3.9 자원 상태 변경 규칙

회의실/장비 상태:

- `available`
- `maintenance`
- `disabled`

의미:

- `available`만 신규 예약 가능
- `maintenance`, `disabled`는 신규 예약 불가
- `maintenance`, `disabled`로 변경 시
  - 미래의 `reserved` 예약만 자동 취소
  - 패널티는 부여하지 않음

## 3.10 동시성 규칙

- 모든 쓰기 작업은 전역 락 아래 수행된다
- 다중 파일 변경은 `UnitOfWork`로 원자적으로 저장된다
- 동시에 같은 자원을 예약하면 먼저 락을 잡은 쪽만 성공한다

## 4. 현재 구현상 눈에 띄는 특징과 불일치

## 4.1 CLI는 시간 입력형이 아니라 날짜 범위형이다

도메인에는 `30분 단위 시간 예약 API`가 있지만 현재 CLI는 사용하지 않는다.
사용자와 관리자가 실제로 입력하는 것은 `시작일/종료일`뿐이다.

즉, 현재 사용자 문법은 다음에 더 가깝다.

```text
회의실 예약 = 이용인원 + 시작일 + 종료일 + 회의실선택
장비 예약 = 장비종류 + 시작일 + 종료일 + 장비선택
```

## 4.2 실제 상태 모델에는 요청 상태가 추가되어 있다

`docs/PLAN.md`, `docs/PLAN2.md`의 초기 상태 설명보다 현재 구현은 다음 중간 상태를 추가로 사용한다.

- 회의실: `checkout_requested`
- 장비: `return_requested`

즉, 현재 흐름은 즉시 퇴실/반납 완료가 아니라 사용자 신청 후 관리자 승인형이다.

## 4.3 상단 메뉴와 목록 선택의 숫자 문법이 다르다

- 메인 메뉴는 문자열 비교라 `"01"`이 실패
- 목록 선택은 정수 변환이라 `"01"`이 성공

같은 숫자 입력인데 화면마다 문법이 다르다.

## 4.4 취소 문법이 전역 공통이 아니다

`q`, `quit`, `취소`는 validator 기반 입력에만 적용된다.
로그인, 회원가입, 메인 메뉴, 자유 텍스트에는 공통 취소 문법이 없다.

## 4.5 관리자 취소 사유는 사실상 선택 입력이 아니다

취소 사유는 입력받지만:

- 빈 문자열 허용
- 길이 제한 없음
- 포맷 제한 없음

즉, 현재는 감사 로그용 선택적 메모에 가깝다.

## 4.6 화면 단위 공통 validator가 완전히 통일되어 있지는 않다

현재 저장소는 문법 규칙을 일부 validator로 분리했지만 실제 메인 흐름 입력은 여전히 화면별 직접 분기로 처리하는 부분이 많다.

## 5. 현재 구현을 한 문장으로 요약

현재 시스템은 `게스트 -> 역할별 CLI 메뉴` 구조를 가지며, 사용자에게는 `일 단위(09:00~18:00 고정) 예약 흐름`을 제공하고, 관리자 승인 기반의 체크인/퇴실 및 대여/반납 흐름, 그리고 `패널티 기반 이용 제한 정책`을 결합한 파일 기반 예약 관리 프로그램으로 동작한다.

## 6. 추가로 의논해야 할 부분

아래 항목들은 현재 코드만으로는 정책 의도가 완전히 확정됐다고 보기 어려운 부분들이다.
즉, 구현은 존재하지만 팀 차원의 명세 결정이 더 필요하다.

## 6.1 이 시스템은 정말 `일 단위 예약`이 맞는가

현재 CLI는 날짜만 입력받고 시간을 고정한다.
반면 도메인 계층에는 30분 단위 시간 예약 API가 남아 있다.

의논할 질문:

- 제품을 `일 단위 예약 시스템`으로 확정할 것인가
- 아니면 이후 CLI를 시간 입력형으로 바꿀 것인가

## 6.2 활성 예약 한도의 공식 해석은 무엇인가

현재 구현은 기본적으로:

- 회의실 최대 3건
- 장비 최대 3건

으로 보이지만, 제한 상태에서는 회의실과 장비를 합쳐 전체 1건만 허용한다.
또한 현재 사용자 CLI는 사실상 한 번에 회의실 1건, 장비 1건을 중심으로 안내한다.

의논할 질문:

- 정상 상태에서 공식 한도는 `회의실 3 + 장비 3`인가
- 아니면 `전체 6건`으로 설명할 것인가
- 사용자에게 한도를 어떤 문장으로 보여줄 것인가

## 6.3 같은 사용자의 동시 사용 중복을 허용할 것인가

현재 충돌 검사는 `같은 자원` 기준이다.
즉, 사용자는 정상 상태라면 같은 날짜/시간대에 서로 다른 회의실 여러 개를 잡을 수 있고, 장비도 여러 건 잡을 수 있다.

의논할 질문:

- 한 사용자가 같은 시간대에 여러 회의실을 동시에 예약해도 되는가
- 한 사용자가 같은 기간에 동일 종류 장비를 여러 개 보유해도 되는가
- `대표 예약 1건 + 보조 장비 n건` 같은 정책이 필요한가

## 6.4 체크인/대여 시작 가능 시점을 제한할 것인가

현재 관리자는 `reserved` 상태면 예약 시작 전이라도 체크인/대여 시작 처리를 할 수 있다.

의논할 질문:

- 시작 시각 이전 체크인/대여를 허용할 것인가
- 허용한다면 몇 분 전부터 가능한가
- 시작 시각 이후 너무 늦은 체크인은 언제까지 허용할 것인가

## 6.5 조기 퇴실/조기 반납을 어떻게 해석할 것인가

현재는 사용자가 종료 시각보다 빨리 퇴실/반납 신청해도 그대로 완료 가능하다.

의논할 질문:

- 조기 종료를 허용할 것인가
- 허용 시 실제 종료 시각을 기록만 하고 예약 종료 시각은 유지할 것인가
- 조기 종료 시 다음 사용자가 앞당겨 이용할 수 있게 할 것인가

## 6.6 노쇼와 직전 취소의 경계값을 어떻게 정의할 것인가

현재 구현은 다음처럼 해석된다.

- 노쇼: 시작 후 정확히 15분이 지난 순간이 아니라 `15분 초과` 시점부터 판정
- 직전 취소: 시작 `60분 미만`일 때만 패널티

즉:

- 정확히 15분 시점은 아직 노쇼가 아님
- 정확히 60분 전 취소는 직전 취소가 아님

의논할 질문:

- 경계값을 `<`로 둘 것인가 `<=`로 둘 것인가
- 사용자 안내 문구를 어떤 표현으로 고정할 것인가

## 6.7 자원 상태 변경 시 진행 중 예약을 어떻게 다룰 것인가

현재 `maintenance` 또는 `disabled`로 바뀌면 미래의 `reserved` 예약만 자동 취소된다.
이미 `checked_in`, `checked_out`, `checkout_requested`, `return_requested`인 건은 그대로 남는다.

의논할 질문:

- 점검 전환 시 진행 중 이용도 강제 종료해야 하는가
- 아니면 현재처럼 미래 예약만 정리하면 되는가
- 긴급 점검 시 별도 강제 종료 흐름이 필요한가

## 6.8 관리자 취소 사유와 패널티 사유를 필수로 할 것인가

현재는 관리자 취소 사유가 빈 문자열이어도 된다.
패널티 사유도 비면 기본 문자열로 대체된다.

의논할 질문:

- 관리자 취소 사유를 필수로 할 것인가
- 파손/오염 패널티 사유를 템플릿화할 것인가
- 이 사유를 사용자에게도 보여줄 것인가

## 6.9 관리자 계정 생성 정책을 어떻게 운영할 것인가

일반 회원가입 화면으로는 관리자 계정을 만들 수 없다.
현재는 시드 스크립트가 `admin / admin123` 계정을 만드는 방식이다.

의논할 질문:

- 운영 환경에서도 시드 계정을 유지할 것인가
- 초기 비밀번호 변경을 강제할 것인가
- 관리자 계정 추가/삭제 흐름이 필요한가

## 6.10 계정/인증 보안을 어디까지 요구할 것인가

현재 비밀번호는 평문 저장이고 변경/재설정 기능이 없다.

의논할 질문:

- 과제 범위상 평문 저장을 유지할 것인가
- 비밀번호 변경 기능이 필요한가
- 로그인 실패 횟수 제한이 필요한가

## 6.11 사용자명 대소문자와 예약어 정책을 정할 것인가

현재 `User`와 `user`는 서로 다른 계정으로 취급된다.
또한 `admin`, `system` 같은 이름 금지 정책이 없다.

의논할 질문:

- 사용자명을 대소문자 구분할 것인가
- 예약어 금지 목록이 필요한가
- 표시용 이름과 로그인용 ID를 분리할 것인가

## 6.12 정책 점검 루틴이 조회 시점에도 상태를 바꿔도 되는가

현재 정책 점검은 로그인 직후, 메뉴 진입 시에도 실행되고 실제 데이터 상태를 바꾼다.
즉, 사용자가 단순 조회만 하려 해도 노쇼 판정이나 자동 취소가 먼저 일어날 수 있다.

의논할 질문:

- 이 부수효과를 유지할 것인가
- 아니면 명시적 배치/작업 진입 시점으로 제한할 것인가
- 사용자에게 자동 처리 발생 사실을 더 적극적으로 보여줄 것인가

## 6.13 자유 텍스트 필드의 길이와 허용 문자를 정할 것인가

현재 취소 사유와 메모는 사실상 무제한 자유 입력이다.

의논할 질문:

- 길이 제한이 필요한가
- 줄바꿈이나 특수문자를 허용할 것인가
- 감사 로그에 남길 텍스트와 사용자 노출 텍스트를 분리할 것인가

## 6.14 패널티 사유 enum을 실제 운영 정책과 맞출 것인가

모델에는 `damage`, `contamination`, `other`가 있지만 현재 관리자 UI는 사실상 `파손/오염`을 하나의 흐름으로 다룬다.

의논할 질문:

- 파손과 오염을 별도 사유로 구분할 것인가
- `other`를 관리자 UI에 노출할 것인가
- 보고서/통계 기준 사유 체계를 어떻게 가져갈 것인가
