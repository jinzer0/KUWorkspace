# 입력 규칙 및 미확정 정책 회의 문서

## 문서 목적

이 문서는 현재 구현과 기획서(`PLAN.md`, `PLAN2.md`)를 기준으로,
이미 코드에 반영된 입력 규칙과 아직 요구자/팀이 판단해야 하는 항목을 분리해 정리한 회의용 문서다.

핵심 목적은 다음 두 가지다.

1. 현재 프로그램이 어떤 입력 규칙을 사실상 채택하고 있는지 공유한다.
2. 아직 정책적으로 확정되지 않은 입력/문법/의미 규칙을 팀 회의 안건으로 정리한다.

---

## 요구자 관점에서 먼저 확정해야 할 요구사항

이 문서는 단순히 "무엇이 미정인지"를 나열하는 데서 끝나면 안 된다.
요구자 입장에서는 아래 항목들을 먼저 확정해서 개발자에게 명시적으로 요구해야 한다.

### 1. 숫자 입력에 대해 요구할 것

- 요구자가 먼저 정해야 할 핵심:
  - 시스템 전체에서 숫자 입력을 기본적으로 정수만 허용할지
  - 예외적으로 `float`를 허용할 필드가 있는지
  - 각 숫자 필드의 최소/최대 범위가 무엇인지
- 개발팀에 이렇게 요구하면 된다:
  - "숫자 입력은 기본적으로 정수만 허용하고, 소수 입력이 필요한 필드는 별도로 지정한다. 각 입력 필드는 최소값/최대값을 명세서에 표로 정리한다."
- 어떻게 결정할지:
  - 실제 입력 필드를 전부 나열한다.
  - 각 필드가 개수/점수/횟수처럼 정수 성격인지, 금액/비율처럼 소수 성격인지 분류한다.
  - 분류가 끝나면 필드별 허용 범위를 확정한다.

### 2. 아이디(사용자명)에 대해 요구할 것

- 요구자가 먼저 정해야 할 핵심:
  - 길이 제한
  - 허용 문자 집합
  - 한글/영문/숫자/특수문자 허용 여부
  - 대소문자 구분 여부
  - 예약어 금지 여부
- 개발팀에 이렇게 요구하면 된다:
  - "사용자명은 문법 규칙과 의미 규칙을 분리해 정의하고, 허용 문자·길이·중복 여부·예약어 정책을 모두 명시한다."
- 어떻게 결정할지:
  - 운영자가 직접 입력/관리하기 쉬운 형식인지 먼저 본다.
  - 로그인 혼동을 줄이기 위해 대소문자/공백/유사 문자 정책을 정한다.
  - 이후 정규표현식과 의미 규칙 문장으로 각각 고정한다.

### 3. 비밀번호에 대해 요구할 것

- 요구자가 먼저 정해야 할 핵심:
  - 최소/최대 길이
  - 조합 규칙 필요 여부
  - 공백 허용 여부
  - 너무 쉬운 비밀번호 차단 여부
- 개발팀에 이렇게 요구하면 된다:
  - "비밀번호는 길이만 검사할지, 문자 조합까지 검사할지 확정하고, 공백 처리 방식까지 포함해 명세서에 고정한다."
- 어떻게 결정할지:
  - 과제 범위와 사용성을 우선할지, 보안 강도를 우선할지 선택한다.
  - 선택이 끝나면 회원가입/로그인/변경 화면 모두 동일 규칙을 쓰도록 요구한다.

### 4. 날짜/시간 입력에 대해 요구할 것

- 요구자가 먼저 정해야 할 핵심:
  - 입력 포맷을 하나로 강제할지
  - 30분 단위를 유지할지
  - 과거 시간/당일/자정 경계 처리 방식
  - 예약 가능 기간 기준
- 개발팀에 이렇게 요구하면 된다:
  - "날짜/시간은 입력 포맷, 단위, 예약 가능 범위, 예외 상황 처리까지 포함해 하나의 공통 정책으로 구현한다."
- 어떻게 결정할지:
  - 사용 시나리오를 기준으로 예시를 만든다.
  - 예: 오늘 예약, 14일 초과 예약, 자정 직전 예약, 잘못된 형식 입력.
  - 각 예시가 허용인지 금지인지 먼저 표로 정한 뒤 구현하게 한다.

### 5. 공통 CLI 입력 방식에 대해 요구할 것

- 요구자가 먼저 정해야 할 핵심:
  - 메뉴 선택 입력 정책
  - 목록 선택 입력 정책
  - `confirm()` 입력 정책
  - `pause()` 입력 정책
  - 취소 키워드 공통화 여부
- 개발팀에 이렇게 요구하면 된다:
  - "메뉴 선택, 목록 선택, 확인 입력, 취소 입력, 계속 진행 입력은 화면마다 다르게 만들지 말고 공통 규칙으로 통일한다."
- 어떻게 결정할지:
  - 사용자가 자주 실수할 입력을 먼저 고른다.
  - 각 입력마다 허용 토큰, 기본값, 재입력 방식, 취소 가능 여부를 정한다.
  - 정한 내용을 모든 메뉴에 공통 적용하도록 요구한다.

### 5-1. 입력 정규화(normalization) 방식에 대해 요구할 것

- 요구자가 먼저 정해야 할 핵심:
  - 입력 전 `trim()`을 항상 적용할지
  - 대소문자 무시 입력을 어디까지 허용할지
  - `01`, `+1` 같은 입력을 정상 숫자로 볼지
  - 빈 Enter를 재입력/기본값/취소 중 무엇으로 볼지
- 개발팀에 이렇게 요구하면 된다:
  - "모든 CLI 입력은 공백 제거, 대소문자 처리, 선행 0, 부호(`+`, `-`) 해석 정책을 공통 규칙으로 문서화하고, 화면마다 다르게 동작하지 않게 구현한다."
- 어떻게 결정할지:
  - 메뉴 선택, 확인 입력, 자유 텍스트 입력을 각각 예시로 둔다.
  - `1`, `01`, `+1`, ` yes `, 빈 Enter 같은 샘플이 각각 허용인지 금지인지 먼저 정한다.
  - 그 결과를 validator와 안내 문구에 동일하게 반영하도록 요구한다.

### 6. 자유 텍스트 입력에 대해 요구할 것

- 요구자가 먼저 정해야 할 핵심:
  - 최대 길이
  - 줄바꿈 허용 여부
  - 공백만 입력했을 때 처리 방식
  - 기본값 대체 여부
- 개발팀에 이렇게 요구하면 된다:
  - "사유/메모/설명 입력은 빈값 처리, 길이 제한, 특수문자 허용 범위를 정하고, 저장 전 정규화 여부까지 명시한다."
- 어떻게 결정할지:
  - 실제로 로그에 남는지, 관리자 화면에 표시되는지, 감사용 기록인지 먼저 구분한다.
  - 그 다음 각 용도별로 길이와 허용 문자를 정한다.

### 7. 검증 순서에 대해 요구할 것

- 요구자가 먼저 정해야 할 핵심:
  - 문법 오류와 의미 오류를 분리할지
  - 검증 순서를 고정할지
- 개발팀에 이렇게 요구하면 된다:
  - "모든 입력은 문법 규칙을 먼저 검사하고, 문법을 통과한 경우에만 의미 규칙을 검사한다. 오류 메시지도 두 종류를 구분한다."
- 어떻게 결정할지:
  - 사용자에게 보여줄 오류 메시지 예시를 먼저 만든다.
  - 형식 오류와 정책 위반 오류가 실제로 다르게 이해되는지 확인한 뒤 확정한다.

## 요구자가 개발자에게 전달할 때의 작성 방식

요구자는 "아이디 제한 좀 걸어주세요"처럼 추상적으로 요청하면 안 되고,
아래 형식으로 항목별 요구사항을 적어주는 것이 가장 안전하다.

### 권장 요구사항 템플릿

1. 입력 항목 이름: 예) 사용자명, 비밀번호, 메뉴 선택, 취소 사유
2. 문법 규칙: 길이, 허용 문자, 포맷, 정규표현식/BNF
3. 의미 규칙: 중복 금지, 범위 제한, 과거 시간 금지 같은 도메인 정책
4. 예외 처리: 빈 입력, 공백 입력, 잘못된 입력, 취소 입력을 어떻게 처리할지
5. 오류 메시지 원칙: 형식 오류와 정책 오류를 분리할지
6. 예시 입력: 허용 예시 / 금지 예시를 최소 2개 이상

### 요구자가 회의에서 바로 던질 질문

- "이 입력은 정수만 받나요, 아니면 소수도 받나요?"
- "최소값/최대값은 무엇인가요?"
- "공백을 제거한 뒤 검사하나요, 원문 그대로 검사하나요?"
- "한글/영문/숫자/특수문자 중 어디까지 허용하나요?"
- "빈 입력은 재입력인가요, 기본값인가요, 취소인가요?"
- "문법 오류와 정책 위반 오류를 사용자에게 다르게 보여주나요?"
- "이 정책은 모든 화면에서 공통인가요, 특정 화면만 예외인가요?"

## 현재 코드 기준으로 이미 사실상 확정된 규칙

### 먼저 봐야 할 주의점

- 현재 구현은 모든 입력을 하나의 공통 validator로 처리하지 않는다.
- 즉, 같은 "선택" 입력처럼 보이더라도 실제 허용 규칙이 화면/함수마다 조금씩 다를 수 있다.
- 따라서 요구자/팀은 "현재 구현을 그대로 확정할지"보다, "어떤 입력 규칙을 전역 공통 정책으로 통일할지"를 먼저 결정해야 한다.

### 1. 사용자명

- 현재 구현 위치: `src/domain/auth_rules.py`
- 현재 문법 규칙:
  - 앞뒤 공백 제거 후 검사한다.
  - 길이: 3자 이상, 20자 이하
  - 허용 문자: 영문 대소문자, 숫자, 밑줄(`_`)
  - 정규표현식:

```regex
^[a-zA-Z0-9_]+$
```

- 현재 의미 규칙:
  - 빈 문자열 불가
  - 기존 사용자명과 중복 불가
  - 입력 전후 공백은 제거한 뒤 저장/검사한다.

### 2. 비밀번호

- 현재 구현 위치: `src/domain/auth_rules.py`
- 현재 문법 규칙:
  - 앞뒤 공백 제거 후 검사한다.
  - 길이: 4자 이상, 50자 이하
  - 문자 종류 제한은 없음
- 현재 의미 규칙:
  - 빈 문자열 불가
  - 회원가입 시 비밀번호 확인 입력과 일치해야 함
  - 현재 과제 범위에서는 평문 저장

### 3. 날짜 입력

- 현재 구현 위치: `src/cli/validators.py`
- 현재 문법 규칙:

```regex
^\d{4}-\d{2}-\d{2}$
```

- 현재 의미 규칙:
  - 실제 존재하는 날짜여야 함
  - 오늘보다 과거일 수 없음
  - 기획 의도는 "오늘 포함 14일 이내"지만, 현재 구현 상한은 `today + 14 days`라서 달력상 15개 날짜가 허용될 수 있음

### 4. 시간 입력

- 현재 구현 위치: `src/cli/validators.py`
- 현재 문법 규칙:

```regex
^\d{2}:\d{2}$
```

- 현재 의미 규칙:
  - 실제 존재하는 시간이어야 함
  - 30분 단위여야 함
  - 날짜와 결합했을 때 현재 시각보다 과거면 불가

### 5. 정수 입력

- 현재 구현 위치: `src/cli/validators.py`
- 현재 문법 규칙:
  - 숫자 문자열만 허용 (`isdigit()` 기준)
  - 음수, 소수점, 부호(`+`, `-`) 불허
- 현재 의미 규칙:
  - 기본 `validate_positive_int()` 범위는 `1~100`
  - 관리자 파손/오염 패널티는 `1~5`만 허용

### 6. 자유 텍스트 입력

- 현재 구현 위치:
  - `src/cli/admin_menu.py`
- 현재 문법 규칙:
  - `취소 사유`, `사유` 입력은 모두 `input(...).strip()` 후 사용한다.
  - 길이 제한, 허용 문자 제한, 줄바꿈 정책은 현재 코드에 없다.
- 현재 의미 규칙:
  - 관리자 예약 취소 사유는 빈 문자열이어도 처리된다.
  - 파손/오염 사유는 빈 입력이면 기본값 `파손/오염`으로 대체된다.

### 7. 최상위 메뉴 선택 입력

- 현재 구현 위치:
  - `src/cli/guest_menu.py`
  - `src/cli/user_menu.py`
  - `src/cli/admin_menu.py`
- 현재 문법 규칙:
  - 각 화면은 `input("선택: ").strip()` 결과를 문자열 그대로 비교한다.
  - 따라서 표시된 번호와 정확히 같은 문자열만 성공한다.
  - 예: `1`은 허용되지만 `01`, `+1`은 허용되지 않는다.
- 현재 의미 규칙:
  - 정의된 번호 외 입력은 모두 "잘못된 선택"으로 처리된다.
  - 빈 입력은 별도 메시지 없이 같은 화면에서 다시 선택하게 된다.

### 8. 공용 menu choice validator

- 현재 구현 위치: `src/cli/menu.py`, `src/cli/validators.py`
- 현재 문법 규칙:
  - `validate_menu_choice()`는 `isdigit()` 기준 숫자만 허용한다.
  - 반면 `BaseMenu`, `MenuRouter`, `select_from_list()` 쪽은 `int()` 변환 기반이라 허용 토큰 해석이 완전히 같지 않다.
- 현재 의미 규칙:
  - 각 메뉴의 허용 범위 안에 있어야 함
  - 일부 메뉴는 `0=취소/종료`를 허용함

### 9. 확인(confirm) 입력

- 현재 구현 위치: `src/cli/menu.py`
- 현재 문법 규칙:
  - 입력값은 `strip().lower()` 후 비교한다.
  - 긍정: `y`, `yes`, `예`, `ㅇ`
  - 부정: `n`, `no`, `아니오`, `ㄴ`
- 현재 의미 규칙:
  - 허용 토큰이 아니면 재입력을 요구한다.
  - 대문자 `Y`, `YES`, 공백 포함 ` yes ` 같은 입력도 trim/lower 후 허용된다.

4. 빈 입력은 계속 재입력으로 둘 것인가, 취소로 볼 것인가?
5. 긴 목록에서 페이지네이션/검색이 필요해질 경우 입력 규칙을 어떻게 확장할 것인가?

### I-2. 문법(syntax) 관점 안건

- 허용 토큰은 숫자만으로 유지할 것인가?
- 숫자 범위 표현을 메뉴 선택과 동일 규칙으로 통합할 것인가?

예시 BNF:

```bnf
<selection> ::= "0" | <positive-integer>
<positive-integer> ::= <non-zero-digit> | <non-zero-digit> <digit-seq>
<digit-seq> ::= "" | <digit> <digit-seq>
```

### I-3. 의미(semantics) 관점 안건

- `0`의 의미를 항상 “취소/복귀”로 고정할지 여부
- 범위를 벗어난 값 입력 시 허용 메시지 형식을 통일할지 여부
- 항목이 0개일 때 선택 UI를 띄우지 않고 즉시 종료하는 현재 정책을 유지할지 여부

## J. 진행/계속(pause) 입력 정책

- 현재 구현 위치: `src/cli/menu.py`의 `pause()`
- 현재 문법 규칙:
  - 화면 안내 문구는 Enter를 요구하지만, 실제 구현은 어떤 문자열을 입력해도 다음 화면으로 진행한다.
- 현재 의미 규칙:
  - 이 입력은 값 자체를 저장하지 않고, 단순히 흐름 진행 신호로만 사용한다.

### J-1. 팀 회의에서 결정해야 할 질문

1. 계속 진행 입력은 Enter만 허용할 것인가?
2. `q`, `취소`, 아무 키나 입력 등 다른 진행 방식도 허용할 것인가?
3. 이 프롬프트에서 시간 초과(timeout)나 자동 진행이 필요한가?
4. 모바일/원격 환경을 고려해 더 명시적인 입력 방식이 필요한가?

### J-2. 문법(syntax) 관점 안건

- Enter만 유효 입력으로 볼 것인가?
- 다른 문자열 입력 시 무시하고 진행할지, 에러 메시지를 띄울지?

### J-3. 의미(semantics) 관점 안건

- 이 입력의 의미를 항상 “다음 화면으로 넘어감”으로 고정할 것인가?
- 일부 위험한 화면에서는 단순 Enter가 아니라 명시적 확인을 요구할 것인가?

## K. 문법 규칙과 의미 규칙을 분리해서 관리할지 여부

회의에서 가장 먼저 합의해야 할 메타 정책이다.

### K-1. 문법 규칙(syntax) 정의 원칙

- “문자들의 나열” 자체만 본다.
- 예:
  - 길이
  - 허용 문자
  - 구분자 위치
  - 날짜/시간 포맷

### K-2. 의미 규칙(semantics) 정의 원칙

- 문법만으로 표현하기 어려운 규칙을 본다.
- 예:
  - 중복 사용자명 금지
  - 과거 날짜 금지
  - 예약 가능 기간 14일 제한
  - 패널티 상태에 따른 예약 제한

### K-3. 권장 적용 순서

1. 문법 규칙 먼저 검증
2. 문법이 통과하면 의미 규칙 검증
3. 의미 규칙까지 통과한 입력만 저장/처리

이 순서를 문서화해두면,
팀원마다 “형식 오류”와 “정책 위반” 메시지를 뒤섞지 않게 된다.

## 요구자 관점 권장 진행 순서

요구자가 실제로 이 문서를 활용할 때는 아래 순서로 진행하는 것이 가장 효율적이다.

1. 입력 항목 전체 목록을 먼저 확정한다.
2. 각 항목별로 문법 규칙을 먼저 정한다.
3. 그 다음 의미 규칙을 정한다.
4. 예외 상황과 오류 메시지 정책을 정한다.
5. 마지막으로 개발자에게 공통 입력 컴포넌트/validator로 통일 구현을 요구한다.

즉, 요구자는 단순히 "검사 강화"를 요청하는 것이 아니라,
"무슨 입력을, 어떤 문법으로, 어떤 의미 규칙으로, 어떤 오류 처리 방식으로 받을지"
를 순서대로 확정해서 전달해야 한다.

## 권장 결론 초안

현재 구현과 가장 충돌이 적은 기본안은 아래와 같다.

- 숫자 입력: 정수만 허용
- 사용자명: `^[A-Za-z0-9_]{3,20}$`
- 비밀번호: 4~50자, 공백 trim 유지 여부는 별도 확정 필요
- 날짜: `YYYY-MM-DD`
- 시간: `HH:MM`, 30분 단위
- 검증 순서: 문법 규칙 -> 의미 규칙
- 자유 텍스트: 길이 제한과 허용 문자 범위를 별도 정책으로 확정

다만 위 초안은 “현재 코드와 가장 가깝다”는 의미일 뿐,
최종 정책은 반드시 요구자/팀 회의를 통해 확정해야 한다.
