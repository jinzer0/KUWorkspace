# KUWorkspace 2차 기능 구현 계획 (plan_0001)

## 목적/비목표

이 문서는 `docs/memo/2차 기능 설계문서 원판.md`와 `docs/memo/2차 기능 기획서.md`를 현재 코드베이스에 적용하기 위한 구현 계획이다. 목표는 1차 기능과 파일 기반 CLI 구조를 보존하면서 2차 기능을 단계적으로 추가하는 것이다.

포함 범위는 예약 `pending` 우선권, 예약 메모, 장비 묶음 예약, 회의실 정기 점검, 장비 미래 상태 변경, 반복 취소 제한, 취소 전 패널티 preview, 운영 시계 자동 처리 확장, 신규 데이터 파일/스키마 검증이다.

비목표는 실제 코드 구현, 대형 manager/viewer/scheduler 프레임워크 도입, `system_clock.txt` 지원, `waiting_list.txt` 별칭 지원, curses/CalendarOverlay 필수 구현, 1차 메뉴 번호/흐름 전면 재배치다. 문서상 클래스명이 현재 구조와 맞지 않는 경우에는 기존 service/menu/repository에 매핑하거나 thin adapter로만 둔다.

## 선결 결정표

| 항목 | 결정 | 구현 영향 |
| --- | --- | --- |
| 운영 시계 파일 | `data/clock.txt`만 사용 | `system_clock.txt`는 문서 불일치로 비채택한다. |
| 운영 시점 | 09:00, 18:00만 허용 | 10:00/19:00 cutoff 표현은 실제 clock slot이 아니라 문구/정책 잔재로 둔다. |
| pending 의미 | 우선권 대기 상태 | `pending`은 확정 점유가 아니며 승격 대상이다. |
| reserved 의미 | 확정 점유 상태 | 자원 충돌 및 사용 상태 판단의 기준이다. |
| 대기 source of truth | booking 파일의 `status=pending` | `room_bookings.txt`, `equipment_booking.txt`가 권위 데이터다. |
| `waitlist.txt` | 비권위 projection/cache 또는 no-read 호환 파일 | 서비스 판단에서 읽지 않는다. 필요 시 pending rows에서 재생성한다. |
| `waiting_list.txt` | 비채택 별칭 | 생성/읽기/문서화하지 않는다. |
| pending tie-breaker | `penalty_points ASC`, `created_at ASC`, `booking_id ASC` | 동률 승격이 항상 결정적이어야 한다. |
| 정기 점검 순서 | 점검 전이를 pending 승격보다 먼저 처리 | 점검 중 자원에 pending이 승격되는 역전을 막는다. |
| 장비 묶음 예약 | 저장된 `group_id` 단위 변경/취소/픽업/반납은 all-or-none | 생성 전 후보 장비 필터링은 가능하나 저장 후 그룹은 원자 단위다. |
| 반복 취소 | `PenaltyService`가 소유 | `PenaltyRepository` 직접 append를 UI/service caller에서 금지한다. |
| 취소 preview | 사용자 확인 전에 service가 계산 | CLI는 preview 표시와 confirm만 담당한다. |
| FORCE/no_show | 핵심 기능보다 정책 정렬/회귀 테스트 항목 | 별도 권한 시스템으로 과확장하지 않는다. |
| CalendarOverlay/curses | 2차 core에서 제외 | 텍스트 날짜 입력을 수용한다. 나중에 구현 시 별도 TTY/fake-key QA가 필요하다. |

## 현재 구현 기준

현재 저장소는 로컬 텍스트 파일 기반 CLI 애플리케이션이다. 데이터는 `data/*.txt`에 pipe 구분 레코드로 저장되고, 저장 계층은 `BaseRepository`, `global_lock`, `UnitOfWork`를 사용한다.

확인된 기준은 다음과 같다.

| 파일 | 현재 상태 | 계획상 의미 |
| --- | --- | --- |
| `src/config.py` | `CLOCK_FILE = DATA_DIR / "clock.txt"`, `CLOCK_SENTINEL = "0000-00-00T00:00"` | 신규 파일 상수와 `DATA_FILES` 등록을 추가하되 clock 파일명은 유지한다. |
| `src/runtime_clock.py` | 허용 slot은 `(9, 0)`, `(18, 0)` | 09/18 모델을 보존한다. |
| `src/domain/models.py` | booking status에 `PENDING` 없음, penalty reason에 `FREQUENT_CANCEL` 없음 | enum과 dataclass record 폭 확장이 필요하다. |
| `src/storage/repositories.py` | active 조회와 conflict 조회가 같은 status 집합에 의존 | quota/conflict/pending competition 조회를 분리해야 한다. |
| `src/domain/policy_service.py` | `global_lock()` + `UnitOfWork()` 안에서 `self.clock.advance()`가 즉시 `clock.txt`를 persist | clock/data 원자 성공 단위를 보강해야 한다. |
| `src/storage/jsonl_handler.py` | `|`, `\` escape와 `\-` None sentinel 사용 | migration test가 escape/sentinel/빈 문자열을 포함해야 한다. |
| `docs/plan/` | 기존에 없음 | 본 계획 파일 작성 대상이다. |

## 데이터/스키마 전략

스키마 변경은 반드시 reader 호환부터 시작한다. 기존 `data/*.txt`와 테스트 fixture를 읽을 수 있게 한 뒤 canonical writer를 새 폭으로 전환한다. 순서는 old reader, new model defaults, record-width tests, new writer, seed/integrity update, service wiring이다.

| 데이터 | 기존 폭 | 신규 폭/필드 | reader 기본값 | writer 원칙 |
| --- | --- | --- | --- | --- |
| `users.txt` | 8 fields | `room_cancel_restricted_until`, `equipment_cancel_restricted_until` 추가 | 누락 시 `None` | 신규 canonical 폭으로 저장한다. 정확한 필드 순서는 구현 전 테스트에 고정한다. |
| `room_bookings.txt` | 13 fields | 14 fields, trailing `memo` | `memo=""` | `PENDING` enum과 `memo`를 포함해 저장한다. |
| `equipment_booking.txt` | 13 fields | 15 fields, trailing `group_id`, `memo` | `group_id=None`, `memo=""` | 같은 묶음은 동일 `group_id`를 저장한다. |
| `equipments.txt` | 7 fields | 8 fields, `future_status_changes` | 빈 schedule | 장비 미래 상태 변경을 직렬화한다. |
| `room_maintenance.txt` | 없음 | 신규 회의실 점검 schedule | 빈 파일 허용 | `RoomMaintenanceSchedule`/repository를 추가한다. |
| `waitlist.txt` | 없음 | projection/cache 또는 no-read 호환 파일 | 빈 파일 허용 | 권위 데이터가 아니며 pending booking rows에서 재생성 가능해야 한다. |
| `penalties.txt` | 기존 reasons | `frequent_cancel` reason 추가 | 기존 reason 유지 | `PenaltyService` 경유로 append한다. |
| `audit_log.txt` | 기존 actions | pending/group/maintenance/frequent_cancel actions 추가 | 기존 log 유지 | 정책 전이마다 audit를 남긴다. |

## 구현 Phase 계획

### Phase 0. 결정 고정 및 테스트 fixture 준비

이 문서의 결정표를 기준으로 `clock.txt`, 09/18 slot, booking `pending` 권위, `waitlist.txt` 비권위, CalendarOverlay 제외를 변경 불가 전제로 둔다. 기존 데이터와 fixture를 복사해 old-width 샘플을 만든다.

검증 게이트는 기존 `pytest`가 현 상태에서 통과하거나, 기존 실패가 있다면 phase-2와 무관한 baseline으로 기록하는 것이다.

### Phase 1. 스키마 reader와 신규 파일 foundation

`models.py`에 신규 필드를 default 포함으로 추가하되 먼저 old-width reader를 통과시킨다. `config.DATA_FILES`, `ensure_data_dir()`, integrity, seed script에 `room_maintenance.txt`, `waitlist.txt`를 등록한다. 이 단계에서 service/CLI는 아직 신규 기능을 호출하지 않는다.

검증 게이트는 old record read, new record read, old read 후 new write, repository reload, `validate_all_data_files()` 통과다.

롤백 기준은 신규 service wiring 없이 model/repository foundation만 남기는 것이다. reader default는 유지해도 1차 예약 흐름을 깨지 않아야 한다.

### Phase 2. repository query semantics 분리

`get_active_by_user()`를 quota 전용으로 정리하고, confirmed conflict query와 pending competition query를 별도 함수로 둔다. quota에는 `pending`을 포함해 spam을 막고, confirmed conflict에는 `pending`을 제외한다. pending competition은 같은 resource/time overlap의 pending만 모아 tie-breaker로 정렬한다.

검증 게이트는 active quota, confirmed conflict, pending competition 각각의 status set 테스트다.

### Phase 3. memo와 계정/검증 저위험 기능

예약 생성/변경에 `memo`를 추가하고, username/password/memo 검증을 확장한다. 신규 user cancel restriction 필드를 reader/writer에 포함하되 실제 반복 취소 정책 적용은 Phase 4에서 연결한다.

검증 게이트는 memo escape, 빈 memo, `\-` sentinel, pipe/backslash round-trip, seed 재생성 테스트다.

롤백 기준은 memo UI 입력만 끄고 reader/writer 기본 필드는 유지하는 것이다.

### Phase 4. frequent_cancel과 cancel preview

`PenaltyReason.FREQUENT_CANCEL`와 `PenaltyService.apply_frequent_cancel()`을 추가한다. 취소 시 service가 late cancel, frequent cancel, domain restriction 변화를 preview로 계산하고 CLI는 확정 전에 표시한다. 같은 취소에서 late_cancel과 frequent_cancel이 중복 부과되지 않도록 중앙 decision 함수로 결정한다.

검증 게이트는 최근 30일 산정, 예약 시작일 포함 14일 이전 제외, 3회째 제한, 4회 이상 추가 penalty, 회의실/장비 독립 제한, preview 후 confirm/cancel 흐름 테스트다.

롤백 기준은 frequent_cancel 적용 method와 CLI preview 연결만 끄고 기존 late_cancel 정책은 유지하는 것이다.

### Phase 5. room maintenance와 equipment future status

회의실 정기 점검은 `room_maintenance.txt`와 repository로 관리한다. 장비 미래 상태 변경은 `equipments.txt.future_status_changes`에 저장한다. 점검/미래 상태는 예약 가능성 검사와 clock automation에 연결하되, 먼저 service 단위에서 overlap과 취소 cleanup을 검증한다.

검증 게이트는 maintenance create/cancel, 점검 기간 예약 차단, future status start/end, future 작업 취소 시 stale cleanup 테스트다.

롤백 기준은 maintenance/future status service wiring을 끄면 기존 자원 상태 변경과 예약 흐름이 유지되는 것이다.

### Phase 6. pending 우선권과 waitlist projection

`RoomBookingStatus.PENDING`, `EquipmentBookingStatus.PENDING`을 service에 연결한다. 충돌 가능 예약은 문서 정책에 따라 `pending`으로 생성하고, 시점 이동에서 tie-breaker로 `reserved` 승격 또는 `cancelled` 탈락을 처리한다. `waitlist.txt`는 필요 시 pending rows에서 생성되는 projection/cache로만 두며 service decision에서 읽지 않는다.

검증 게이트는 단독 pending 즉시/시점 승격, 경쟁 pending tie-break, penalty_points 우선순위, 같은 `created_at`에서 `booking_id` 동률 해소, pending quota 포함, `waitlist.txt` 비권위 테스트다.

롤백 기준은 pending 생성 경로를 막고 기존 `reserved` 예약 경로만 유지하는 것이다. 기존 pending record reader는 남겨도 된다.

### Phase 7. equipment group booking

장비 묶음 예약은 `EquipmentService` 안에서 구현한다. 생성 전에는 요청 장비별 가능 여부를 평가할 수 있으나, 저장된 booking rows는 동일 `group_id`를 공유한다. 저장 후 변경/취소/픽업/반납/조회는 group 단위 all-or-none으로 처리한다.

검증 게이트는 group create, group_id 동일성, 일부 후보 제외 정책, 저장 후 group cancel/change all-or-none, audit 1건과 실제 booking N건 정합성, failure rollback 테스트다.

롤백 기준은 group UI/service entry를 끄고 `group_id` reader default는 유지하는 것이다.

### Phase 8. `PolicyService.advance_time()` atomicity 보강

현재 `self.clock.advance()`는 UoW 중 `clock.txt`를 즉시 persist하므로, phase-2 automation 전에 clock persistence를 data mutation과 같은 성공/실패 단위로 맞춘다. 구현 방식은 staged clock write, clock repository, 또는 advance 후 실패 불가 구조 중 하나를 선택하되 테스트로 clock/data 불일치를 막아야 한다.

검증 게이트는 automation 중 예외 주입 시 clock이 전진하지 않는 테스트, audit 정합성 테스트, FORCE owner 테스트다.

### Phase 9. CLI 통합

service 테스트가 통과한 뒤 `UserMenu`, `AdminMenu`, `ClockMenu`에 연결한다. 기존 메뉴 번호는 가능한 유지한다. 저장이 발생하는 예약 생성/변경/취소/승인 경로에는 preview, confirm, `0. 돌아가기`, 오류 입력 처리를 추가한다. 날짜 선택은 text input을 사용하고 CalendarOverlay는 제외한다.

검증 게이트는 monkeypatch input으로 `0`, 빈 입력, 숫자 아님, 범위 밖, confirm 실패, 목록 재렌더, 사용자/관리자 이벤트 노출 차이를 확인하는 것이다.

### Phase 10. 전체 회귀와 수동 QA

전체 `pytest`를 실행하고, tmux 또는 동일한 실제 CLI 표면에서 `python main.py`를 실행해 사용자/관리자 핵심 시나리오를 수동 검증한다.

검증 게이트는 회원가입/로그인, room pending, equipment group, maintenance 차단, cancel preview, clock advance, FORCE 안내, 기존 1차 예약/승인/반납 흐름 통과다.

## 정책/상태 전이 규칙

pending/reserved 상태 의미는 다음과 같다.

| 상태 | 의미 | quota | confirmed conflict | pending competition |
| --- | --- | --- | --- | --- |
| `pending` | 우선권 대기 | 포함 | 제외 | 포함 |
| `reserved` | 확정 예약 | 포함 | 포함 | 제외 |
| `checkin_requested`, `pickup_requested` | 사용자 처리 요청 | 포함 | 포함 | 제외 |
| `checked_in`, `checked_out` | 사용 중 | 포함 | 포함 | 제외 |
| `checkout_requested`, `return_requested` | 종료 요청 | 포함 | 포함 | 제외 |
| `completed`, `returned`, `cancelled`, `admin_cancelled` | 종료 상태 | 제외 | 제외 | 제외 |

`PolicyService.advance_time()`의 목표 순서는 다음과 같다.

1. `global_lock()` 획득 및 UoW 시작.
2. `validate_all_data_files()`와 advance blocker 계산.
3. 현재 slot의 미처리 시작/종료 이벤트 처리.
4. 정기 점검 시작/종료, 장비 future status start/end/cancel cleanup 적용.
5. pending competition resolver 실행.
6. 제한 해제, 이용 금지, 90일 초기화 등 기존 maintenance 정책 실행.
7. clock 값을 다음 slot으로 staged persistence 처리.
8. audit log 기록.
9. data files와 clock file을 하나의 성공 단위로 commit.

FORCE 책임은 테스트 가능한 표로 고정한다.

| 상황 | 책임/패널티 원칙 |
| --- | --- |
| 관리자/system FORCE | 원 예약자 정책을 유지하되 audit에 actor를 기록한다. |
| 일반 사용자 FORCE | 해당 사용자의 강행 책임을 명시하고 패널티 귀속을 테스트한다. |
| 이미 요청된 checkin/pickup/checkout/return 자동 승인 | 패널티 없음. |
| 미요청 종료/반납 | 기존 late_return 정책으로 처리한다. |
| 미요청 시작 | `no_show` reason을 새 enum으로 되살리지 않는다. 자동 취소/패널티 여부는 구현 전 테스트에서 기존 정책과 문서 요구를 대조해 한 decision 함수에 고정한다. |

취소 패널티는 `PenaltyService`의 중앙 decision에서 계산한다. late_cancel, frequent_cancel, FORCE 귀속, no_show legacy cleanup이 여러 service에서 중복 계산되면 안 된다.

## CLI/UI 수용 기준

CLI는 입력과 표시, service는 정책 계산과 저장을 담당한다. CLI가 repository를 직접 조작하거나 패널티를 직접 append하면 안 된다.

| 입력 상황 | 수용 기준 |
| --- | --- |
| `0` | 현재 절차를 중단하고 상위 메뉴로 복귀한다. 저장 전 상태여야 한다. |
| 빈 입력 | 문서 요구 메시지를 출력하고 같은 목록/단계로 돌아온다. |
| 숫자 아님 | 숫자 입력 요구 메시지를 출력하고 재입력한다. |
| 범위 밖 | 선택 범위 오류 메시지를 출력하고 목록을 고정 재렌더한다. |
| confirm 실패 | 저장하지 않고 preview 또는 이전 단계로 돌아간다. |
| 날짜 입력 | CalendarOverlay 대신 text date input을 사용한다. |
| 사용자/관리자 clock 결과 | actor 권한에 맞는 이벤트만 노출하고 audit는 전체 정보를 보존한다. |

CalendarOverlay/curses를 나중에 되살리려면 fake-key 테스트, 작은 터미널, 비-TTY fallback 또는 명시 실패 메시지, 한글 폭 검증, tmux 수동 QA를 별도 phase로 추가한다.

## 검증 게이트

검증은 아래 순서로 좁은 범위에서 넓은 범위로 진행한다.

| 순서 | 게이트 | 필수 확인 |
| --- | --- | --- |
| 1 | model unit tests | enum 추가, old/new `from_record()`, canonical `to_record()`, default 값 |
| 2 | storage encoding tests | pipe/backslash escape, 빈 문자열, `\-` sentinel, datetime minute normalization |
| 3 | repository migration tests | old read, new read, old read 후 new write, reload, `validate_all_data_files()` |
| 4 | service policy tests | memo, frequent_cancel, maintenance, pending, group booking, cancel preview |
| 5 | clock transition tests | atomicity, ordering, FORCE owner, audit, failure rollback |
| 6 | CLI monkeypatch tests | `0`, 빈 입력, 숫자 아님, 범위 밖, confirm, preview, 목록 재렌더 |
| 7 | E2E tests | user/admin room/equipment lifecycle, pending promotion, group cancellation, maintenance conflict |
| 8 | full regression | 전체 `pytest` |
| 9 | manual QA | tmux 또는 실제 터미널에서 `python main.py` 핵심 흐름 검증 |

각 phase는 자기 게이트가 통과하기 전 다음 phase에 연결하지 않는다. 전체 E2E 실패를 뒤늦게 디버깅하지 않도록 model/repository/service/CLI 순서로 고정한다.

## 롤백/제거 기준

| Phase | 제거 기준 | 보존할 것 |
| --- | --- | --- |
| 스키마 foundation | service wiring 제거 | old/new reader default, 신규 파일 생성 안전성 |
| memo | CLI 입력 제거 | `memo` reader/writer default |
| frequent_cancel | `apply_frequent_cancel()` 호출 제거 | 기존 late_cancel과 penalty flow |
| maintenance | service/CLI entry 제거 | `room_maintenance.txt` reader와 빈 파일 무결성 |
| future status | admin entry 제거 | equipment old/new reader |
| pending | pending 생성 경로 제거 | 기존 `reserved` 예약 생성/승인 흐름 |
| waitlist projection | projection writer 제거 | booking `pending` authority |
| group booking | group UI/service entry 제거 | `group_id` default reader와 단건 장비 예약 |
| clock atomicity refactor | phase-2 automation 연결 보류 | 기존 clock advance baseline |
| CLI 2차 화면 | 신규 화면 entry 제거 | 기존 1차 메뉴 번호와 주요 E2E |

롤백은 failing test를 삭제하거나 약화하는 방식이 아니다. 해당 phase의 service/CLI 연결만 끊고 persisted data reader 호환성은 유지한다.

## 잔여 리스크/후속 범위

- `users.txt` 신규 필드 순서는 구현 전 테스트에서 최종 고정해야 한다.
- 문서의 일반 활성 예약 3건과 현재 상수 1건의 차이는 `MAX_ACTIVE_*` 정책 변경 여부를 별도 decision으로 남긴다.
- username 첫 글자 대문자, password 영문+숫자 규칙은 기존 seed 계정 `admin/admin123`과 충돌할 수 있으므로 seed/test 갱신 계획을 포함해야 한다.
- 회의실명 `회의실 2A` vs `회의실2A` 형식 차이는 validator와 seed 기준을 맞춘다.
- `future_status_changes` 직렬화 형식은 escape/sentinel 규칙과 함께 테스트로 고정해야 한다.
- `waitlist.txt`가 채점상 파일 존재를 요구하는지 확인이 필요하다. 존재가 필요해도 권위 데이터로 쓰지 않는다.
- `no_show` legacy trace는 enum 재도입 없이 audit/penalty fixture cleanup 범위를 따로 정한다.
- CalendarOverlay/curses는 후속 UI phase로만 다룬다.
- clock/data 원자성 보강 방식은 구현자가 선택하되 실패 시 clock이 전진하지 않는 테스트가 반드시 먼저 있어야 한다.
