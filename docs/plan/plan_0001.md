# 최종 수정 계획 (plan_0001)

출처: Hyperplan team_create 실패로 사용자 승인 후 fallback 검수로 작성.

## 1. 검수 결론

`docs/memo/second_1.md`를 기준 명세로 본다. `docs/draft/draft_0001.md`의 주요 지적은 대부분 유효하지만, 일부 항목은 현재 구현에 이미 일부 기반이 있으므로 “전면 미구현”이 아니라 “명세 기준으로 모델/생성/상태 전이/CLI를 재정렬”하는 계획으로 다룬다.

| 초안 항목 | 검수 판정 | 최종 처리 방향 |
| --- | --- | --- |
| 활성 예약 한도 | 유효 | 정상 사용자는 회의실 3건, 장비 3건까지 허용하고 제한 사용자는 각 1건으로 유지한다. `src/config.py`, `src/domain/policy_service.py`, 회의실/장비 서비스, CLI 안내 문구를 함께 정렬한다. |
| 대기 예약 | 유효 | 현재 `waitlist.txt`는 pending 예약 projection에 가깝다. 명세의 `waiting_list.txt` 7필드 요청 모델, repository, 사용자 흐름을 별도로 추가한다. |
| 예약 우선권 | 부분 유효 | pending resolver 기반은 있으나 생성/누적 규칙과 18:00 다음날 선착순 예외를 명세 기준으로 보강해야 한다. |
| 회의실 정기 점검 | 유효 | 현재 reason 기반 7필드/삭제식 처리에서 명세의 8필드 status 기반 스키마와 `scheduled/active/completed/cancelled` 전이로 바꾼다. |
| CUI 복귀 정책 | 유효 | Enter-only `pause()` 중심 흐름을 `0. 돌아가기` 및 저장 전 검토 화면으로 전환한다. |
| 빈번 취소 패널티 | 유효 | 3회째부터 `frequent_cancel` 1점, 4회 이상 추가 1점, 직전 취소와 중복 시 frequent_cancel 미부과로 변경한다. |

| 메모 검증 | 유효 | 예약 메모 최대 50자, 줄바꿈 및 파일 구분 보호 문자 거부로 변경한다. |
| 장비 미래 상태 | 유효 | JSON 기반 rich schedule을 명세의 `날짜, 상태; ...` 저장 형식과 maintenance 중심 규칙에 맞춘다. |
| 관리자 자원 관리 메뉴 | 유효 | 회의실 추가/삭제/수정과 장비 `+. 편집` 흐름을 명세 위치와 조건에 맞춰 통합한다. |

추가로 반드시 포함할 사항은 다음과 같다.

- 파일 스키마 변경 전후의 round-trip 및 migration/compatibility 테스트를 먼저 둔다.
- policy, schema, CLI rewiring을 한 번에 섞지 말고 단계별로 독립 검증한다.
- 명세와 충돌하는 기존 `docs/plan/plan_0001.md`의 `waiting_list.txt 비채택`, `waitlist.txt 비권위 projection` 같은 결정을 폐기한다.

## 2. 실행 원칙

- 각 phase는 실패하는 테스트를 먼저 추가한 뒤 구현한다.
- `docs/memo/second_1.md`를 source of truth로 둔다.
- persisted data 파일 형식이 바뀌는 phase는 reader/migration/무결성 검증을 먼저 처리한다.
- UI 변경은 domain/service 정책이 통과한 뒤 연결한다.
- 실패 테스트를 삭제하거나 약화해서 통과시키지 않는다.
- 기존 데이터 호환은 실제 `data/*.txt` 또는 fixture가 깨질 수 있는 범위에 한해서만 둔다.

## 4. Phase 1 - 데이터 스키마와 저장 계층 정렬

목표: policy와 CLI를 바꾸기 전에 파일 형식을 안전하게 맞춘다.

대상 파일:

- `src/config.py`
- `src/domain/models.py`
- `src/storage/repositories.py`
- `src/storage/integrity.py`
- `scripts/seed_data.py`
- 관련 model/repository 테스트

작업:

- `WAITLIST_FILE`을 `data/waiting_list.txt`로 변경하고 `DATA_FILES`에 등록한다.
- 명세 필드 `waiting_id|username|related_type|related_id|user_count|created_at|updated_at`를 가진 waiting-list 모델과 repository를 추가한다.
- 기존 pending projection용 `waitlist.txt` 로직은 제거하거나 명확히 격리해 권위 데이터와 혼동되지 않게 한다.
- `RoomMaintenanceSchedule`을 `schedule_id|room_id|start_time|end_time|status|created_at|updated_at|cancelled_at` 구조로 변경한다.
- 7필드 기존 정기 점검 레코드는 안전 migration 또는 명확한 integrity error 중 하나로 정책을 고정한다.
- 장비 `future_status_changes`를 JSON 배열이 아니라 명세의 `날짜, 상태; ...` 형식으로 읽고 쓰도록 바꾼다.
- 기존 JSON 장비 미래 상태 데이터는 호환 읽기 또는 명시 migration 테스트를 둔다.
- seed 데이터와 fixture를 강화된 회원가입 규칙에 맞춘다.

검증 게이트:

- 신규 스키마 레코드가 round-trip 된다.
- `ensure_data_dir()`가 `waiting_list.txt`, `room_maintenance.txt`를 포함해 필수 파일을 만든다.
- `validate_all_data_files()`가 신규 스키마를 검증한다.
- 호환 대상 기존 레코드는 migration 되거나 의도한 integrity error를 낸다.

## 5. Phase 2 - 핵심 예약/패널티 정책 정렬

목표: 예약 한도, pending 우선권, 반복 취소 패널티를 명세와 맞춘다.

대상 파일:

- `src/config.py`
- `src/domain/policy_service.py`
- `src/domain/room_service.py`
- `src/domain/equipment_service.py`
- `src/domain/penalty_service.py`
- `src/domain/restriction_rules.py`
- `src/cli/user_menu.py`

작업:

- `MAX_ACTIVE_ROOM_BOOKINGS`, `MAX_ACTIVE_EQUIPMENT_BOOKINGS`를 정상 사용자 기준 3으로 변경한다.
- 제한 사용자는 회의실 1건, 장비 1건만 허용하도록 별도 제한 계산을 둔다.
- `check_user_can_book()`, `get_max_bookings_for_user()`, `get_user_flow_limits()`의 하드코딩된 1건 로직을 제거한다.
- 회의실/장비 service 기본 max_active가 정상 사용자 3건을 사용하도록 정렬한다.
- 동일 운영 시점/동일 자원/동일 기간 경쟁 신청이 pending으로 누적될 수 있게 생성 로직을 수정한다.
- 단독 비충돌 신청은 즉시 `reserved`로 확정한다.
- 시점 이동 시 pending 경쟁은 penalty_points 오름차순, created_at 오름차순으로 확정/탈락 처리한다.
- 현재 시점이 18:00이고 예약 시작일이 다음날인 경우 첫 신청만 즉시 `reserved`, 이후 동일 자원/기간 신청은 생성하지 않고 거부한다.
- `FREQUENT_CANCEL_PENALTY`를 1로 변경한다.
- 3회째 qualifying cancel부터 frequent_cancel 패널티와 제한을 적용한다.
- 4회 이상 qualifying cancel은 제한 종료일을 갱신하지 않고 추가 1점만 부과한다.
- 같은 취소가 late_cancel이면 frequent_cancel 점수는 중복 부과하지 않는다.

검증 게이트:

- 예약 한도, pending resolver, 18:00 예외, frequent_cancel 단위 테스트가 통과한다.
- confirmed 예약 간 이중 예약이 생기지 않는다.
- 제한/이용금지 상태 메시지가 명세 문구와 맞는다.

## 6. Phase 3 - 대기 예약, 정기 점검, 장비 미래 상태 동작 구현

목표: 스키마 기반 기능을 service와 운영 시계에 연결한다.

대상 파일:

- `src/domain/models.py`
- `src/storage/repositories.py`
- `src/domain/room_service.py`
- `src/domain/equipment_service.py`
- `src/domain/policy_service.py`
- `src/cli/user_menu.py`
- `src/cli/admin_menu.py`
- `src/cli/clock_menu.py`

작업:

- 사용자 대기 예약 신청 흐름을 추가하고 `waiting_list.txt`에 저장한다.
- waiting-list를 pending 예약 projection이 아니라 별도 요청 데이터로 다룬다.
- 회의실 정기 점검 시작일은 18:00, 종료일은 09:00으로 해석한다.
- 정기 점검 상태를 `scheduled`, `active`, `completed`, `cancelled`로 전이한다.
- 정기 점검 취소는 레코드 삭제가 아니라 `cancelled`와 `cancelled_at` 갱신으로 처리한다.
- 정기 점검 종료는 삭제가 아니라 `completed` 전이로 처리한다.
- 예약 가능 회의실 목록에서 겹치는 scheduled/active 정기 점검을 제외한다.
- 장비 미래 상태 저장을 `날짜, 상태; ...` 형식으로 통일한다.
- 미래 장비 상태는 명세상 maintenance 중심으로 제한하고, 즉시 disabled 상태 변경은 기존 상태 변경 기능과 분리한다.
- 장비 예약 가능성 판단에 미래 maintenance를 반영한다.

검증 게이트:

- `waiting_list.txt`가 명세 7필드로 기록된다.
- `room_maintenance.txt`가 cancelled/completed 레코드를 보존한다.
- 시점 이동이 정기 점검 status를 정확히 갱신한다.
- 장비 미래 상태가 명세 형식으로 저장/조회된다.

## 7. Phase 4 - 관리자 자원 관리 메뉴 정렬

목표: 관리자 CUI를 명세의 메뉴 위치와 기능 조건에 맞춘다.

대상 파일:

- `src/cli/admin_menu.py`
- `src/domain/room_service.py`
- `src/domain/equipment_service.py`
- `src/domain/field_rules.py`
- `src/storage/repositories.py`

작업:

- 관리자 회의실 수정 메뉴를 추가한다.
- 회의실 추가는 최대 20개, 이름 중복 금지, 수용 인원/위치 검증, 초기 `available` 상태를 적용한다.
- 회의실 삭제는 available, 활성/미래 예약 없음, 정기 점검 없음 조건에서만 허용한다.
- 회의실 수정은 available이고 예약이 없을 때 수용 인원과 위치만 변경하도록 한다.
- 장비 목록 하단 `+. 편집` 흐름을 구현한다.
- 장비 이름 수정은 대여 중 장비에서 금지한다.
- 장비 삭제는 대여 중/예약 중 장비에서 금지하고 최소 12개 보유 조건을 지킨다.
- 장비 추가는 최대 20개, 기존 종류 선택 또는 직접 입력, 시리얼 자동 부여, 이름/종류/시리얼 검증을 적용한다.
- 기존 별도 20~23번 장비 미래 상태/취소 메뉴가 명세 흐름과 충돌하면 통합 또는 제거한다.

검증 게이트:

- 관리자 메뉴 dispatch 테스트가 통과한다.
- 회의실/장비 add/delete/edit service 테스트가 통과한다.
- 실패한 delete/edit 시 데이터가 변경되지 않는다.

## 8. Phase 5 - 입력 검증과 CUI 복귀/검토 정책

목표: 전역 입력 규칙과 사용자 상호작용 방식을 명세와 맞춘다.

대상 파일:

- `src/domain/field_rules.py`
- `src/cli/validators.py`
- `src/cli/menu.py`
- `src/cli/guest_menu.py`
- `src/cli/user_menu.py`
- `src/cli/admin_menu.py`
- `src/cli/clock_menu.py`

작업:

- 사용자명은 3~20자, 첫 글자 A-Z, 영문/숫자/`_`만 허용, 공백 금지를 검증한다.
- 비밀번호는 4~50자, 영문자 1개 이상, 숫자 1개 이상, 공백 금지를 검증한다.
- 회원가입 prompt와 CLI validator 메시지를 갱신한다.
- 예약 메모 최대 길이를 50자로 변경한다.
- 예약 메모에서 줄바꿈, `|`, `\`를 거부한다.
- 조회/결과 화면의 Enter-only `pause()`를 `0. 돌아가기` 입력으로 대체한다.
- 저장/변경/취소/승인/패널티 부여 직전에는 `1. 저장/처리`, `2. 다시 입력`, `0. 취소` 검토 화면을 둔다.
- 값 입력 필드 자체에는 무분별하게 `0. 돌아가기`를 넣지 않고, 입력 시작 전 `1. 입력 시작`, `0. 돌아가기` 화면을 제공한다.

검증 게이트:

- validator 테스트가 통과한다.
- 주요 mutation flow는 검토 확인 전 파일을 쓰지 않는다.
- 조회/결과 화면은 Enter-only 복귀에 의존하지 않는다.

- 정상 사용자로 장비 3건 예약 후 4번째 차단을 확인한다.
- 제한 사용자가 회의실/장비 각 1건 이후 차단되는지 확인한다.
- 동일 자원/기간 pending 경쟁을 만들고 시점 이동으로 확정/탈락을 확인한다.
- 18:00 다음날 예약 선착순 예외를 확인한다.
- 3회째/4회 이상 반복 취소 패널티와 late_cancel 중복 방지를 확인한다.
- 관리자 회의실 추가/삭제/수정을 확인한다.
- 장비 `+. 편집`에서 수정/삭제/추가를 확인한다.
- 회의실 정기 점검 등록/취소/진행/완료를 확인한다.
- 장비 미래 maintenance 등록 후 예약 가능성 반영을 확인한다.

데이터 호환성 확인 파일:

- `data/users.txt`
- `data/rooms.txt`
- `data/equipments.txt`
- `data/room_bookings.txt`
- `data/equipment_booking.txt`
- `data/room_maintenance.txt`
- `data/waiting_list.txt`
- `data/penalties.txt`
- `data/audit_log.txt`

최종 exit gate:

- 전체 테스트가 통과한다.
- 수동 CLI smoke 시나리오가 통과한다.
- 파일 round-trip에서 필드 손실, 빈 줄, 잘못된 delimiter, 의도하지 않은 레코드 삭제가 없다.

## 10. 권장 구현 순서

1. 한 phase의 실패 테스트만 추가한다.
2. 해당 phase를 통과시키는 최소 구현만 한다.
3. phase별 검증 게이트를 실행한다.
4. 녹색 상태를 확인한 뒤 다음 phase로 이동한다.
5. schema 변경, policy 변경, menu rewiring을 한 slice에 섞지 않는다.

## 12. 리스크

- 강화된 사용자명 검증은 기존 lowercase 계정 데이터와 충돌할 수 있다. seed/fixture 또는 명시 migration 기준을 먼저 정한다.
- room maintenance와 equipment future status 스키마 변경은 startup integrity를 깨기 쉽다. compatibility/migration 테스트가 먼저 필요하다.
- pending 경쟁은 생성, availability, 시점 이동을 동시에 건드린다. 부분 구현 시 중복 confirmed 예약이 생길 수 있다.
- CUI 복귀 정책은 호출 지점이 많다. 데이터 변경 flow를 우선하고 단순 조회 화면은 후순위로 정리한다.
- 장비 미래 상태는 현재 JSON 모델보다 표현력이 줄어든다. 이 축소가 명세 정렬이라는 점을 테스트로 고정한다.
