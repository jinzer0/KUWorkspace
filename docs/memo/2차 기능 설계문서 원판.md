## **1.3.1 SystemClock 클래스**

### 클래스 개요

| 클래스명 | 역할 |
| ----- | ----- |
| SystemClock | 현재 운영 시점을 관리 및 제어 |

### UML-스타일 요약

* **생성자:** \+ SystemClock(start\_time : datetime)  
* **속성:** \- \_current\_time : datetime  
* **메서드:** \+ now() : datetime, \+ now\_iso() : str, \+ current\_slot() : str, \+ next\_slot() : datetime, \+ advance() : datetime, \+ set\_time(new\_time : datetime) : datetime

### 속성 상세

| 구분 | 변수명 | 설명 |
| ----- | ----- | ----- |
| 전역 변수 | ALLOWED\_CLOCK\_SLOTS | 허용되는 운영 시점 집합 {(09:00), (18:00)} |
| 전역 변수 | \_active\_clock | 현재 세션에서 활성화된 운영 시계 객체 |
| 전역 변수 | \_runtime\_clock | 서비스 계층에서 사용하는 RuntimeClock 객체 |
| 인스턴스 변수 | \_current\_time | 현재 운영 시점을 저장하는 datetime 객체 |

### 메서드 상세

| 함수명 | 설명 |
| ----- | ----- |
| now() | 현재 운영 시점을 반환하는 함수 |
| now\_iso() | 현재 운영 시점을 문자열 형식으로 반환하는 함수 |
| current\_slot() | 현재 운영 시각을 반환하는 함수 |
| next\_slot() | 다음 운영 시점을 계산하는 함수 |
| advance() | 다음 운영 시점으로 이동하는 함수 |
| set\_time() | 현재 운영 시점을 변경하는 함수 |

### 동작 상세 설명

1. **now()** 현재 운영 시점을 datetime 객체 형태로 반환한다.  
2. **now\_iso()** 현재 운영 시점을 ISO 형식 문자열로 반환한다.  
3. **current\_slot()** 현재 운영 시각이 09:00인지 18:00인지 판별하여 반환한다.  
4. **next\_slot()** 현재 시점이 09:00이면 같은 날짜의 18:00을 반환하고, 18:00이면 다음 날짜의 09:00을 반환한다.  
5. **advance()** 다음 운영 시점으로 이동하고 변경된 시점을 저장한다.  
6. **set\_time()** 입력받은 시점으로 현재 운영 시점을 변경한다.

## **1.3.5 PolicyService 클래스**

### 클래스 개요

| 클래스명 | 역할 |
| ----- | ----- |
| PolicyService | 시점 이동 관련 정책 점검 및 자동 처리를 담당하는 서비스 |

### UML-스타일 요약

* **메서드:** \+ prepare\_advance(), \+ advance\_time(), \+ run\_all\_checks()

### 메서드 상세

| 함수명 | 설명 |
| ----- | ----- |
| prepare\_advance() | 시점 이동 가능 여부와 예상 이벤트를 조회하는 함수 |
| advance\_time() | 다음 운영 시점으로 이동하는 함수 |
| run\_all\_checks() | 전체 정책 점검을 수행하는 함수 |

advance\_time() 장비 미래 상태 변경 적용: 모든 장비의 future\_status\_changes 를 순회하며 현재 운영 시점 이하인 레코드를 해당 장비의 status에 즉시 적용하고 해당 레코드를 제거한다. 변경 발생 시 파일에 저장.

### 동작 상세 설명

1. **prepare\_advance()** 현재 시점을 기준으로 다음 시점 이동 가능 여부와 발생 예정 이벤트를 조회한다.  
2. **advance\_time()** 운영 시계를 다음 시점으로 이동시키고 관련 자동 처리를 수행한다.  
3. **run\_all\_checks()** 패널티 초기화, 이용 제한 만료, 예약 취소 등 정책 검사를 수행한다.

### 자동 처리 기능

PolicyService 클래스의 *advance\_time()* 메서드 실행 시 수행되는 자동 처리 기능은 다음과 같다.

* 회의실 예약 상태 변경  
* 장비 예약 상태 변경  
* pending 예약 우선권 처리  
* 노쇼 처리  
* 지연 반납 처리  
* 이용 제한 갱신  
* 자원 상태 복구

# **1.4 회의실 관련 클래스 및 모듈**

## **1.4.1 UserRoomMenu**

\[클래스 개요\]

클래스명: UserRoomMenu

역할:  
사용자 회의실 메뉴의 기본 흐름은 1차 구현과 동일하게 사용한다.  
2차 구현에서는 회의실 목록 조회 시 정기 점검 예정 정보를 함께 출력하고, 회의실 예약 시 메모 입력 절차를 추가한다.

\[UML-스타일 요약\]

생성자:  
\+ UserRoomMenu(username : str, room\_service : RoomService, booking\_service : RoomBookingService)

속성:  
\- username : str  
\- room\_service : RoomService  
\- booking\_service : RoomBookingService

메서드:  
\+ run() : void  
\- show\_room\_list() : void  
\- create\_room\_booking() : void  
\- change\_room\_booking() : void  
\- request\_checkin() : void  
\- request\_checkout() : void

\[2차 변경 동작\]

1\. show\_room\_list()

회의실 목록 조회의 기본 흐름은 1차 구현과 동일하다.  
단, 2차 구현에서는 RoomService의 format\_maintenance\_display() 결과를 함께 받아 회의실 상태 오른쪽에 정기 점검 예정 정보를 출력한다.

출력 항목:  
\- 회의실 이름  
\- 수용 인원  
\- 위치  
\- 현재 상태  
\- 정기 점검 예정 정보

정기 점검 예정 정보가 없으면 "-"로 표시한다.  
정기 점검 예정 정보가 있으면 가장 가까운 scheduled 정기 점검 1건만 표시한다.

2\. create\_room\_booking()

회의실 예약 입력 흐름은 1차 구현과 동일하게 사용한다.  
단, 2차 구현에서는 예약 메모 입력 단계가 추가된다.

추가 입력:  
\- memo : str

memo는 빈 문자열을 허용한다.  
memo는 줄바꿈 문자를 포함할 수 없다.  
memo는 문자 기준 50자를 초과할 수 없다.

예약 생성 시 UserRoomMenu는 입력값을 직접 파일에 저장하지 않고 RoomBookingService.create\_booking()을 호출한다.

3\. change\_room\_booking()

예약 변경의 기본 흐름은 1차 구현과 동일하게 사용한다.  
단, 2차 구현에서는 새 예약 기간이 정기 점검 예정 기간과 겹치는지 추가로 검증한다.  
정기 점검 기간과 겹치면 변경하지 않고 오류 메시지를 출력한다.

4\. request\_checkin()

체크인 요청은 reserved 상태인 본인 예약에 대해서만 수행한다.  
예약의 start\_time이 현재 운영 시점과 정확히 일치하는 경우에만 요청 가능하다.  
요청 성공 시 예약 상태는 reserved에서 checkin\_requested로 변경된다.

5\. request\_checkout()

퇴실 신청은 checked\_in 상태인 본인 예약에 대해서만 수행한다.  
요청 성공 시 예약 상태는 checked\_in에서 checkout\_requested로 변경된다.  
실제 퇴실 승인은 관리자 기능에서 처리한다.

## **1.4.2 RoomService**

\[클래스 개요\]

클래스명: RoomService

역할:  
회의실의 기본 조회 기능은 1차 구현과 동일하게 사용한다.  
2차 구현에서는 정기 점검 예정 정보 조회와 예약 가능 회의실 필터링 조건이 추가된다.

\[UML-스타일 요약\]

속성:  
\- room\_repository : RoomRepository  
\- booking\_repository : RoomBookingRepository  
\- maintenance\_repository : RoomMaintenanceRepository

메서드:  
\+ get\_room\_list(current\_time : datetime) : list\[Room\]  
\+ get\_available\_rooms(user\_count : int, start\_date : date, end\_date : date, current\_time : datetime) : list\[Room\]  
\+ format\_maintenance\_display(room\_name : str, current\_time : datetime) : str

\[속성 상세\]

room\_repository:  
rooms.txt의 회의실 정보를 읽는 객체이다.

booking\_repository:  
room\_bookings.txt의 회의실 예약 정보를 읽는 객체이다.

maintenance\_repository:  
room\_maintenance.txt의 회의실 정기 점검 예약 정보를 읽는 객체이다.  
사용자 회의실 파트에서는 정기 점검을 생성하거나 취소하지 않고 조회만 수행한다.

\[메서드 상세\]

1\. get\_room\_list(current\_time : datetime) : list\[Room\]

회의실 목록을 rooms.txt에서 읽어 반환한다.  
목록 정렬 방식은 1차 구현과 동일하게 사용한다.  
2차 구현에서는 각 회의실에 대해 format\_maintenance\_display()를 호출하여 출력용 정기 점검 예정 정보를 함께 제공한다.

2\. get\_available\_rooms(user\_count : int, start\_date : date, end\_date : date, current\_time : datetime) : list\[Room\]

이용 인원과 예약 기간을 기준으로 예약 가능한 회의실 목록을 반환한다.

검증 조건:  
1\. 회의실 status가 available이어야 한다.  
2\. 회의실 capacity가 user\_count 이상이어야 한다.  
3\. 같은 회의실의 reserved, checkin\_requested, checked\_in, checkout\_requested 상태 예약과 기간이 겹치면 제외한다.  
4\. pending 상태 예약은 예약 가능 목록 필터링 단계에서는 충돌로 보지 않는다.  
5\. 예약하려는 기간이 scheduled 또는 active 정기 점검 기간과 겹치면 제외한다.

정렬 기준:  
1\. 수용 인원 오름차순  
2\. 수용 인원이 같으면 회의실 이름 오름차순

3\. format\_maintenance\_display(room\_name : str, current\_time : datetime) : str

room\_maintenance.txt에서 room\_name에 해당하는 정기 점검 예약을 조회한다.  
현재 운영 시점 이후의 scheduled 상태 정기 점검 중 가장 가까운 1건만 출력 대상으로 삼는다.

반환 규칙:  
\- 예정 정기 점검이 없으면 "-"  
\- 하루 일정이면 "YYYY-MM-DD"  
\- 여러 날짜에 걸친 일정이면 "YYYY-MM-DD \~ YYYY-MM-DD"

\[관리자 기능 분리\]

다음 함수는 사용자 회의실 기능이 아니라 관리자 회의실 기능이므로 1.4에서 제외하고 관리자 기능 관련 절로 이관한다.

\- schedule\_room\_maintenance()  
\- cancel\_room\_maintenance()  
\- create\_room()  
\- delete\_room()  
\- update\_room\_info()  
\- get\_active\_or\_scheduled\_room\_maintenance()

## **1.4.3 RoomBookingService**

\[클래스 개요\]

클래스명: RoomBookingService

역할:  
회의실 예약의 기본 생성, 조회, 변경, 취소 흐름은 1차 구현을 유지한다.  
2차 구현에서는 메모 저장, pending/reserved 처리, 정기 점검 충돌 검증, 체크인/퇴실 요청 상태 전이를 추가로 반영한다.

\[UML-스타일 요약\]

속성:  
\- clock : RuntimeClock  
\- booking\_repository : RoomBookingRepository  
\- room\_service : RoomService  
\- validator : RoomBookingValidator  
\- audit\_repository  
\- penalty\_repository  
\- user\_repository

메서드:  
\+ create\_booking(username : str, user\_count : int, start\_date : date, end\_date : date, room\_name : str, memo : str) : BookingResult  
\+ change\_booking\_date(username : str, booking\_id : str, new\_start\_date : date, new\_end\_date : date) : BookingResult  
\+ cancel\_booking(username : str, booking\_id : str) : BookingResult  
\+ request\_checkin(username : str, booking\_id : str) : BookingResult  
\+ request\_checkout(username : str, booking\_id : str) : BookingResult

\[메서드 상세\]

1\. create\_booking(username, user\_count, start\_date, end\_date, room\_name, memo) : BookingResult

회의실 예약을 생성한다.

2차 추가 처리:  
1\. memo를 예약 정보와 함께 저장한다.  
2\. 예약 기간과 정기 점검 예정 기간이 겹치면 예약을 생성하지 않는다.  
3\. 동일 회의실, 동일 기간의 pending 예약이 없으면 reserved 상태로 저장한다.  
4\. 동일 회의실, 동일 기간의 pending 예약이 있으면 pending 상태로 저장한다.  
5\. 현재 운영 시점이 18:00이고 예약 시작일이 다음 날인 경우에는 pending 상태를 만들지 않고 선착순 처리 규칙을 따른다.  
6\. 예약 생성 결과는 room\_bookings.txt에 저장한다.  
7\. 예약 생성 로그는 audit\_log.txt에 기록한다.

start\_time은 start\_date의 09:00으로 저장한다.  
end\_time은 end\_date의 18:00으로 저장한다.  
created\_at과 updated\_at은 현재 운영 시점으로 저장한다.

2\. change\_booking\_date(username, booking\_id, new\_start\_date, new\_end\_date) : BookingResult

회의실 예약 날짜를 변경한다.

2차 추가 처리:  
1\. 새 예약 기간이 scheduled 또는 active 정기 점검 기간과 겹치면 변경하지 않는다.  
2\. 새 예약 기간이 다른 활성 예약과 겹치면 변경하지 않는다.  
3\. 변경 가능 조건을 만족하면 start\_time, end\_time, updated\_at을 갱신한다.  
4\. 변경 결과는 room\_bookings.txt에 저장한다.  
5\. 변경 로그는 audit\_log.txt에 기록한다.

회의실 자체는 변경하지 않는다.

3\. cancel\_booking(username, booking\_id) : BookingResult

회의실 예약을 취소한다.

2차 유지/연동 처리:  
1\. reserved 상태인 본인 예약만 취소 가능하다.  
2\. 이미 시작된 예약은 취소할 수 없다.  
3\. 현재 운영 시점이 예약 시작 시각과 같은 경우 직전 취소로 판단하여 패널티를 부과할 수 있다.  
4\. 취소 성공 시 status를 cancelled로 변경한다.  
5\. cancelled\_at과 updated\_at은 현재 운영 시점으로 저장한다.  
6\. 취소 결과는 room\_bookings.txt에 저장한다.

4\. request\_checkin(username, booking\_id) : BookingResult

회의실 체크인 요청을 처리한다.

조건:  
1\. 본인 소유 예약이어야 한다.  
2\. status가 reserved여야 한다.  
3\. 예약 start\_time이 현재 운영 시점과 정확히 일치해야 한다.

처리:  
1\. status를 checkin\_requested로 변경한다.  
2\. requested\_checkin\_at을 현재 운영 시점으로 저장한다.  
3\. updated\_at을 현재 운영 시점으로 저장한다.  
4\. 변경 결과를 room\_bookings.txt에 저장한다.  
5\. 실제 체크인 승인은 관리자 기능에서 처리한다.

5\. request\_checkout(username, booking\_id) : BookingResult

회의실 퇴실 신청을 처리한다.

조건:  
1\. 본인 소유 예약이어야 한다.  
2\. status가 checked\_in이어야 한다.

처리:  
1\. status를 checkout\_requested로 변경한다.  
2\. requested\_checkout\_at을 현재 운영 시점으로 저장한다.  
3\. updated\_at을 현재 운영 시점으로 저장한다.  
4\. 변경 결과를 room\_bookings.txt에 저장한다.  
5\. 실제 퇴실 승인은 관리자 기능에서 처리한다.

## **1.4.4 RoomBookingValidator**

\[클래스 개요\]

클래스명: RoomBookingValidator

역할:  
회의실 예약 입력값과 예약 가능 조건을 검증한다.  
2차 구현에서는 메모 검증, 정기 점검 충돌 검증, pending/reserved 구분에 필요한 활성 예약 검증을 추가로 수행한다.

\[UML-스타일 요약\]

메서드:  
\+ validate\_memo(memo : str) : ValidationResult  
\+ has\_booking\_conflict(room\_name : str, start\_time : datetime, end\_time : datetime, ignore\_booking\_id : str | None \= None) : bool  
\+ has\_maintenance\_conflict(room\_name : str, start\_time : datetime, end\_time : datetime) : bool  
\+ has\_active\_room\_booking\_limit(username : str) : bool

\[메서드 상세\]

1\. validate\_memo(memo : str) : ValidationResult

memo가 None이면 빈 문자열로 처리한다.  
memo는 빈 문자열을 허용한다.  
memo에 줄바꿈 문자가 포함되면 실패 결과를 반환한다.  
memo의 문자 기준 길이가 50자를 초과하면 실패 결과를 반환한다.  
그 외의 경우 성공 결과를 반환한다.

2\. has\_booking\_conflict(room\_name, start\_time, end\_time, ignore\_booking\_id \= None) : bool

room\_bookings.txt에서 room\_id가 room\_name과 일치하는 예약을 조회한다.  
충돌 검사 대상 상태는 다음과 같다.

\- reserved  
\- checkin\_requested  
\- checked\_in  
\- checkout\_requested

pending 상태 예약은 예약 가능 목록 검사에서는 충돌로 보지 않는다.  
ignore\_booking\_id가 전달된 경우 해당 예약은 충돌 검사에서 제외한다.

충돌 판정식:  
기존 예약의 start\_time \< 새 예약의 end\_time  
그리고  
기존 예약의 end\_time \> 새 예약의 start\_time

두 조건이 모두 참이면 기간이 겹치는 것으로 판단한다.

3\. has\_maintenance\_conflict(room\_name, start\_time, end\_time) : bool

room\_maintenance.txt에서 room\_id가 room\_name과 일치하는 정기 점검 예약을 조회한다.  
scheduled 또는 active 상태의 정기 점검만 충돌 검사 대상으로 삼는다.

충돌 판정식:  
정기 점검의 start\_time \< 예약 end\_time  
그리고  
정기 점검의 end\_time \> 예약 start\_time

두 조건이 모두 참이면 정기 점검 기간과 예약 기간이 겹치는 것으로 판단한다.

4\. has\_active\_room\_booking\_limit(username : str) : bool

room\_bookings.txt에서 user\_id가 username과 일치하는 활성 회의실 예약 수를 계산한다.

활성 상태:  
\- pending  
\- reserved  
\- checkin\_requested  
\- checked\_in  
\- checkout\_requested

일반 유저의 활성 회의실 예약 제한은 2차 기획서의 최종 기준을 따른다.  
제한(restricted) 상태인 유저는 활성 회의실 예약이 1건 이상이면 추가 예약을 차단한다.

**\[동작 상세\]**

**\_edit\_equipment() 절차 상세 (가\~라)**

장비의 이름만 수정할 수 있다. 시리얼 번호, 종류, 장비의 상태는 수정할 수 없다. 최초 화면에 보이는 장비 목록은 커널 화면 상 최상단에 고정된다. 해당 장비가 현재 대여중인 경우에는 수정이 불가능하며, 예약만 존재하는 경우에는 편집이 가능하다.

1. 가. 편집할 장비 선택(번호) 0 입력 시 이전 메뉴로 복귀. 목록에 없는 숫자 또는 숫자가 아닌 문자/공백류 입력 시 해당 오류 메시지 출력 후 재입력. 현재 대여중인 장비 번호 입력 시 '대여중인 장비는 편집할 수 없습니다.' 출력 후 재입력. 편집 가능한 번호 입력 시 다음 절차로.  
2. 나. 새로운 이름 입력 빈 입력/공백류만 시 '이름을 입력해주세요.' 출력 후 재입력. 한글 미포함 시 '한글을 반드시 포함해야 합니다.' 출력 후 재입력. 10글자 초과 시 '10글자 이하로 입력해주세요.' 출력 후 재입력. 전체 장비 중 동일 이름 존재 시 '이미 존재하는 이름입니다.' 출력 후 재입력. 0 단독 입력 시 이전단계(상위 메뉴)로 복귀. 올바른 입력 시 다음 절차로.  
3. 다. 정말로 수정하시겠습니까?\[y/n\] y/yes/예/ㅇ 입력 시 이름 변경 후 '✓ 이름이 변경되었습니다: \[변경된 이름\]' 출력. n/no/아니오/ㄴ 입력 시 취소 후 관리자 메뉴로 복귀. 그 외 입력 시 'y 또는 n을 입력해주세요.' 출력 후 재입력.  
4. 라. 완료 다.번 절차 완료 시 0을 입력하여 관리자 메뉴로 복귀.

**\_delete\_equipment() 절차 상세 (가\~다)**

기존 목록에서 원하는 장비를 삭제할 수 있다. 해당 장비가 현재 대여중이거나 예약이 존재할 경우 삭제할 수 없다. 최소 장비 수는 12개로, 장비가 12개인 경우 삭제할 수 없다.

1. 가. 삭제할 장비 선택(번호) 목록에 없는 숫자 또는 숫자가 아닌 문자/공백류 입력 시 해당 오류 메시지 출력 후 재입력. 대여중이거나 예약 존재 시 '해당 장비는 삭제할 수 없습니다.' 출력 후 재입력. 장비가 현재 12개 상태에서 삭제 시도 시 '장비는 최소 12개 이상 유지되어야 합니다.' 출력 후 재입력. 0 입력 시 이전단계(상위 메뉴)로 복귀. 올바른 번호 입력 시 다음 절차로.  
2. 나. 정말로 삭제하시겠습니까?\[y/n\] y/yes/예/ㅇ 입력 시 장비 삭제 후 '✓ 장비를 삭제했습니다: \[삭제된 장비 이름(시리얼번호)\]' 출력. n/no/아니오/ㄴ 입력 시 취소 후 장비 편집 메뉴로 복귀. 그 외 입력 시 'y 또는 n을 입력해주세요.' 출력 후 재입력.  
3. 다. 완료 다.번 절차 완료 시 0을 입력하여 상위메뉴로 복귀.

**\_add\_equipment() 절차 상세 (가\~라)**

기존에 없는 새로운 장비를 추가할 수 있다. 장비는 기본으로 등록된 12개를 포함하여 최대 20개까지 추가할 수 있으며, 장비가 20개인 경우 추가할 수 없다. 장비가 추가되면 최초의 상태는 사용 불가(disabled)가 되며, 이는 자동 처리로직으로 상태가 사용 가능으로 변경되지 않는다. 반드시 관리자가 수동으로 상태를 사용가능으로 변경해야 한다.

1. 가. 장비 종류 선택 현재 등록된 장비 종류 목록을 번호와 함께 출력한다. 마지막 번호는 항상 '직접입력'으로 고정되며, 장비 종류 추가/삭제 시 목록은 자동으로 갱신된다. 종류 목록은 오름차순으로 정렬된다. 목록에 없는 숫자 또는 숫자가 아닌 문자/공백류 입력 시 해당 오류 메시지 출력 후 재입력. 0 입력 시 이전 단계(상위 메뉴)로 복귀. 직접입력 번호 입력 시 가-1.번 절차로 이동. 직접입력 외 올바른 번호 입력 시 나.절차로 이동.  
2. 가-1. 장비 종류 직접 입력 영어 외 문자 또는 공백류 포함 시 '영어만 입력해주세요.' 출력 후 재입력. 15글자 초과 시 '15글자 이하로 입력해주세요.' 출력 후 재입력. 0 입력 시 이전단계로 복귀. 올바른 입력 시 소문자로 자동 변환 저장 후 나.절차로 이동.  
3. 나. 장비 이름 입력 빈 입력/공백류만 시 '이름을 입력해주세요.' 출력 후 재입력. 한글 미포함 시 '한글을 반드시 포함해야 합니다.' 출력 후 재입력. 10글자 초과 시 '10글자 이하로 입력해주세요.' 출력 후 재입력. 전체 장비 중 동일 이름 존재 시 '이미 존재하는 이름입니다.' 출력 후 재입력. 0 입력 시 이전단계로 복귀. 올바른 입력 시 다음 절차로.  
4. 다. 시리얼 번호 영문 약자 입력 기존 종류를 선택한 경우 assign\_serial()로 자동 부여되며 이 절차는 생략한다. 직접입력으로 새 종류를 추가한 경우에만 진행한다. 영어 외 문자/공백류 입력 시 '영어만 입력해주세요.' 출력 후 재입력. 2글자가 아닌 경우 '영어 약자는 두 글자여야 합니다.' 출력 후 재입력. 이미 사용 중인 약자 입력 시 '이미 존재하는 약자입니다.' 출력 후 재입력. 0 입력 시 이전단계로 복귀. 올바른 입력 시 대문자로 자동 변환, 해당 종류 첫 번째 장비이므로 시리얼 번호 \[약자\]-001로 자동 부여 후 다음 절차로.  
5. 라. 정말로 추가하시겠습니까?\[y/n\] 종류/이름/시리얼번호를 재출력 후 확인. y/yes/예/ㅇ 입력 시 장비 추가(초기 상태: disabled) 후 '✓ 장비가 추가되었습니다: \[추가된 장비 종류 / 이름(시리얼번호)\]' 출력. n/no/아니오/ㄴ 입력 시 취소 후 장비 편집 메뉴로 복귀. 그 외 입력 시 'y 또는 n을 입력해주세요.' 출력 후 재입력. 라.번 절차 완료 시 0을 입력하여 상위메뉴로 복귀.

**assign\_serial() 동작 상세**

1. assign\_serial() 해당 종류의 기존 장비 시리얼 번호에서 번호 부분만 추출하여 집합으로 만든 뒤, 1부터 순서대로 비어있는 가장 작은 번호를 찾아 '\[약자\]-NNN' 형식으로 반환한다. 예를 들어 KB-001, KB-003이 존재하면 KB-002를 반환한다.

**함수명**

**함수 및 모듈에 대한 추가 설명**  
1\. create\_group()

흐름은 다음 순서로 진행된다.

1\. EquipmentBookingManager.\_check\_blocked\_conditions() 호출. 차단 시 사용자 메뉴 복귀.  
2\. EquipmentBookingManager.\_input\_dates() 재사용.  
3\. 입력 기간에 예약 가능한 전체 available 장비를 종류별로 그룹화하여 출력. 비어있으면 "해당 기간에 예약 가능한 장비가 없습니다." 출력 후 날짜 입력 복귀.  
4\. 사용자 입력을 받아 \`\_parse\_selection\_input()\`에 전달. 결과 종류에 따라 분기.  
5\. EquipmentBookingManager.\_input\_memo() 재사용. 묶음 전체에 동일 메모 1개 적용.  
6\. \`\_show\_group\_confirm\_list()\`로 선택 장비 목록 출력 후 "예약하시겠습니까? (y/n)" 확인.  
7\. n이면 사용자 메뉴 복귀.  
8\. \`\_save\_group\_bookings()\` 호출.  
8-1. 선택 도중 장비가 점검중 또는 사용불가 상태로 바뀐 경우, "장비 상태에 변경이 있습니다. 이전 메뉴로 돌아갑니다." 출력 후 사용자 메뉴 복귀. 8-2. 선택 도중 시점이 변경되어 다른 사용자가 해당 장비에 이미 예약한 경우, "이미 예약된 장비입니다" 출력 후 사용자 메뉴 복귀. 8-3. 선택 이후 장비가 점검중 또는 사용불가 상태로 바뀐 경우, 출력 화면 최상단에 "장비 상태에 대한 변경이 있습니다" 출력. (여기서 "선택 도중"이란 장비 선택 후 "✓ 예약 신청이 접수되었습니다." 출력 이전 단계를 의미한다. "선택 이후"란 예약 완료 후를 의미한다.) (장비는 반드시 available 상태일 때만 예약 가능하며 후보 목록에 표시된다.)   
9\. "✓ 예약 신청이 접수되었습니다." 출력 후 "0. 돌아가기" 표시.

2\. \_parse\_selection\_input()

사용자 입력 raw를 분석하여 4가지 결과 중 하나를 반환한다.

| 입력 | 반환 |
| ----- | ----- |
| "0" | ("cancel", \[\], "") |
| 빈 문자열 | ("error", \[\], "번호를 입력해주세요.") |
| 정수 1개 (정규식 ^\\d+$) | 범위 내 → ("single", \[idx\], "") / 범위 외 → ("error", \[\], "잘못 입력했습니다. 다시 입력해주세요.") |
| 공백/콤마 구분 다중 정수 | 아래 검증 통과 시 ("group", indices, ""), 실패 시 ("error", \[\], 사유 메시지) |
| 그 외 형식 | ("error", \[\], "올바른 번호를 입력해주세요. (예: 1 2 3 또는 1,2,3 또는 1, 2, 3)") |

다중 입력 검증 순서:

| 조건 | 메시지 |
| ----- | ----- |
| 토큰 개수 \> 3 | "묶음 예약은 최대 3건까지만 가능합니다." |
| 중복 번호 | "중복된 번호가 있습니다. (예: 1, 1, 2\) 다시 입력해주세요." |
| 범위 외 번호 | "없는 번호입니다. 다시 입력해주세요." |
| 같은 asset\_type 둘 이상 | "같은 종류의 장비는 예약하실 수 없습니다." |

3\. \_show\_group\_confirm\_list()

묶음 내 장비들을 가나다(ㄱㄴㄷ) 순서로 정렬하여 다음 형식으로 한 줄 출력한다.

\[묶음\] {장비1}, {장비2}, {장비3}

4\. \_save\_group\_bookings()

처리 순서:

1\. 새 \`group\_id \= uuid.uuid4()\` 생성.  
2\. equipment\_ids 각각에 대해 EquipmentBookingManager.\_save\_booking()을 호출. 인자로 동일 \`group\_id\`와 동일 \`memo\`, 그리고 \`\_decide\_status()\` 결과 status 전달.  
3\. "선착순 예외 ①"로 None을 반환받은 장비는 묶음에서 제외되고 나머지만 저장된다.  
4\. 저장 후 audit\_log.txt에 "create\_equipment\_group\_booking" 로그를 group\_id 기준 1건 추가.

 **1.5.5 EquipmentBookingViewer 클래스**  
클래스명: EquipmentBookingViewer

역할:  
\- 본인의 장비 예약 내역 조회 기능(메뉴 10, §6.5.2.3)을 담당하는 클래스이다.

\- 1차에 존재하던 \`\_show\_my\_equipment\_bookings()\`를 확장하며, 상태 매핑 보완·묶음 표시·메모 표시·복귀 방식 변경이 본 설계 대상이다.

변수명

| 변수명 | 변수 설명 |
| :---- | :---- |
| bookings: list\[EquipmentBooking\]  | \_show()에 선언하는 지역변수. 사용자의 전체 예약 목록 |
| sorted\_bookings: list\[EquipmentBooking\]  |  \_show()에 선언하는 지역변수. start\_time 내림차순으로 정렬한 결과 |

| equipment\_names: list\[str\]  | \_format\_row에 선언하는 지역변수. 묶음 내 장비들을 ㄱㄴㄷ순으로 정렬한 이름 목록 |

함수명

| 함수명 | 함수 설명 | 특이사항 |
| :---- | :---- | :---- |
| show() \-\> None  | 용자의 장비 예약 내역을 시작일 내림차순으로 출력하는 함수 | 1차 구현 함수에서 묶음 표시·메모 표시·복귀 방식 변경 |
| \_group\_by\_group\_id(bookings) \-\> list\[list\[EquipmentBooking\]\]  | 예약 목록을 group\_id 기준으로 그룹화하는 함수. 단일(group\_id=\`\\-\`)은 각자 1개짜리 그룹 | 신규 |
| \_format\_row(group) \-\> str  | 메모를 화면 표시용으로 변환하는 함수. memo가 \`\\-\`이면 빈 문자열 반환 | 신규  |
| \_format\_memo(memo) \-\> str  | 한 행(단일 또는 묶음)을 화면 출력용 문자열로 변환하는 함수 | 신규 |

1\. show()  
1차 구현된 정렬·출력 로직은 그대로 유지하되, 다음 3가지가 추가된다.

\- 출력 전 \`\_group\_by\_group\_id()\`를 호출하여 묶음을 한 그룹으로 묶는다.  
\- 각 그룹마다 \`\_format\_row()\`로 한 행을 만들고, 대표 레코드의 memo가 \`\\-\`가 아니면 \`\_format\_memo()\`로 다음 줄을 추가한다.  
\- 출력 후 복귀 방식이 1차의 \`input("계속하려면 Enter...")\` → "0. 돌아가기" 입력 대기로 변경된다. 0 입력 시 사용자 메뉴 복귀, 그 외 입력 시 "잘못 입력했습니다. 다시 입력해주세요." 재입력.

상태 매핑은 STATUS\_MAP을 사용하며, 1차에 없던 "pending" → "\[예약 대기중\]" 매핑이 추가된다.

2\. \_group\_by\_group\_id()

예약 목록을 group\_id 키로 임시 저장소(딕셔너리)에 모은다. group\_id \== \`\\-\`인 레코드는 각각 자기 id를 키로 사용해 1개짜리 그룹으로 만든다.

3\. \_format\_row()

\- 그룹 크기 1 (단일): \`{booking.id 앞 8자리} {장비명} {기간} {상태}\` 한 줄.  
\- 그룹 크기 2 이상 (묶음): \`{대표 id 앞 8자리} \[묶음\] {시리얼1}{장비1}, {시리얼2}{장비2}, ... {기간} {상태}\`. 대표 id는 묶음 내 가장 작은 booking id.

장비명은 ㄱㄴㄷ순 정렬.

4\. \_format\_memo()

memo가 \`\\-\`이면 빈 문자열 반환. 그 외에는 들여쓰기 10칸 \+ "메모: " \+ 메모 내용 반환.

**1.5.6 EquipmentBookingModifier 클래스**  
\[클래스 개요\]

클래스명: EquipmentBookingModifier

\- 장비 예약 변경 기능(메뉴 11, §6.5.2.4)을 담당하는 클래스이다.

\- 장비 ID는 유지하고 시작/종료 날짜만 변경 가능하다.

| \_select\_booking(modifiable) \-\> object | None  | 변경 가능한 목록을 출력하고 사용자 선택을 받는 함수. 0 입력 시 None 반환 | 1차 구현 함수에서 묶음 표시 추가 | 1차 구현 함수에서 묶음 표시 추가 |

\[함수 모듈 및 설명 \]

1\. modify()

흐름은 다음 순서로 진행된다.

1\. 진입 차단: banned 차단, 활성 예약 3건 이상 차단. 메시지 출력 후 복귀.  
2\. 사용자의 reserved 예약을 group\_id 기준 그룹화하여 변경 가능 목록 구성. 비어있으면 "변경 가능한 예약이 없습니다." 출력 후 복귀.  
3\. \`\_select\_booking()\` 호출.  
4\. 선택 항목의 start\_time이 current\_time보다 이전이면 "이미 시작된 예약은 변경할 수 없습니다." 출력 후 복귀.  
5\. "1. 입력 시작 / 0\. 돌아가기" 화면 출력. 0이면 복귀.  
6\. 새 날짜 입력 (1차 헬퍼 사용).  
7\. 새 기간이 기존과 같으면 "변경된 내용이 없습니다. 다른 날짜를 입력해주세요." 재입력.  
8\. 충돌 검사: 단일은 \`is\_period\_conflicted()\` 1회, 묶음은 묶음 내 모든 장비에 대해 검사. 하나라도 충돌이면 변경 불가.  
9\. 충돌 시 "해당 기간에 예약 가능한 장비가 없습니다." 출력 후 날짜 입력 복귀.  
10\. "정말 변경하시겠습니까? (y/n)" 확인. n이면 복귀. (1차 문구 "장비를 예약하시겠습니까?"에서 변경)  
11\. \`\_modify\_group\_booking()\` 또는 \`\_modify\_single\_booking()\` 호출.  
12\. "✓ 예약이 변경되었습니다. 새 기간: YYYY-MM-DD 09:00 \~ YYYY-MM-DD 18:00" 출력.

2\. \_select\_booking()

목록 표시 형식:

| 종류 | 형식 |
| ----- | ----- |
| 단일 | {번호}. {장비명} \- {기간} |
| 묶음 | {번호}. \[묶음\] {장비들 ㄱㄴㄷ순} \- {기간} |

정수 아닌 입력은 "잘못입력되었습니다. 다시 입력해주세요." 재입력. 0 입력은 None 반환. 빈 입력은 “번호를 입력해주세요.”

3\. \_modify\_group\_booking()

group\_id로 묶음 내 모든 레코드 조회 후 일괄 갱신:

1\. 각 레코드에 대해 \`is\_period\_conflicted(serial, new\_start, new\_end, exclude\_id=각 booking\_id)\` 검사.  
2\. 모두 통과 시 EquipmentBookingRepository를 통해 start\_time, end\_time, updated\_at 일괄 갱신. group\_id 유지.  
3\. audit\_log.txt에 "modify\_equipment\_group\_booking" 로그를 group\_id 기준 1건 추가.  
3-1. 변경할 예약 선택 시 빈 입력이 들어오면 "번호를 입력해주세요." 출력 후 재입력을 받는다.  
3-2. 변경 도중 일부 장비가 점검중 또는 사용불가 상태로 변경된 경우, "장비 상태에 변경이 있습니다. 이전 메뉴로 돌아갑니다." 출력 후 사용자 메뉴 복귀. 이후 해당 장비의 예약은 취소된다.

**1.5.6 EquipmentBookingCanceller**

\[클래스 개요\]

클래스명: EquipmentBookingCanceller

역할:역할:

\- 장비 예약 취소 기능(메뉴 12, §6.5.2.5)을 담당하는 클래스이다.

\- reserved 예약과 pending 예약 모두 취소할 수 있으며, pending은 \[대기\] 태그로 가나다 번호, reserved는 일반 숫자 번호로 표시한다.

\- 묶음 예약은 묶음 전체를 한 번에 취소하며, 직전 취소의 경우 묶음 내 장비 개수만큼 패널티를 부과한다.

\- 30일 카운트 검사 및 frequent\_cancel 1점 패널티 부과·equipment\_cancel\_restricted\_until 갱신 로직을 본 클래스에서 함께 처리한다. PenaltyService.apply\_fixed\_penalty()는 2점 고정 패널티만 다루므로, 1점인 frequent\_cancel은 본 클래스에서 PenaltyRepository를 직접 사용하여 penalties.txt에 레코드를 추가한다.

\- 1차에 존재하던 \`\_cancel\_equipment\_booking()\`을 확장하며, pending 취소·묶음 취소·frequent\_cancel 전체가 본 설계 대상이다.

### \- 함수 설명

1\. cancel()

예약 취소 전체 흐름을 다음 순서로 수행한다.

1. 진입 차단: banned만 검사 (활성예약 3건 차단은 적용 안 함).  
2. \_build\_cancellable\_list() 호출. 비어있으면 "취소 가능한 예약이 없습니다." 출력 후 복귀.  
3. \_select\_booking() 호출. None이면 복귀.  
4. reserved 항목의 start\_time \< current\_time이면 "이미 시작된 예약은 취소할 수 없습니다." 출력 후 복귀 (묶음은 전체 시작된 경우에만).  
5. pending 항목 → \_cancel\_pending() 호출.  
6. reserved 항목 → \_cancel\_reserved() 호출.  
7. \_build\_cancellable\_list()

목록 구성 순서:

1. status=pending 항목을 group\_id 기준 그룹화. 묶음 일부만 pending이면 pending된 장비만 표시. 가나다 순서로 가/나/다 번호 부여.  
2. status=reserved 항목을 group\_id 기준 그룹화. 1/2/3 숫자 번호 부여.

표시 형식:

| 종류 | 형식 |
| ----- | ----- |
| 단일 pending | 가. \[대기\] {장비} / {기간} |
| 묶음 전체 pending | 가. \[대기\] \[묶음\] {장비들 ㄱㄴㄷ} / {기간} |
| 묶음 일부 pending | 가. \[대기\] {장비} / {기간} (단일 형식) |
| 단일 reserved | 1\. {장비} / {기간} |
| 묶음 reserved | 1\. \[묶음\] {시리얼1}{장비1}, ... / {기간} |

3\. \_select\_booking()

목록 출력 후 "0. 취소" 추가. 정수 아닌 입력은 "잘못 입력했습니다. 다시 입력해주세요." 출력 후 재입력. 빈 입력은 "번호를 입력해주세요." 출력 후 재입력. 범위 외는 "잘못 입력했습니다. 다시 입력해주세요." 출력 후 재입력. 0 입력 시 None 반환.

4\. \_cancel\_pending()

1. "정말 취소하시겠습니까? (y/n)" 출력.  
2. y → 해당 pending 레코드(또는 같은 group\_id의 모든 pending 레코드)의 status를 cancelled로, cancelled\_at·updated\_at을 현재 시각으로 갱신.  
3. 패널티 부과 없음, 30일 카운트에도 산정 안 함 (\_apply\_frequent\_cancel\_if\_needed 호출 안 함).  
4. "예약 대기가 취소되었습니다." 출력.

pending 취소는 6.5.2.5의 패널티 정책에서 명시적으로 제외되므로 어떠한 패널티도 부과되지 않는다.

1. \_cancel\_reserved()  
2. \_is\_late\_cancel() 호출하여 직전 취소 여부 판정. 묶음은 묶음 내 직전 취소 해당 장비 개수를 late\_count로 산정.  
3. 직전 취소이면 "직전 취소 패널티 N점이 부과됩니다. 정말로 취소하시겠습니까? (y/n)" 출력. 단일은 N=2, 묶음은 N=장비 개수.  
4. 직전 취소 아니면 "정말로 취소하시겠습니까? 취소하시는 경우 패널티 부과 및 장비 예약 정상 이용이 일부 제한될 수 있습니다. (y/n)" 출력.  
5. n이면 복귀.  
6. y이면:  
   * 단일 → equipment\_service.cancel\_booking(user, booking\_id) 호출 (1차 구현 그대로). 직전 취소이면 PenaltyService를 통해 late\_cancel 2점이 자동 부과됨.  
   * 묶음 → \_cancel\_group\_booking() 호출. 묶음 내 직전 취소 해당 레코드 개수만큼 late\_cancel 패널티 부과.  
7. \_apply\_frequent\_cancel\_if\_needed(booking\_id, is\_late\_cancel) 호출. 반환된 메시지 줄들을 순서대로 화면에 출력.  
8. \_cancel\_group\_booking()

처리 순서:

1. group\_id로 묶음 내 모든 레코드 조회. 모두 reserved 상태 확인.  
2. 모든 레코드의 start\_time이 current\_time보다 이전이면 "이미 시작된 예약은 취소할 수 없습니다." 예외 발생.  
3. 하나라도 start\_time \== current\_time이면 직전 취소 적용, late\_count를 직전 취소 해당 장비 개수로 산정.  
4. 묶음 내 모든 레코드를 status=cancelled, cancelled\_at=now, updated\_at=now로 일괄 갱신.  
5. 직전 취소이면 PenaltyService.apply\_fixed\_penalty()를 묶음 내 직전 취소 해당 레코드별로 호출 (장비 개수 × 2점). 본 호출은 equipment\_service를 통해 위임된다.  
6. 반환: (취소된 레코드 리스트, is\_late, late\_count).  
7. \_is\_late\_cancel()

booking.start\_time을 datetime으로 파싱하여 다음 조건 검사:

* start\_time의 날짜 \== current\_time의 날짜  
* start\_time의 시각 \== 09:00

두 조건 모두 만족 시 True 반환.

1. \_apply\_frequent\_cancel\_if\_needed()

\_cancel\_reserved()의 마지막 단계에서 호출된다. 처리 순서는 다음과 같다.

1. \_count\_cancellations\_30()을 호출하여 cnt 획득 (이번 취소 포함 30일 내 취소 횟수).  
2. 분기에 따라 패널티 부과 및 제한 갱신을 결정한다.

| is\_late\_cancel | cnt | 처리 내용 |
| ----- | ----- | ----- |
| True | \== 3 | 제한 갱신만 수행 (\_update\_restriction\_until 호출). frequent\_cancel 패널티는 부과하지 않음. |
| True | \>= 4 | 제한 갱신·패널티 부과 모두 수행하지 않음. |
| False | \== 3 | frequent\_cancel 1점 부과(\_append\_frequent\_cancel\_penalty 호출) \+ 제한 갱신. |
| False | \>= 4 | frequent\_cancel 1점 부과만 수행, 제한 갱신은 안 함. |
| 임의 | \< 3 | 아무 처리 없음. |

1. \_build\_penalty\_message()를 호출하여 출력 메시지 줄 목록을 결정하고 반환한다.

is\_late\_cancel \== True 시 frequent\_cancel을 부과하지 않는 이유는, 직전 취소(late\_cancel) 2점이 이미 PenaltyService에 의해 부과되었기 때문이며, 동일한 취소 사건에 대해 두 종류 패널티 중복 부과를 방지하기 위함이다.

1. \_count\_cancellations\_30()

사용자의 30일 내 장비 취소 횟수를 다음 산정 규칙에 따라 카운트한다.

1. EquipmentBookingRepository에서 user.id의 status=cancelled 인 모든 예약을 조회한다. admin\_cancelled는 카운트에서 제외한다.  
2. cancelled\_at의 날짜가 \[current\_date \- 29일, current\_date\] 범위 내인 레코드만 포함한다.  
3. 위 2번을 만족하는 레코드 중, cancelled\_at의 날짜가 start\_time의 날짜 \- 14일보다 늦은(즉 더 가까운 미래의) 레코드만 포함한다. 예약 시작 14일 이전의 미리 취소는 30일 카운트에서 제외한다.  
4. 묶음 예약(group\_id \!= "-")은 묶음 내 레코드 개수만큼 각각 카운트한다. 예를 들어 묶음 3건이 한꺼번에 취소되면 3회로 산정된다.  
5. 위 조건을 모두 만족하는 레코드 수를 합산하여 정수로 반환한다. 이때 호출 시점의 취소 건(이번 취소)도 이미 cancelled 상태로 저장된 후 카운트되므로 반환값에 포함된다.  
6. \_append\_frequent\_cancel\_penalty()

penalties.txt에 frequent\_cancel 1점 레코드를 다음 필드 값으로 생성하여 직접 추가한다.

| 필드 | 값 |
| ----- | ----- |
| penalty\_id | uuid.uuid4() 새로 생성 |
| username | user.id |
| reason | "frequent\_cancel" |
| points | 1 |
| related\_type | "equipment\_booking" |
| related\_id | 인자로 받은 booking\_id (마지막으로 취소한 예약의 ID) |
| memo | "" (빈 문자열) |
| created\_at | 현재 시각 (now\_iso()) |
| updated\_at | 현재 시각 (now\_iso()) |

생성 후 PenaltyRepository를 통해 penalties.txt에 append 방식으로 저장한다. 그리고 user.penalty\_points를 1 증가시켜 UserRepository를 통해 users.txt를 갱신한다. 사용자 상태(normal/restricted/banned)가 전이되는 경우 §6.6.3.3.1 기준에 따라 자동으로 반영된다.

audit\_log.txt에 actor=user.id, action="apply\_frequent\_cancel", target\_type="penalty", target\_id=새로 생성한 penalty\_id, details="장비 30일 반복 취소"로 로그를 추가한다.

본 클래스에서 PenaltyService를 거치지 않고 PenaltyRepository를 직접 사용하는 이유는, PenaltyService.apply\_fixed\_penalty()가 2점 고정 패널티만 처리하고 frequent\_cancel의 1점 가변 부과를 지원하지 않기 때문이다.

11\. \_update\_restriction\_until()

신규 예약 제한 해제 시각을 다음과 같이 계산하여 갱신한다.

release\_date \= current\_date \+ timedelta(days=7)  
restriction\_until \= datetime.combine(release\_date, time(9, 0)).isoformat()

예시: current\_date가 2026-05-10이면 release\_date는 2026-05-17이고 restriction\_until은 "2026-05-17T09:00"이 된다.

UserRepository를 통해 user.equipment\_cancel\_restricted\_until 필드를 갱신하고 users.txt에 저장한다. user.updated\_at도 함께 현재 시각으로 갱신한다.

audit\_log.txt에 actor="system", action="update\_restriction\_until", target\_type="user", target\_id=user.id, details="장비 신규 예약 7일 제한"로 로그를 추가한다.

본 갱신은 다음 두 가지 경우에만 호출된다.

* is\_late\_cancel \== True 이고 cnt \== 3 (직전 취소이면서 30일 카운트 처음 3회 도달)  
* is\_late\_cancel \== False 이고 cnt \== 3 (일반 취소이면서 30일 카운트 처음 3회 도달)

cnt \>= 4인 경우에는 이미 첫 3회 시점에 제한이 설정되어 있으므로 추가 갱신하지 않는다. 이를 통해 4회째 이후 추가 취소가 발생해도 제한 해제 시각이 늘어나지 않는다.

1. \_build\_penalty\_message()

cnt와 is\_late\_cancel 조합에 따라 화면에 출력할 메시지 줄 목록을 반환한다.

| 조건 | 출력 메시지 |
| ----- | ----- |
| cnt \< 3 | \[" 장비 예약 취소가 완료되었습니다."\] |
| is\_late\_cancel \== True, cnt \== 3 | \[" 장비 예약 취소가 완료되었습니다.", "최근 30일 내 장비 예약 취소가 3회를 초과하였습니다.", "{해제일}까지 신규 예약이 제한됩니다."\] |
| is\_late\_cancel \== True, cnt \>= 4 | \[" 장비 예약 취소가 완료되었습니다."\] |
| is\_late\_cancel \== False, cnt \== 3 | \[" 장비 예약 취소가 완료되었습니다.", "최근 30일 내 장비 예약 취소가 3회를 초과하였습니다.", "패널티 1점이 부과되며, {해제일}까지 신규 예약이 제한됩니다."\] |
| is\_late\_cancel \== False, cnt \>= 4 | \[" 장비 예약 취소가 완료되었습니다.", "추가 취소로 인해 패널티 1점이 부과되었습니다."\] |

위 메시지의 {해제일}은 "YYYY-MM-DD" 형식으로 치환된다. 예를 들어 release\_date가 2026-05-17이면 "2026-05-17까지 신규 예약이 제한됩니다."로 출력된다.

반환된 messages 리스트는 호출자(\_cancel\_reserved())가 한 줄씩 print()로 출력한다.

5\. 묶음의 경우 선택한 묶음 내 장비 목록 출력:

선택한 묶음:  
\- {장비1} ({시리얼1})

3\. \_select\_booking()

목록 출력 \+ "0. 취소" 표시 후 입력 받기. 정수 아닌 입력은 "잘못 입력했습니다. 다시 입력해주세요." 재입력. 0 입력은 None 반환. 빈입력시 "번호를 잘못입력했습니다"

**1.5.10 EquipmentPriorityResolver 클래스**  
\[클래스 개요\]  
클래스명: EquipmentPriorityResolver

역할:

장비 예약 우선권 처리 시스템(6.5.2.9)을 담당하는 클래스이다.  
시점 이동 시 호출되어 모든 pending 예약을 그룹화하고 penalty\_points·created\_at 기준으로 정렬하여 1순위는 reserved, 나머지는 cancelled로 일괄 처리한다.  
시점 이동 결과 화면에서 본인의 결과 알림 문자열을 생성하는 함수도 제공한다.  
본 호출은 시점 이동 함수(다른 팀원 파트)에서 트리거되며, 본 클래스는 우선권 해소 로직과 알림 생성 로직만 담당한다.  
1차 기능에 없던 신규 클래스이며 6.5.2.9 전체가 본 설계 대상이다.

| 변수  | 변수명 |
| :---- | :---- |
| bookings: list\[EquipmentBooking\]  | resolve\_all에 선언하는 지역변수. 전체 장비 예약 목록  |
| pendings: list\[EquipmentBooking\]  | resolve\_all에 선언하는 지역변수. status=pending인 레코드만 필터링  |
| groups: dict\[tuple, list\]  | resolve\_all에 선언하는 지역변수. (equipment\_id, start\_time, end\_time) 키로 그룹화한 결과  |
| confirmed\_ids: list\[str\]  | resolve\_all에 선언하는 지역변수. reserved로 확정될 booking\_id 목록  |
| cancelled\_ids: list\[str\]  | resolve\_all에 선언하는 지역변수. cancelled로 처리될 booking\_id 목록  |
| sorted\_group: list\[EquipmentBooking\]  | resolve\_all에 선언하는 지역변수. 우선권 정렬된 그룹  |
| user: User  | \_sort\_by\_priority에 선언하는 지역변수. 각 예약 사용자의 정보 조회  |
| user\_confirmed: list\[EquipmentBooking\]  | build\_user\_notifications에 선언하는 지역변수. 해당 유저의 확정된 예약  |
| user\_cancelled: list\[EquipmentBooking\]  | build\_user\_notifications에 선언하는 지역변수. 해당 유저의 탈락된 예약  |
| success\_lines: list\[str\]  | build\_user\_notifications에 선언하는 지역변수. 성공 알림 출력 줄들  |
| fail\_lines : List\<String\> | build\_user\_notifications에 선언하는 지역변수. 실패 알림 출력 줄들  |

함수명

| 함수명 | 함수 설명 | 특이사항 |
| :---- | :---- | :---- |
| resolve\_all(current\_time) \-\> tuple\[list\[str\], list\[str\]\]  | 모든 pending 예약을 그룹화하고 우선권 기준에 따라 확정·탈락을 결정하여 일괄 처리하는 함수. 시점 이동 함수(다른 팀원)에서 호출  | 신규  |
| \_group\_pendings(pendings) \-\> dict\[tuple, list\[EquipmentBooking\]\]  | pending 레코드를 (equipment\_id, start\_time, end\_time) 키로 그룹화  | 신규 |
| \_sort\_by\_priority(group) \-\> list\[EquipmentBooking\]  | 그룹 내 레코드를 penalty\_points 오름차순 → created\_at 오름차순으로 정렬  | 신규 |
| build\_user\_notifications(user\_id, confirmed\_ids, cancelled\_ids) \-\> tuple\[list\[str\], list\[str\]\]  | 특정 유저의 결과 알림 문자열을 생성하는 함수. 시점 이동 결과 화면에서 호출  | 신규 |

1\. resolve\_all()

시점 이동 시 호출되어 모든 pending 예약의 우선권을 해소한다.

1\. EquipmentBookingRepository에서 모든 예약 로드.  
2\. status=pending 레코드만 필터링.  
3\. \`\_group\_pendings()\`로 (equipment\_id, start\_time, end\_time) 키 기준 그룹화.  
(묶음 예약의 경우 group\_id가 같더라도 그룹화 키 (equipment\_id, start\_time, end\_time)에 따라 각 장비가 독립적인 그룹에 속하게 되어, 묶음 내 장비별로 개별 우선권이 적용된다.)  
4\. 각 그룹에 대해 \`\_sort\_by\_priority()\` 호출 후:  
   \- 그룹 크기 1 (단독 pending): 1순위만 reserved 확정.  
   \- 그룹 크기 2 이상 (충돌 pending): 1순위만 reserved, 나머지는 cancelled.  
5\. confirmed 레코드: status=reserved, updated\_at=current\_time 갱신.  
6\. cancelled 레코드: status=cancelled, cancelled\_at·updated\_at=current\_time 갱신.  
7\. 일괄 저장 후 audit\_log.txt에 "resolve\_equipment\_priority" 로그를 그룹별 1건씩 추가.  
8\. 반환: (confirmed\_ids, cancelled\_ids).

2\. \_group\_pendings()  
pending 레코드들을 (equipment\_id, start\_time, end\_time) 튜플을 키로 하는 딕셔너리에 모은다.

3\. \_sort\_by\_priority()

그룹 내 레코드들을 두 단계로 정렬:

1\. 1순위: 사용자의 penalty\_points 오름차순.  
2\. 2순위 (동점): created\_at 오름차순.

추가된 필드는 기획서 5.1절 문법 규칙에 따라 “YYYY-MM-DDTHH:MM” 형식으로 저장된다.

# **2.2 rooms.txt**

회의실 예약 파일의 기본 구조는 1차 구현을 따른다.  
2차 구현에서는 다음 필드를 사용자 회의실 기능에서 사용한다.

\- status  
\- requested\_checkin\_at  
\- requested\_checkout\_at  
\- cancelled\_at  
\- created\_at  
\- updated\_at  
\- memo

2차에서 사용하는 주요 상태 전이:  
\- pending → reserved  
\- pending → cancelled  
\- reserved → checkin\_requested  
\- checkin\_requested → checked\_in  
\- checked\_in → checkout\_requested  
\- checkout\_requested → completed  
\- reserved → cancelled

사용자 회의실 기능에서 직접 수행하는 상태 전이:  
\- reserved → checkin\_requested  
\- checked\_in → checkout\_requested  
\- reserved → cancelled

운영 시계 또는 관리자 기능에서 수행하는 상태 전이:  
\- pending → reserved  
\- pending → cancelled  
\- checkin\_requested → checked\_in  
\- checkout\_requested → completed

# **2.3 room\_bookings.txt**

사용자 회의실 파트에서는 room\_maintenance.txt를 직접 수정하지 않는다.  
다만 회의실 목록 조회와 예약 가능 여부 판단을 위해 RoomMaintenanceRepository를 통해 조회한다.

사용 목적:  
\- 회의실 목록 조회 시 정기 점검 예정 정보 표시  
\- 회의실 예약 생성 시 정기 점검 기간 충돌 검증  
\- 회의실 예약 변경 시 정기 점검 기간 충돌 검증

날짜: 2026-01-01 과 같이 년도-월-일 로 저장  
상태: 상태가 점검중일 경우 maintenance 로 저장, 예약이 존재하면 disabled, 예약이 없으면서 데이터파일에 저장되어 있지 않은 날짜의 경우 available로 간주하며, 데이터파일에 별도 표시하지 않음

날짜 기준으로 오름차순 정렬하여 작성하며, 미래 상태가 변경될 시, 해당 날짜의 future\_status\_changes 를 삭제

한 행에 레코드 1개 표현, 빈 행 없음.

# **2.5 equipment\_booking.txt**

파일 경로  
data/equipment\_booking.txt

저장 형식  
\<booking\_id\>|\<user\_id\>|\<serial\_id\>|\<start\_time\>|\<end\_time\>|\<status\>|\<checked\_out\_at\>|\<requested\_pickup\_at\>|\<requested\_return\_at\>|\<returned\_at\>|\<cancelled\_at\>|\<created\_at\>|\<updated\_at\>|\<group\_id\>|\<memo\>  
필드 순서는 고정이며 총 15개이다. 1차에는 13개 필드였으나, 2차에서 마지막에 group\_id와 memo 2개 필드가 추가되었다.

단일 예약 예시)

25c7cbb1-d5ff-4ba0-8b43-63c8f69cbbeb|student1|PJ-001|2026-04-10T09:00|2026-04-10T18:00|reserved|-|-|-|-|-|2026-04-05T12:10|2026-04-05T12:10|-|발표 준비용  
묶음 예약 예시) (같은 group\_id를 공유하는 2건)

a1b2c3d4-e5f6-7890-abcd-ef1234567890|student2|CB-001|2026-05-01T09:00|2026-05-06T18:00|pending|-|-|-|-|-|2026-04-28T18:00|2026-04-28T18:00|39fjsl1a-l3js-3lsj-8nf3-ek1sf69cksq|-  
b2c3d4e5-f6a7-8901-bcde-f12345678901|student2|NB-001|2026-05-01T09:00|2026-05-06T18:00|pending|-|-|-|-|-|2026-04-28T18:00|2026-04-28T18:00|39fjsl1a-l3js-3lsj-8nf3-ek1sf69cksq|-

설명  
장비 예약 정보를 저장하는 파일이다.  
booking\_id는 UUID v4 형식의 장비 예약 식별자이며 중복될 수 없다.  
user\_id는 users.txt에 존재하는 username이어야 한다.  
serial\_id는 equipments.txt에 존재하는 serial\_number이어야 한다.  
start\_time은 예약 시작 시각이며 날짜+시간 형식으로 저장한다.  
end\_time은 예약 종료 시각이며 start\_time보다 빠를 수 없다.  
유저 예약의 start\_time은 입력한 시작 날짜의 09:00으로 저장한다.  
유저 예약의 end\_time은 입력한 종료 날짜의 18:00으로 저장한다.  
status는 pending, reserved, pickup\_requested, checked\_out, return\_requested, returned, cancelled, admin\_cancelled 중 하나만 저장할 수 있다. (1차에는 pending이 없었고 2차에서 추가됨)  
checked\_out\_at은 관리자가 픽업을 승인한 시각이다.  
requested\_pickup\_at은 유저가 픽업을 요청한 시각이다.  
requested\_return\_at은 유저가 반납을 신청한 시각이다.  
returned\_at은 관리자가 반납을 승인하여 예약이 완료된 시각이다.  
cancelled\_at은 예약이 취소된 시각이다.  
created\_at은 예약 생성 시각이다.  
updated\_at은 예약 마지막 수정 시각이다.  
group\_id (2차 신규 필드) 는 묶음 예약을 식별하기 위한 UUID v4 형식의 식별자이며 묶음 내 모든 레코드가 동일한 값을 공유한다. 단일 예약의 경우 "-"로 저장한다. 묶음 내 모든 레코드는 user\_id, start\_time, end\_time, memo, group\_id가 동일해야 하며 serial\_id만 서로 다르다.  
memo (2차 신규 필드) 는 유저가 예약 시 입력한 메모이며 0자 이상 50자 이하 문자열이다. 묶음 예약의 경우 묶음 내 모든 레코드에 동일한 값을 저장한다. 입력값이 없는 경우 "-"로 저장한다.  
값이 없는 시각 필드는 "-"로 저장한다.  
읽는 기능  
장비 목록 조회: 등록된 장비를 출력할 때 읽는다. (단, 장비 정보 자체는 equipments.txt를 읽음)  
장비 예약하기: 같은 장비의 기존 예약과 충돌하는지 확인할 때 읽는다.  
내 장비 예약 조회: 현재 유저의 예약 이력을 출력할 때 읽는다.  
장비 예약 변경: 변경 가능한 예약 목록과 충돌 여부를 확인할 때 읽는다.  
장비 예약 취소: 취소 가능한 예약 목록을 확인할 때 읽는다. pending 상태 예약도 함께 조회한다.  
장비 픽업 요청: 픽업 요청 가능한 예약 목록을 확인할 때 읽는다.  
장비 반납 신청: 반납 신청 가능한 예약 목록을 확인할 때 읽는다.  
예약 우선권 처리 (2차 신규): 시점 이동 시 pending 상태인 모든 예약을 읽는다.  
쓰는 기능  
장비 예약하기: 새 예약 레코드를 추가한다. 묶음 예약의 경우 동일한 group\_id, memo, start\_time, end\_time을 가진 복수의 레코드를 일괄 추가한다. 동시 신청 충돌 시 status를 pending으로, 그 외에는 reserved로 저장한다.  
장비 예약 변경: start\_time, end\_time, updated\_at을 갱신한다. 묶음 예약의 경우 동일 group\_id를 가진 모든 레코드를 일괄 갱신한다.  
장비 예약 취소: status를 cancelled로 변경하고 cancelled\_at, updated\_at을 갱신한다. 묶음 예약의 경우 동일 group\_id를 가진 모든 레코드를 일괄 갱신한다. pending 상태 예약 취소도 동일하게 처리한다.  
장비 픽업 요청: status를 pickup\_requested로 변경하고 requested\_pickup\_at, updated\_at을 갱신한다. 묶음 예약의 경우 동일 group\_id를 가진 모든 레코드를 일괄 갱신한다.  
장비 반납 신청: status를 return\_requested로 변경하고 requested\_return\_at, updated\_at을 갱신한다. 묶음 예약의 경우 동일 group\_id를 가진 모든 레코드를 일괄 갱신한다.  
예약 우선권 처리 (2차 신규): 동일 (serial\_id, start\_time, end\_time) 조합으로 그룹화된 pending 예약 중 1순위 유저의 레코드는 status를 reserved로, 나머지는 cancelled로 일괄 갱신한다. 1순위는 사용자의 패널티 점수 오름차순, 동점 시 created\_at 오름차순으로 결정한다.  
상태 전이  
(신규) → reserved  
(신규) → pending          (2차 신규)  
pending → reserved        (2차 신규)  
pending → cancelled       (2차 신규)  
reserved → pickup\_requested  
pickup\_requested → checked\_out  
checked\_out → return\_requested  
return\_requested → returned  
reserved → cancelled  
reserved → admin\_cancelled  
유저 장비 기능에서 직접 수행하는 상태 전이

reserved → pickup\_requested  
checked\_out → return\_requested  
reserved → cancelled  
pending → cancelled                  (2차 신규)  
관리자 또는 운영 시계 기능에서 수행하는 상태 전이

pending → reserved                   (2차 신규)  
pending → cancelled                  (2차 신규)  
pickup\_requested → checked\_out  
return\_requested → returned  
reserved → admin\_cancelled

# **2.7 clock.txt**

파일 경로  
data/clock.txt

저장 형식  
\<현재 날짜\>T\<현재 시각\>  
의 형태로 저장된다.

예시)  
2026-05-01T09:00  
2026-05-01T18:00

설명  
\- 현재 시스템의 운영 시점을 저장하는 파일이다.  
\- 운영 시계는 실제 시간과 독립적인 가상 시점이다.  
\- 프로그램 최초 실행 시 입력한 시작 시점을 저장한다.  
\- 시점 이동이 발생할 경우 변경된 운영 시점을 저장한다.  
\- 프로그램 재실행 시 저장된 시점을 불러와 현재 운영 시점으로 사용한다.  
\- 운영 시각은 09:00 또는 18:00만 저장 가능하다.  
\- 09:00 다음 시점은 같은 날짜의 18:00이다.  
\- 18:00 다음 시점은 다음 날짜의 09:00이다.  
\- 파일이 비어 있거나 최초 생성된 경우에는 CLOCK\_SENTINEL 값을 사용하여 아직 운영 시점이 설정되지 않았음을 나타낸다.

# **2.8 room\_maintenance.txt**

파일 경로  
  data/room\_maintenance.txt

역할  
  회의실 정기 점검 예약 정보를 저장하는 파일이다.  
  rooms.txt가 회의실의 현재 운영 상태를 저장한다면,  
  room\_maintenance.txt는 미래에 예정된 정기 점검 또는 현재 진행 중인 정기 점검 정보를 저장한다.

저장 형식  
  \<schedule\_id\>|\<room\_id\>|\<start\_time\>|\<end\_time\>|\<status\>|\<created\_at\>|\<updated\_at\>|\<cancelled\_at\>

예시  
  b7a9c8d3e4f12a98d0c1b65f9e8123aa74f2a7e1d5c9b8a6e3f0d41c7a91b2c3|회  
  의실 4A|3000-01-10T18:00|3000-01-12T09:00|scheduled|3000-01-  
  01T09:00|3000-01-01T09:00|-

필드 설명  
\- schedule\_id: 정기 점검 예약 식별자이다. SHA-256 해시 결과인 64자리 16진수 소문자 문자열이며, 다른 정기 점검 예약과 중복될 수 없다. RoomService.schedule\_room\_maintenance()에서 정기 점검 예약을 생성할 때 다음 값을 순서대로 결합한 문자열의 SHA-256 해시값으로 생성한다.  
    \-\> room\_id|start\_time|end\_time|current\_time|random\_value  
room\_id는 정기 점검 대상 회의실 식별자이다.  
    	start\_time은 정기 점검 시작 시각이다.  
    	end\_time은 정기 점검 종료 시각이다.  
    	current\_time은 정기 점검 예약 생성 당시의 현재 운영 시점이다.  
    	random\_value는 동일한 회의실, 동일한 점검 기간, 동일한 운영 시점에서도 식별자가 달라지도록 하기 위한 난수 문자열이다.

\- room\_id: 정기 점검 대상 회의실 식별자이다. rooms.txt에 존재하는 회의실 name과 일치해야 한다.  
\- start\_time: 정기 점검 시작 시각이다. 관리자가 입력한 시작일의 18:00으로 저장한다.  
\- end\_time: 정기 점검 종료 시각이다. 관리자가 입력한 종료일의 09:00 으로 저장한다.  
\- status: 정기 점검 상태이다. scheduled, active, completed,cancelled 중 하나만 저장할 수 있다.  
\- created\_at: 정기 점검 예약이 생성된 시각이다.  
\- updated\_at: 정기 점검 예약 정보가 마지막으로 수정된 시각이다.  
\- cancelled\_at: 정기 점검 예약이 취소된 시각이다. 취소되지 않은 경우 "-"로 저장한다.

  status 값 의미  
  \- scheduled: 미래에 시작될 예정인 정기 점검 예약 상태이다.  
  \- active: 현재 운영 시점 기준 진행 중인 정기 점검 상태이다.  
  \- completed: 정기 점검 종료 시각이 지나 정상적으로 완료된 상태이다.  
  \- cancelled: 관리자가 정기 점검 예약 또는 진행 중 정기 점검을 취소한  
  상태이다.

# **2.9 waitlist.txt**

\[파일 경로\]

data/waitlist.txt

\[역할\]

공유 오피스 자원(회의실 및 장비)에 대한 사용자의 대기 예약 신청 내역을 보관하는 파일이다. 

\[형식\]

\<wait\_id\>|\<user\_id\>|\<resource\_type\>|\<resource\_id\>|\<target\_date\>|\<time\_slot\>|\<rank\>|\<created\_at\> 의 형태로 저장된다.

\[저장 예시\]

a1b2c3d4-5678-90ab-cdef-123456789012|user001|room\_booking|4A|2026-05-20|13:00\~14:00|1|2026-05-18T10:00

\[각 필드 설명 및 규칙\]

* wait\_id: 대기 예약 식별자 (다른 대기 건과 절대 중복될 수 없는 UUID v4)  
* user\_id: 대기를 신청한 사용자 ID (users.txt에 실재하는 회원이어야 함)  
* resource\_type: 예약 대상 자원의 종류 (room\_booking, equipment\_booking 두 가지 리터럴 값만 허용)  
* resource\_id: 대상 회의실의 명칭(예: 4A) 또는 대여 장비의 고유 시리얼 번호  
* target\_date: 대기 대상이 되는 스케줄의 날짜 (YYYY-MM-DD 포맷)  
* time\_slot: 대기 대상이 되는 스케줄의 이용 시간대 (HH:mm\~HH:mm 포맷)  
* rank: 현재 대기 순번 (1 이상의 양의 정수만 허용. 앞선 1순위가 승격되어 파일에서 삭제되면 시스템 연산에 의해 후순위들의 숫자가 일괄적으로 1씩 자동 차감 갱신됨)  
* created\_at: 사용자가 최종적으로 대기 신청을 완료한 시각 (YYYY-MM-DDTHH:mm 포맷)
